## このセクションで学ぶこと

- ローカル state がマシン上のファイルとして保存されることを理解する
- チーム開発でローカル state を使うと衝突・喪失が起きる理由を説明できる
- state を Git にコミットしてはいけない理由を把握する

## ローカル state は手元のファイルでしかない

これまで使ってきた `terraform.tfstate` は、Terraform を実行したマシンの **ローカルディスク** に置かれます。これが既定の **ローカルバックエンド** です。あわせて、apply のたびに直前の状態を退避した `terraform.tfstate.backup` も作られます。

一人で学習する分には、手元にファイルがあるだけで十分に動きます。しかし、この「手元にしかない」という性質が、実務では大きな弱点になります。

## チームで使うと衝突・喪失が起きる

複数人が同じインフラを管理する場面を考えてみましょう。A さんと B さんがそれぞれ自分の PC でローカル state を持っていると、次のような問題が起きます。

- **state がずれる**: A さんが apply して作ったリソースは、A さんの state にしか記録されません。B さんの state には存在しないため、B さんが apply すると同じリソースをもう一度作ろうとします。
- **同時実行の衝突**: 二人が同時に apply すると、互いの変更を知らないまま実物を書き換え、どちらの state も実態と食い違います。
- **喪失リスク**: PC の故障や誤削除で state を失うと、Terraform は管理対象を丸ごと見失います。

ローカル state には、複数人で安全に共有する仕組みも、同時実行を防ぐ仕組みもありません。**state は本来、チーム全員が参照する「唯一の真実」であるべき** なのに、ローカルではそれが各自のディスクに分散してしまうのです。

## state を Git にコミットしてはいけない

「では state を Git で共有すれば?」と考えたくなりますが、これは避けるべきです。理由は 2 つあります。

1 つは **マージ衝突** です。state は機械が生成する JSON で、人間が手でマージできる形式ではありません。コンフリクトを手作業で解決すると壊れます。

もう 1 つは **秘匿情報の漏洩** です。state には、リソースの属性として DB のパスワードや接続文字列などが **平文で残る** ことがあります。これをリポジトリに含めると、閲覧権限を持つ全員に漏れます。`.gitignore` に `*.tfstate*` を加えてコミット対象から外すのが定石です。

これらの問題を根本的に解決するのが、次のセクションで扱う **リモート state** です。

## まとめ

- ローカル state は実行マシンのディスクにあるだけで、共有や排他制御の仕組みがない。
- チームで使うと state のずれ・同時実行の衝突・喪失が起きる。
- state には秘匿情報が平文で残りうるため Git にコミットしない(`*.tfstate*` を ignore)。
