## このセクションで学ぶこと

- AWS が「クラウドサービスの集合体」であることを理解する
- 代表的なサービスカテゴリ(コンピューティング・ストレージ・ネットワーク・データベース)を把握する
- マネージドサービスという考え方とその利点を説明できる

## AWS は「サービスの巨大な集まり」

AWS(Amazon Web Services)は、Amazon が提供するクラウドサービスの集合体です。第 1 章で学んだとおり、クラウドは「サーバーやストレージを所有せず、必要なときに必要なだけ利用する」仕組みでした。AWS はその考え方を、実に 200 を超える個別サービスとして提供しているプラットフォームだと考えてください。

ここで大切なのは、AWS を「ひとつの製品」ではなく「役割ごとに分かれた部品の集まり」として捉えることです。家を建てるときに木材・配管・電気設備をそれぞれ調達するように、システムを作るときも目的別のサービスを組み合わせて使います。最初からすべてを覚える必要はなく、よく使うものから少しずつ把握していけば十分です。

## 代表的なサービスカテゴリ

AWS のサービスは役割ごとにカテゴリ分けされています。本書で扱う、特に基礎的なものは次のとおりです。

- **コンピューティング**: 処理を実行する場所。代表は仮想サーバーの **EC2**(第 3 章)。
- **ストレージ**: データを保存する場所。代表はオブジェクトストレージの **S3**(第 3 章)。
- **ネットワーク**: サービス同士をつなぎ、外部との通信を制御する。代表は仮想ネットワークの **VPC**(第 4 章)。
- **データベース**: 構造化したデータを管理する。代表はリレーショナルDBの **RDS**(第 5 章)。

これらは「サーバー・保存・接続・データ管理」というシステムの基本要素にそのまま対応しています。具体的な使い方は後の章で順に学びますが、ここでは「AWS の世界にはこういう棚が並んでいる」というイメージを持てれば十分です。

## マネージドサービスという発想

AWS を理解するうえで欠かせないのが **マネージドサービス** という考え方です。これは、サーバーの構築・障害対応・ソフトウェアの更新といった運用作業の大部分を AWS 側が引き受けてくれるサービスのことです。

たとえばデータベースを自前のサーバーに入れて運用する場合、OS の更新やバックアップ、障害時の復旧まですべて自分で面倒を見る必要があります。一方、マネージドサービスである RDS を使えば、こうした地味で手間のかかる作業の多くを AWS に任せられます。利用者は「データベースをどう設計し、どう使うか」という本来やりたいことに集中できるわけです。

### 注意点

マネージドサービスは便利ですが、「すべてを AWS が面倒を見てくれる」わけではありません。設定ミスやアクセス権限の管理など、利用者側が責任を持つ部分は必ず残ります。この責任の線引きは「責任共有モデル」として第 5 章で詳しく扱います。今は「任せられる部分と、自分で守るべき部分がある」とだけ覚えておきましょう。

## まとめ

- AWS は 200 を超えるサービスの集合体で、役割ごとのカテゴリに分かれている。
- コンピューティング・ストレージ・ネットワーク・データベースが基礎となる代表カテゴリ。
- マネージドサービスは運用の手間を AWS に任せられる一方、利用者側の責任も残る。
