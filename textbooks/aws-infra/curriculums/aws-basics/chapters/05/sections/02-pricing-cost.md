## このセクションで学ぶこと

- AWS の従量課金(使った分だけ支払う)の考え方
- 主な課金要素(コンピューティング・ストレージ・通信)の見方
- コストを把握・抑制するための基本的な仕組み

## 従量課金という考え方

第 1 章で見たように、クラウドの大きな特徴は「所有から利用へ」の転換です。AWS の料金もこれを反映していて、基本は **従量課金** です。サーバーを動かした時間、保存したデータの量、流れた通信量など、**実際に使った分だけ**支払います。使わなければ費用は発生しません。

これは、初期に大きな設備投資をして後はずっと使うオンプレミスとは正反対の発想です。小さく始めて、必要になったら増やし、不要になったら止める——この柔軟さが従量課金の強みです。一方で、止め忘れたリソースがそのまま課金され続ける、という落とし穴も生まれます。

## 主な課金要素

AWS の料金は無数のサービスごとに細かく決まっていますが、初学者はまず次の 3 つの大きな軸で捉えると整理しやすくなります。

- **コンピューティング**: EC2 のようなサーバーを動かした時間に対する料金。インスタンスの性能(CPU・メモリ)が高いほど単価も上がります。
- **ストレージ**: S3 や EBS に保存したデータの量(GB 単位)に対する料金。保存しているだけで毎月かかります。
- **データ転送**: 特に **AWS から外部インターネットへ出ていく通信(アウトバウンド)** に料金がかかります。AWS へ入ってくる通信は基本的に無料です。

意外と見落とされやすいのがデータ転送料金です。「サーバー代は安いはずなのに請求が膨らむ」というケースの多くは、外部への大量の通信が原因です。

なお、長く使い続けることが分かっているサーバーには、一定期間の利用を前提に割引を受けられる仕組み(リザーブドインスタンスや Savings Plans)もあります。常時動かす本番サーバーなどは、こうした割引を活用すると都度払いより費用を抑えられます。まずは従量課金で小さく始め、利用が安定してきたら割引の検討に進む、という順番が実務では分かりやすいでしょう。

## コストを把握・抑制する仕組み

使った分だけ払うからこそ、「今いくら使っているか」を見える化することが重要です。AWS には次のような仕組みがあります。

- **Cost Explorer**: 過去の利用料金をサービス別・期間別にグラフで確認できます。
- **AWS Budgets**: 予算額を決めておき、超えそうになったらメールなどで通知してくれます。
- **無料利用枠(Free Tier)**: 新規アカウントでは一定量まで無料で試せる枠が用意されています。

実務では、まず Budgets で予算アラートを設定し、使っていないリソースをこまめに止める・削除するのが基本です。EC2 を起動したまま放置する、検証用に作った RDS を消し忘れる、といった「止め忘れ」が請求の無駄を生む典型例です。

## まとめ

- AWS は従量課金で、使った分だけ支払う(使わなければ無料)
- 課金はコンピューティング・ストレージ・データ転送の 3 軸で捉えると整理しやすい
- Budgets で予算アラートを設定し、止め忘れをなくすのがコスト管理の基本
