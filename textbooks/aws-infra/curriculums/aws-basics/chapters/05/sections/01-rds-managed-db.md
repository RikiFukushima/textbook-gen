## RDS とは何か

**RDS(Amazon Relational Database Service)** は、リレーショナルデータベースを AWS が運用代行してくれる **マネージドサービス** です。リレーショナルデータベースとは、表(テーブル)の形でデータを管理し、SQL で問い合わせを行う、業務システムでもっとも広く使われている種類のデータベースです。

RDS では MySQL・PostgreSQL・MariaDB・Oracle・SQL Server といった主要なデータベースエンジンを選んで起動できます。利用者はエンジンの種類とサーバーの性能(CPU・メモリ・ストレージ容量)を指定するだけで、数クリックで使えるデータベースが立ち上がります。

第 3 章で扱った EC2 上に自分でデータベースをインストールして運用することもできますが、RDS はその運用負担を肩代わりしてくれる点が大きく違います。

## 自前運用との違い

EC2 にデータベースを自前で入れる場合、OS やデータベースソフトのインストール、セキュリティパッチの適用、定期バックアップの仕組み作り、障害時の復旧手順まで、すべて自分で用意する必要があります。これらは地味ですが手間がかかり、ミスがそのままデータ消失につながります。

RDS は、こうした作業の多くを AWS が自動で行います。代表的なものは次のとおりです。

- **自動バックアップ**: 日次のバックアップと、指定した時点へ巻き戻せる仕組みを標準で提供します。
- **パッチ適用**: データベースエンジンの更新を、指定したメンテナンス時間に自動適用できます。
- **マルチ AZ 配置**: 別のアベイラビリティーゾーンに待機系を置き、障害時に自動で切り替える冗長構成をオプションで選べます。
- **リードレプリカ**: 読み取り専用の複製を増やし、参照の多いアプリの負荷を分散できます。

つまり RDS を使うと、利用者は「データベースの中身(テーブル設計やデータ)」に集中でき、「サーバーやソフトの世話」から解放されます。

## 注意点と使いどころ

便利な一方で、RDS には自由度の制約もあります。OS に直接ログインできない、データベースの細かい設定の一部が変更できない、といった制限があるため、特殊なチューニングや独自拡張が必須の場合は EC2 上での自前運用が向くこともあります。

実務では、特別な理由がなければまず RDS を選ぶのが定石です。運用の手間とミスのリスクを大きく減らせるためです。なお、次のセクションで扱うように、マルチ AZ やリードレプリカは便利な反面、その分だけ料金も増えるため、可用性の要件と費用のバランスを見て構成を決めます。

## まとめ

- RDS はリレーショナルデータベースの運用を AWS が肩代わりするマネージドサービス
- バックアップ・パッチ適用・冗長構成(マルチ AZ)を自動化でき、自前運用の手間とリスクを減らせる
- 自由度の制約はあるが、特別な理由がなければまず RDS を選ぶのが定石
