## このセクションで学ぶこと

- フィードバックを Run に紐づけて記録する意味とデータ構造を理解する
- SDK でユーザー評価(👍/👎やスコア)を Run に送る方法を把握する
- 暗黙的フィードバックと明示的フィードバックの違いと使い分けを理解する

## なぜフィードバックを集めるのか

ダッシュボードのレイテンシやエラー率は「動いているか」を教えてくれますが、「良い回答だったか」は教えてくれません。LLM アプリの品質は最終的に **ユーザーが満足したか**で決まり、それはコードからは分かりません。そこで実際の利用者からの評価、すなわち **フィードバック** を集めて Run に紐づけます。

LangSmith のフィードバックは「どの Run か(`run_id`)」「指標名(key)」「値(score など)」の組で記録されます。たとえば `key="user_thumbs"` に `score=1`(👍)/`0`(👎)を付ける、といった形です。これにより、後から「低評価の回答だけを集めて中身を確認する」「評価の良し悪しでレイテンシやコストを比べる」といった分析ができるようになります。

## 具体例:SDK でフィードバックを送る

アプリ側で回答を返したあと、UI のボタン操作などをトリガーに、対象 Run へフィードバックを送ります。Run の ID は実行時に取得できます。

```python
from langsmith import Client

client = Client()

# ユーザーが 👍 を押したとき
client.create_feedback(
    run_id=run_id,        # 対象の実行 ID
    key="user_thumbs",    # 指標名
    score=1,              # 1=良い / 0=悪い
    comment="期待どおりの回答だった",  # 任意の自由記述
)
```

`score` は数値なので、👍/👎 の二値だけでなく 1〜5 の星評価のような連続値も扱えます。`comment` に自由記述を残しておくと、後から低評価の理由を読み解く手がかりになります。

## 明示的フィードバックと暗黙的フィードバック

ユーザーが明示的にボタンを押す **明示的フィードバック**は信頼度が高い一方、押してくれる人は一部に限られます。実際のアプリでは、評価ボタンを押す人は全体の数パーセントということも珍しくありません。これを補うのが **暗黙的フィードバック**で、「回答をコピーした」「会話を続けた」「すぐ離脱した」といった行動から品質を推定し、同じくスコアとして送ります。たとえば「回答をコピーした」は役に立った可能性が高いシグナル、「やり直しを何度も求めた」は不満のシグナルとして扱えます。両者を併用すると、少数だが確度の高い明示評価と、多数だがノイズを含む行動シグナルの両面から品質を捉えられます。

## 注意点

フィードバックは対象の `run_id` が正しくないと別の実行に紐づいてしまうため、回答を返した実行の ID を確実に保持しておくことが重要です。また、`comment` に個人情報や機微な入力をそのまま転記しないよう、収集時にマスキングを検討してください。

## まとめ

- フィードバックは run_id・key・score の組で Run に紐づけ、品質を後から分析可能にする。
- SDK の `create_feedback` で 👍/👎 やスコア、自由記述を対象 Run に送れる。
- 明示的フィードバックと行動由来の暗黙的フィードバックを併用すると品質を広く捉えられる。
