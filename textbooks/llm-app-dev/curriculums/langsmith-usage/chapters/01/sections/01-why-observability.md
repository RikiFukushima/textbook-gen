## このセクションで学ぶこと

- LLM アプリが従来のソフトウェアと違って「非決定的」で中身が見えにくいことを理解する
- 観測性(observability)が解決する具体的な困りごとを把握する
- LangSmith がその課題にどう答えるサービスかをつかむ

## LLM アプリは「動いているのに、なぜそうなったか分からない」

従来のソフトウェアは、同じ入力に対して必ず同じ出力を返す決定的な振る舞いをします。バグが出れば、ログを追い、ブレークポイントを置けば原因にたどり着けます。ところが LLM を組み込んだアプリは事情が違います。LLM は確率的に応答を生成するため、同じ質問でも実行のたびに答えが変わる「非決定的」な性質を持っています。

さらに、最近のアプリは LLM を単体で呼ぶのではなく、プロンプトの組み立て・検索(RAG)・ツール呼び出し・複数ステップのエージェント推論といった複数の処理を連鎖させています。最終的な回答がおかしいとき、原因が「検索した文書が悪かった」のか「プロンプトの指示が弱かった」のか「モデルが指示を無視した」のかは、出力を眺めているだけでは切り分けられません。

## 観測性が解決する困りごと

ここで必要になるのが**観測性(observability)**です。観測性とは、システムの内部で何が起きたかを、外から記録した情報を通じて後から追える度合いのことです。LLM アプリにおける観測性とは、具体的には次のような問いに答えられる状態を指します。

- ユーザーの入力に対して、最終的に LLM へ渡されたプロンプトは何だったのか
- RAG はどの文書を何件取ってきたのか
- エージェントはどのツールを、どんな引数で呼んだのか
- 各ステップに何秒かかり、トークンを何個消費し、いくらコストがかかったのか
- どのステップでエラーや想定外の出力が起きたのか

print デバッグでこれらを全部ログに出すこともできますが、ステップが増えるほど追いきれなくなり、本番環境では現実的ではありません。

## LangSmith はこの課題に答えるためのプラットフォーム

**LangSmith** は、LangChain 社が提供する LLM アプリ向けの観測・評価・運用プラットフォームです。アプリの実行を「トレース」として自動で記録し、各ステップの入出力・レイテンシ・トークン・コストを階層構造で可視化します。さらに、記録したデータをそのまま評価用のデータセットに育てたり、本番トラフィックを継続的に監視したりできます。

注意したいのは、LangSmith は「あれば便利」な分析ツールというより、LLM アプリを**まともに運用・改善するための土台**だという点です。観測できないものは改善できません。次のセクションでは、LangSmith が提供する主要機能の全体像を俯瞰します。

## まとめ

- LLM アプリは非決定的で多段構成のため、出力を見るだけでは原因が分からない
- 観測性とは、内部で何が起きたかを記録から後追いできる度合いのこと
- LangSmith はトレース・評価・モニタリングを通じてその観測性を提供する土台である
