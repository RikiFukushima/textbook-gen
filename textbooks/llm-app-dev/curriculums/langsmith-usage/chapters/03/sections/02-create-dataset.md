## このセクションで学ぶこと

- データセットが「入力 / 参照出力」の Example の集まりであることを理解する
- UI・トレースからの取り込み・SDK の 3 つの作成方法を使い分けられる
- 良いデータセットの設計指針(代表性・エッジケース・参照出力の扱い)を押さえる

## データセットと Example の構造

評価を回すには、まず「何を入力したら、どうあってほしいか」を集めたデータセット(Dataset)が要ります。データセットは Example の集まりで、各 Example は入力(inputs)と、任意で参照出力(outputs)を持ちます。参照出力はその入力に対する正解・模範解答で、これがあると「出力が正解にどれだけ近いか」を測れます。一方、正解を一つに定めにくいタスク(自由記述の要約など)では参照出力なしで作り、後述の LLM-as-judge などで品質を判定します。

つまりデータセットは、テストコードにおけるテストケース集に相当します。一度作っておけば、プロンプトやモデルを変えても同じ入力群で繰り返し評価でき、品質の変化を比較できます。

## 3 つの作り方

LangSmith でデータセットを作る経路は主に 3 つあります。

- **UI で手入力**: 少数の代表ケースを画面から直接登録します。最初の数件を素早く用意するのに向きます。
- **トレースから取り込む**: 実際のアプリ実行(第 2 章で見た Run / Trace)の入出力を、UI のボタンや SDK でそのまま Example に追加します。本番の生データを評価対象にできるのが強みです。気になる失敗ケースを見つけたらデータセットに足す、という運用が回せます。
- **SDK でまとめて作る**: 既存の正解データや CSV から一括登録します。コードでバージョン管理したい場合に向きます。

SDK での作成は次のように書けます。

```python
from langsmith import Client

client = Client()

dataset = client.create_dataset(
    dataset_name="support-summaries",
    description="問い合わせ要約ボットの評価用データセット",
)

client.create_examples(
    dataset_id=dataset.id,
    examples=[
        {
            "inputs": {"text": "注文がまだ届きません。今日中に発送状況を知りたいです。"},
            "outputs": {"summary": "未着の注文について発送状況の確認を希望"},
        },
        {
            "inputs": {"text": "先日購入した商品を返品したいのですが手続きを教えてください。"},
            "outputs": {"summary": "購入商品の返品手続きの案内を希望"},
        },
    ],
)
```

## 良いデータセットの設計指針

データセットの価値は件数より中身で決まります。実運用で来る入力を反映した代表性のある例を入れること、そして見落としがちなエッジケース(空文字・極端に長い入力・多言語・攻撃的な文面など)を意識的に混ぜることが重要です。参照出力は、後で評価器に合わせて「完全一致を求めるのか、意味が合っていれば良いのか」を考えて粒度を決めます。

## 注意点:データセットは育てるもの

最初から完璧なデータセットは作れません。本番トレースから失敗例を継続的に追加し、修正したバグは Example として残す、という形で育てていきます。なお、顧客の生データを取り込むときは個人情報の扱いに注意し、必要なら匿名化してから登録します。

## まとめ

- データセットは入力 / 参照出力を持つ Example の集まりで、評価のテストケース集に当たります。
- UI 手入力・トレースからの取り込み・SDK 一括登録を場面に応じて使い分けます。
- 代表性とエッジケースを意識し、本番トレースから継続的に育てるのが実務の基本です。
