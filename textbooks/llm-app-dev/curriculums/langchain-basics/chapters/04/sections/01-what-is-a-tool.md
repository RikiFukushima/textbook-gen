## このセクションで学ぶこと

- LLM 単体では外部の世界に手を伸ばせないという制約を理解する
- Tool が「LLM から呼べる関数」であることと、その構成要素(名前・説明・引数スキーマ)を説明できる
- LangChain で関数を Tool 化する基本的な書き方を理解する

## LLM は「閉じた箱」である

ChatModel に「今日の東京の天気は?」と聞いても、正確な答えは返ってきません。LLM は学習時点までのテキストから言葉を生成しているだけで、リアルタイムの情報にアクセスしたり、計算を厳密に実行したり、社内データベースを検索したりはできないからです。LLM はあくまで「閉じた箱」の中で文章を作る存在だと考えてください。

この箱の外とつなぐための仕組みが **Tool** です。Tool とは、ひとことで言えば「LLM から呼び出せる関数」です。天気 API を叩く関数、電卓の関数、社内 DB を検索する関数 ── こうした処理を Tool として渡しておくと、LLM は「自分では計算できないから、この電卓ツールを使おう」と判断して、外部の処理を呼び出せるようになります。

## Tool を構成する 3 つの要素

LLM が「いつ・どのツールを・どんな引数で」呼ぶべきかを判断するには、ツール側がそれを説明する情報を持っていなければなりません。Tool は次の 3 要素で構成されます。

- **名前(name)**: ツールの識別子。`get_weather` のように処理内容がわかる名前を付けます。
- **説明(description)**: 「このツールは何をするものか」を自然言語で書いた説明文。LLM はこの説明を読んでツールを選ぶため、ここの質が呼び出し精度を左右します。
- **引数スキーマ(args schema)**: ツールが受け取る引数の名前・型・意味。たとえば `city: str(都市名)` のように定義します。

LangChain では `@tool` デコレータを使うと、普通の Python 関数を簡単に Tool 化できます。関数名がツール名に、docstring が説明文に、型ヒントが引数スキーマに、それぞれ自動でマッピングされます。

```python
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """指定した都市の現在の天気を返す。"""
    # 実際にはここで天気 API を呼び出す
    return f"{city} は晴れ、気温 22 度です。"

print(get_weather.name)         # get_weather
print(get_weather.description)  # 指定した都市の現在の天気を返す。
print(get_weather.args)         # {'city': {'title': 'City', 'type': 'string'}}
```

## 注意点: 説明文は LLM への「仕様書」

ここで初学者がつまずきやすいのが、docstring を「人間向けのコメント」と軽く考えてしまうことです。Tool の説明文は **LLM がツールを選ぶための仕様書**であり、ここが曖昧だと LLM は誤ったツールを呼んだり、必要なときに呼ばなかったりします。「何をするか」「いつ使うべきか」を具体的に書くことが、後段の Agent の精度に直結します。

なお、関数を `@tool` で囲んだだけでは LLM が自動的に呼ぶわけではありません。ツールを「定義する」ことと、LLM に「渡して呼ばせる」ことは別の段階です。実際にどう呼び出されるかは次のセクションで見ていきます。

## まとめ

- LLM 単体は外部情報・計算・検索ができない「閉じた箱」で、それを補うのが Tool。
- Tool は「名前・説明・引数スキーマ」を持つ、LLM から呼べる関数。
- 説明文(docstring)は LLM がツールを選ぶ仕様書であり、品質が呼び出し精度を決める。
