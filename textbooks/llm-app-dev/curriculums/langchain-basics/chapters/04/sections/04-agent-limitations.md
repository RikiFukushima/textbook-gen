## このセクションで学ぶこと

- 自律的なループがもたらす制御性の低下(ループ回数・コスト・経路の予測しにくさ)を説明できる
- Agent のデバッグが難しい理由を理解する
- 実務で Agent を扱うときの基本的な防御策(反復上限・観測・段階的な権限)を把握する

## 自律性の代償: 何が起きるか読みにくい

前のセクションで見たとおり、Agent は「思考 → ツール実行 → 観測」を **自分で何回でも繰り返せる**のが強みでした。ところがこの自律性こそが、実務で最初にぶつかる壁になります。

Chain(第 2 章)であれば、入力が決まれば通る経路もコストもほぼ予測できました。しかし Agent は、同じ依頼を投げても **毎回違う経路をたどる**ことがあります。具体的には次のような厄介さが生まれます。

- **ループ回数が読めない**: 2 往復で終わることもあれば、ツール選択を誤って何度も呼び直すこともある。
- **コスト・レイテンシが膨らむ**: ループのたびに LLM を呼ぶので、往復が増えるほど料金も応答時間も増える。
- **暴走のリスク**: 終了条件にたどり着けず、同じツールを延々と呼び続けることがある。

## デバッグが難しい理由

普通のプログラムなら、バグはコードのどこかにあり、同じ入力なら同じ結果が再現します。Agent はそうはいきません。失敗の原因が **LLM の思考そのもの**にあるため、「なぜそのツールを選んだのか」がコードを読んでも分からないのです。

しかも LLM の出力は確率的なので、**同じ依頼でも成功したり失敗したりします**。再現性が低いと、原因の切り分けが一気に難しくなります。「さっきは動いたのに」という状況が頻発するわけです。

だからこそ、Agent では **実行の中身を記録して可視化すること(可観測性)**が欠かせません。どの順でツールを呼び、何を観測し、どこで判断を誤ったのか——このトレースを後から追えるようにしておくことが、改善の出発点になります。LangChain エコシステムの LangSmith は、まさにこのトレースの可視化を担うツールです。

## 実務での基本的な防御策

Agent を本番で使うときは、自律性を野放しにせず **手綱を付ける**のが定石です。最低限おさえたい三つを挙げます。

- **反復上限を設ける**: ループ回数に上限(recursion limit)を設定し、暴走を物理的に止める。
- **トレースを取る**: 思考・ツール呼び出し・観測を記録し、失敗時に原因を追えるようにする。
- **権限を絞る**: 副作用のあるツール(削除・送金・メール送信など)はいきなり実行させず、確認を挟むか読み取り専用から始める。

```python
# 反復上限を設定して暴走を防ぐ例
result = agent.invoke(
    {"messages": [("user", "...")]},
    config={"recursion_limit": 10},
)
```

これらは「Agent を信用しすぎない」ための仕組みです。自律性が高いほど、外側のガードレールが重要になります。

## まとめ

- Agent の自律性は、ループ回数・コスト・経路が読みにくいという制御性の低さと裏表になっている。
- LLM の思考が確率的で再現性が低いため、可観測性(トレースの記録・可視化)がデバッグの前提になる。
- 実務では反復上限・トレース・権限の制限といったガードレールで自律性に手綱を付ける。
