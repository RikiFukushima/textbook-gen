## このセクションで学ぶこと

- LCEL の Chain が「一方向に流れる」前提で設計されていることを理解する
- Agent の実行ループが「自由すぎて制御しにくい」という課題を押さえる
- 状態・分岐・ループ・人間の介在が必要になると Chain/Agent では苦しくなることを把握する

## Chain は「一直線」が前提

第 2 章で学んだ LCEL の **Chain** は、`prompt | model | parser` のようにコンポーネントを左から右へパイプでつなぐものでした。入力を渡すと出力が返る、この **一方向に流れる**(線形な)処理がチェーンの本質です。シンプルで読みやすく、デバッグもしやすいのが強みです。

ところが実務では、「一直線では書けない」要求がすぐに出てきます。たとえば次のようなケースです。

- LLM の出力をチェックして、**品質が低ければもう一度やり直す**(ループ)
- ユーザーの質問の種類によって、**処理を枝分かれさせる**(分岐)
- 途中結果を保存しておき、**後のステップで参照する**(状態の保持)
- 重要な操作の前に、**人間に確認を取ってから先に進む**(人間の介在)

LCEL でも `RunnableBranch` で簡単な分岐は書けますが、「条件によって前のステップに戻る」「複数ステップにまたがって状態を持ち回る」といった処理を線形なパイプで表現しようとすると、すぐに複雑で読みにくいコードになります。

## Agent は「自由すぎて」制御しにくい

では第 4 章の **Agent** ならどうでしょうか。第 4 章で使った `create_react_agent` に代表される Agent の実行ループは、「思考 → ツール実行 → 観測」を LLM の判断で繰り返すので、ループも分岐も LLM がやってくれます(なお、初期の LangChain ではこの実行ループを **AgentExecutor** が担っていました。現在は legacy 扱いで、後継が `create_react_agent`/LangGraph です)。柔軟ではありますが、今度は逆の問題が起きます。

```text
ユーザー: 注文をキャンセルして返金して
Agent: (考える) まず注文を検索しよう → search_order 実行
Agent: (考える) 見つかった、ではキャンセル → cancel_order 実行
Agent: (考える) 返金処理しよう → refund 実行  ← 確認なしで実行してしまう
```

このように、**どのツールをいつ呼ぶか、いつ止まるかをすべて LLM 任せ**にしているため、「返金の前に必ず人間の承認を挟む」「同じツールを 3 回以上呼んだら打ち切る」といった**決まった制御を差し込みにくい**のです。動きが毎回変わるためデバッグも難しく、第 4 章の最後で見た「制御性・デバッグの難しさ」がここに現れます。

## 足りなくなるのは「状態を持った複雑な流れ」

整理すると、Chain は **制御しやすいが柔軟性に欠け**、Agent は **柔軟だが制御しにくい**、という両極端です。その中間、つまり「**自分で流れを決めたいが、状態・分岐・ループ・人間の介在も扱いたい**」という領域こそが、Chain でも Agent でも書きにくい部分です。

注意したいのは、これは「Chain/Agent が劣っている」という話ではない点です。単純な変換やシンプルな自律処理なら Chain/Agent で十分です。あくまで**作りたいものが複雑な状態遷移を含むようになったとき**に限界が見えてくる、ということです。次のセクションでは、この限界に応える LangGraph の考え方を見ていきます。

## まとめ

- Chain は一方向に流れる線形処理が前提で、ループや状態の持ち回りには向きません。
- Agent は柔軟ですが流れが LLM 任せで、決まった制御や人間の介在を差し込みにくいです。
- 状態・分岐・ループ・人間の介在を含む複雑な流れになると、Chain/Agent では苦しくなります。
