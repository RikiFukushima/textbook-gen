## このセクションで学ぶこと

- 代表的なユースケースごとに LangChain と LangGraph のどちらが向くかを判断できる
- 「まず LangChain、複雑になったら LangGraph」という現実的な進め方を理解する
- 選定時に陥りやすい誤り(過剰設計・置き換え誤解)を避けられる

## ユースケースで当てはめてみる

前のセクションの境界線(状態・分岐・ループ・人間の介在)を、具体的なユースケースに当てはめてみましょう。判断に迷ったら、「この処理は一直線で書けるか?」を最初に問うのが近道です。

| ユースケース | 向くもの | 理由 |
| --- | --- | --- |
| 文章の要約・翻訳 | LangChain | 入力→出力の一直線で完結する |
| 1 回限りの RAG 質問応答 | LangChain | 検索→文脈注入→生成の線形チェーン |
| 質問の種類で簡単に振り分け | LangChain | `RunnableBranch` で十分まかなえる |
| 自己修正する文章生成(品質 NG なら書き直し) | LangGraph | 条件で前のノードに戻るループが必要 |
| 承認を挟む業務自動化(返金・発注など) | LangGraph | 人間の介在(中断・再開)が必須 |
| 複数エージェントの協調・長い対話の管理 | LangGraph | 状態を持ち回り、流れを明示制御したい |

線を引くと、**単発で完結する処理は LangChain、状態を持って流れが折り返す処理は LangGraph**という傾向がはっきり見えてきます。

## 現実的な進め方: まず小さく、必要になったら移す

実務では、最初から LangGraph で大きく設計するより、**まず LangChain(LCEL)で素直に書き始める**のが堅実です。要約や RAG のような単純な処理ならそのまま完成しますし、コードも短く保守しやすいからです。

そして開発を進めるうちに、「やり直しのループを入れたい」「承認ステップを挟みたい」といった要求が出てきたら、その時点で **LangGraph に載せ替える**ことを検討します。第 2 セクションで見たとおり LangGraph のノードの中身は LangChain の部品なので、書き直しは骨格部分だけで済み、移行のコストは大きくありません。

## 注意点: 最初から作り込みすぎない

ありがちな失敗は、「将来複雑になるかもしれない」と考えて、単純な処理にまで LangGraph を持ち込む**過剰設計**です。状態グラフは制御力が高い反面、ノードやエッジの定義というボイラープレートが増え、一直線で済む処理にはかえって重くなります。

もう一つの誤解は、「LangGraph は LangChain の上位互換だから乗り換えるべき」という思い込みです。両者は置き換え関係ではなく**レイヤーの違い**でした。迷ったら「いま本当に状態・ループ・人間の介在が要るか」に立ち返り、要らなければ LangChain のままにしておくのが正解です。

## まとめ

- 一直線で完結する処理は LangChain、状態を持って折り返す処理は LangGraph が向きます。
- まず LangChain で書き始め、複雑な制御が必要になった時点で LangGraph に載せ替えます。
- 単純な処理への過剰設計や「上位互換」という誤解を避け、必要性で判断します。
