## このセクションで学ぶこと

- LLM を API で直接呼ぶ「素朴な実装」がどこで破綻するかを理解する
- プロンプト管理・外部知識・複数ステップ連携など、共通して発生する課題を整理する
- LangChain のようなフレームワークが「何を肩代わりしてくれるのか」を把握する

## LLM を直接叩くのは意外と簡単、だが…

LLM を使ったアプリ開発は、最初の一歩だけならとても簡単です。OpenAI や Anthropic が提供する API に対して、API キーを添えてプロンプト(指示文)を投げれば、数行のコードで応答が返ってきます。「ユーザーの質問をそのまま LLM に渡し、返ってきた文章を画面に表示する」程度のチャットボットであれば、フレームワークは不要です。

しかし、実務で「使えるアプリ」に育てようとした途端、素朴な実装はあちこちで破綻し始めます。よく出てくる課題を整理してみましょう。

- **プロンプトの管理が煩雑になる**: 役割設定・入力データ・出力形式の指定などを文字列連結で組み立てると、変数の差し込みや使い回しがすぐに散らかります。
- **モデルを差し替えにくい**: OpenAI から別ベンダーのモデルへ乗り換えたいとき、API の呼び出し方が違うとコードを大きく書き直すことになります。
- **外部知識を渡せない**: LLM は学習時点の知識しか持たず、社内ドキュメントや最新情報を知りません。これを補うには、関連文書を検索してプロンプトに差し込む仕組み(後の章で扱う RAG)が必要です。
- **複数ステップの連携がつらい**: 「要約してから翻訳する」「検索結果を踏まえて回答する」のように処理を数珠つなぎにすると、途中結果の受け渡しや出力形式の変換を毎回手書きすることになります。
- **構造化出力の扱いが面倒**: LLM は基本的に文字列を返すため、JSON など決まった形式で受け取りたい場合は、指示の工夫とパースの後処理が必要です。

## 具体例:回答に社内文書を踏まえさせたい

たとえば「社内規定について答えるチャットボット」を作るとします。素朴な実装では、規定の全文を毎回プロンプトに貼り付ける必要がありますが、文書が長いとトークン上限を超えますし、無関係な部分まで渡すと精度もコストも悪化します。

現実的には「質問に関連する箇所だけを検索して渡す」「検索 → プロンプト組み立て → LLM 呼び出し → 整形」という複数ステップを組む必要があります。これを毎回ゼロから書くと、プロジェクトごとに似たコードが量産され、保守が難しくなります。

## フレームワークが肩代わりしてくれること

LangChain のようなフレームワークは、こうした「LLM アプリでほぼ必ず発生する処理」を共通部品として提供します。プロンプトの組み立て、モデルの抽象化(差し替えやすさ)、外部知識の検索連携、複数ステップの連結などを、決まった作法でつなげられるようにしてくれます。

つまりフレームワークの価値は「LLM を呼ぶこと」そのものではなく、**LLM を中心に据えたアプリを組み立てる際の周辺処理を標準化し、再利用可能にすること**にあります。次のセクションでは、その部品が具体的にどう分かれているのかを俯瞰します。

## 注意点:小さなアプリには過剰なこともある

一方で、フレームワークは万能ではありません。「API を一回叩くだけ」のような小さな用途では、フレームワークの抽象化がかえって学習コストや見通しの悪さを生むこともあります。フレームワークはあくまで「複雑さが増えたときに効いてくる道具」だと捉え、規模や要件に応じて採否を判断するのが実務的です。

## まとめ

- 素朴な LLM 実装は、プロンプト管理・モデル差し替え・外部知識・複数ステップ連携で破綻しやすい。
- LangChain は、LLM アプリで頻出する周辺処理を共通部品として標準化・再利用可能にする。
- フレームワークは複雑さが増えたときに効く道具で、小規模用途では過剰なこともある。
