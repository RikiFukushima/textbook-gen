## このセクションで学ぶこと

- AgentExecutor が「ループを内部に隠した」実行器であることを理解する
- 分岐・再試行・途中介入といった制御を後から差し込みにくい理由を説明できる
- 制御フローを自分の手で持ちたくなる場面を具体例で把握する

## AgentExecutor は便利だが「中身が見えない」

LangChain で Agent を組むとき、多くの人が最初に使うのが `AgentExecutor` です。LLM に「使えるツールの一覧」と「ユーザーの質問」を渡すと、内部で次のループを自動で回してくれます。

1. LLM が「次にどのツールを、どんな引数で呼ぶか」を考える
2. そのツールを実行する
3. 実行結果を LLM に戻し、もう一度考えさせる
4. 「もう十分」と LLM が判断したら最終回答を返す

このループを自分で書かずに済むのが `AgentExecutor` の魅力です。プロトタイプを作る段階では、数行で「ツールを使えるエージェント」が動いてしまいます。

しかし便利さの裏返しとして、**ループのロジックが実行器の中に隠れている**という性質があります。「何回ツールを呼ぶか」「どんなときに止めるか」「途中でどう分岐するか」は LLM の判断に委ねられ、開発者からは中身が見えにくいブラックボックスになります。

## 実務で「制御を握りたくなる」場面

プロトタイプから実務アプリへ進むと、この隠れたループが次々と壁になります。具体的な例を挙げます。

- **条件分岐させたい**: 「ユーザーが管理者なら DB を更新するツールに進み、一般ユーザーなら閲覧専用ツールに進む」。この分岐を LLM の気まぐれではなく、こちらが定義した条件で確実に切り替えたい。
- **再試行をループさせたい**: 生成したコードがテストに失敗したら、エラーメッセージを添えてもう一度生成させる。成功するまで、ただし最大 3 回まで反復したい。
- **途中で人間を挟みたい**: 高額な決済や外部メール送信の直前で実行を止め、人間の承認を待ってから続行したい(human-in-the-loop)。

```python
# AgentExecutor は「考える→使う」のループを内部に隠して回す
agent_executor = AgentExecutor(agent=agent, tools=tools)
result = agent_executor.invoke({"input": "在庫を確認して発注して"})
# 何回ツールを呼んだか、どこで分岐したかは外から制御しにくい
```

これらはいずれも「**どこで、どんな条件で、次に進むか**」という制御フローの問題です。`AgentExecutor` はこの制御を LLM の内部判断に押し込んでいるため、外から確実に差し込むのが難しいのです。

## 注意点 — AgentExecutor が悪いわけではない

誤解しないでほしいのは、`AgentExecutor` が劣っているという話ではない点です。「ツールを自律的に使わせたいだけ」のシンプルな用途では、むしろ最短で動く優れた選択肢です。問題になるのは、**フローの形を自分で設計し、確実に再現したくなったとき**だけです。

## まとめ

- `AgentExecutor` は「考える→ツールを使う」のループを内部に隠して回す便利な実行器。
- 分岐・再試行・人間の介在など、確実な制御フローが必要になると中身が見えにくく差し込みにくい。
- シンプルな自律エージェント用途では今でも有効。制御を握りたくなった時が見直しの合図。
