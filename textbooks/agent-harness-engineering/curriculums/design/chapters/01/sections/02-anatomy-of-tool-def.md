## このセクションで学ぶこと

- ツール定義は名前・説明・入力スキーマの 3 要素から成る
- 入力スキーマは JSON Schema で引数の型と必須項目を宣言する
- 定義はモデルへの仕様書であり、harness にとっての契約でもある

## ツール定義の 3 要素

前節で「ツールはモデルの手だ」と述べました。では、そのツールをモデルにどう伝えるのでしょうか。答えは**ツール定義**です。ツール定義は、ほぼどの SDK でも次の 3 つの要素で構成されます。

- **名前(name)**: ツールを一意に指す識別子。`read_file` や `search_web` のように、何をするかが一目でわかる動詞句が望ましい。
- **説明(description)**: そのツールが何をするもので、いつ使うべきかを自然言語で書いた文。
- **入力スキーマ(input schema)**: ツールが受け取る引数の型・必須項目・各引数の意味を構造化して宣言したもの。

たとえば Claude のツール定義は、おおむね次のような形をしています。

```json
{
  "name": "get_weather",
  "description": "指定した都市の現在の天気を取得する。ユーザーが天気を尋ねたときに使う。",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": { "type": "string", "description": "都市名(例: Tokyo)" },
      "unit": { "type": "string", "enum": ["celsius", "fahrenheit"] }
    },
    "required": ["city"]
  }
}
```

## スキーマは「引数の契約」

入力スキーマは多くの場合 **JSON Schema** で書きます。上の例では、`city` は文字列で必須、`unit` は `celsius` か `fahrenheit` のどちらかという制約を宣言しています。これにより、モデルが生成する引数の形が事前に定まります。

スキーマには 2 つの役割があります。1 つはモデルへの**仕様の提示**で、「このツールにはこういう引数を渡せばよい」とモデルに教えます。もう 1 つは harness にとっての**契約**で、モデルが返した引数が型に合っているかを検証する根拠になります。`required` を満たさない呼び出しや型違いは、実行前にはじけます。

各プロパティに添えた `description` も見逃せません。上の例の `"都市名(例: Tokyo)"` のように、引数ごとに意味や書式の例を示しておくと、モデルは値を正しく組み立てやすくなります。スキーマは型だけでなく、引数の**意味**まで伝える場でもあるのです。`enum` で取りうる値を限定したり、必須・任意を `required` で分けたりすることも、モデルの迷いを減らす設計の一部になります。

## 注意点 — description は飾りではない

3 要素のうち、つい軽視されがちなのが description です。スキーマは型を縛りますが、「どんなときにこのツールを使うべきか」は型では表せません。それを担うのが description です。モデルはこの説明文を読んでツールを選ぶため、説明が曖昧だと選択を誤ります。なぜ description がそれほど効くのかは次節で詳しく扱いますが、ここでは「**定義の 3 要素はどれも欠かせない**」とだけ押さえてください。

## まとめ

- ツール定義は名前・説明・入力スキーマの 3 要素でできている。
- 入力スキーマ(多くは JSON Schema)が引数の型と必須項目を縛る。
- スキーマはモデルへの仕様であり、harness にとっての検証の契約でもある。
