## このセクションで学ぶこと

- モデルは description を読んで、どのツールを使うかを判断する
- description には「何をするか」だけでなく「いつ使うか・使わないか」を書く
- 説明文の曖昧さや重複が、誤った選択やツールの使いすぎを招く

## モデルは説明文を読んで「選んで」いる

ツールが複数あるとき、モデルはどれを使うかをどう決めているのでしょうか。手がかりは主に **description** です。モデルはツール定義の説明文を読み、いまの状況に最も合うものを選びます。つまり description は単なるドキュメントではなく、**モデルの行動を方向づけるプロンプトの一部**として働いています。

ここが設計上の勘所です。スキーマがどれだけ正確でも、description が曖昧なら選択を誤ります。たとえば `search` というツールに「検索する」とだけ書いてあったら、モデルは Web 検索なのか社内文書検索なのかコード検索なのか区別できません。説明文の質が、そのままツール選択の質になります。

## 良い説明文は「いつ使うか」を語る

description には「何をするか」だけでなく、**いつ使うべきか・いつ使わないべきか**を書くのが効きます。次の 2 つを比べてください。

```text
悪い例:
  search_docs: "ドキュメントを検索する"

良い例:
  search_docs: "社内の技術ドキュメントを全文検索する。製品仕様や
    社内ルールに関する質問に答えるときに使う。一般的な知識や
    最新ニュースには使わない(その場合は search_web を使う)。"
```

良い例は、対象(社内ドキュメント)・使う場面(製品仕様や社内ルール)・使わない場面(一般知識は別ツール)まで述べています。モデルはこの境界を読んで、`search_docs` と `search_web` を正しく使い分けられます。Claude Code のツール群でも、たとえば「`grep` でなくこの検索ツールを使え」といった**他ツールとの使い分け指示**が説明文に書き込まれており、選択を誘導しています。

## 注意点 — 曖昧さと重複が事故を生む

説明文の典型的な失敗は 2 つあります。1 つは**曖昧さ**で、対象や用途が読み取れないと、モデルは当てずっぽうで選ぶか、必要な場面で使わずに済ませてしまいます。もう 1 つは**重複**です。似たツールが 2 つあって説明文の境界が曖昧だと、モデルはどちらを選ぶか迷い、一貫しない挙動を見せます。

対策はシンプルです。ツールごとに役割の境界を説明文ではっきり分け、「これはこういうときに使い、こういうときは使わない」と書くこと。description はコードのコメントではなく、**モデルに向けた指示文**だと考えると、自然と書き方が変わります。

実務では、説明文を一度書いて終わりにせず、エージェントの挙動を観察しながら直していくことになります。モデルが必要な場面でツールを使わなかったり、別のツールと取り違えたりしたら、それは説明文が境界をうまく伝えられていないサインです。プロンプトをチューニングするのと同じ感覚で、description も繰り返し調整する対象だと捉えてください。

## まとめ

- モデルは description を読んでツールを選ぶ。説明文は実質プロンプトである。
- 「何をするか」に加え「いつ使うか・使わないか」を書くと選択が安定する。
- 曖昧さと重複は誤選択を招く。役割の境界を説明文で明示する。
