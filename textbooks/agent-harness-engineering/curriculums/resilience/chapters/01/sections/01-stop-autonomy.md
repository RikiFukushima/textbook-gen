## このセクションで学ぶこと

- なぜ「止める設計」が自律エージェントの安全性の中心になるのか
- ループの主語は harness であり、止める責任も harness にあること
- 暴走が「事故」になる三つの代表的な被害(コスト・副作用・信頼)

## 自律は「動かす」より「止める」が難しい

エージェントを動かすだけなら難しくありません。モデルに次の行動を考えさせ、その行動を実行し、結果を観測して、また考えさせる。この推論→行動→観測のサイクルを回し続ければ、見た目には「自律的に動くエージェント」が完成します。

問題は、このループが **いつ・どうやって止まるか** です。foundations カリキュラムで確認したとおり、ループを回している主語はモデルではなく harness です。モデルは「次に何をするか」を提案するだけで、その提案を実行するか、もう一周回すか、ここで打ち切るかを決めているのは harness のコードです。つまり **止める責任は harness 側にあります**。モデルが「終わったつもり」でも harness が回し続ければ止まりませんし、逆に harness が打ち切ればモデルがどれだけ続けたくても止まります。

止める設計を後回しにすると、エージェントは「とりあえず動くが、いつ終わるか保証できない」ものになります。デモでは動いても、本番に出した瞬間に事故を起こします。

## 暴走が「事故」になる三つの被害

止まらない自律がなぜ「事故」なのか。被害は大きく三つに分かれます。

第一に **コスト** です。ループが回るたびにモデルへの API 呼び出しが発生し、トークンを消費します。1 周あたり数円でも、止まらなければ数千周し、気づけば請求が膨れ上がります。深夜に放置したエージェントが朝には高額請求になっていた、という事故は珍しくありません。

第二に **副作用** です。エージェントがファイルを書き換える、メールを送る、外部 API を叩くといった行動は、取り消しにくい変更を外界に残します。ループが暴走すれば、同じファイルを何百回も上書きしたり、同じ通知を連投したりします。読み取りだけなら無駄で済みますが、書き込み系の行動は実害になります。

第三に **信頼** です。一度でも暴走を起こしたエージェントは、ユーザーから「放っておけないもの」と見なされます。自律の価値は「任せられること」にあるのに、止まる保証がなければ常に人が張り付くことになり、自律の意味そのものが失われます。

## 「止める」は機能ではなく前提

注意したいのは、止める仕組みを「あとから足す追加機能」と考えないことです。終了条件・上限・タイムアウトは、ループを書く時点で最初から組み込む **前提条件** です。動くものを作ってから安全装置を付けるのではなく、安全装置のないループはそもそも本番に出さない、という順序で考えてください。この章ではその「止める設計」を、終了条件・無限ループ検知・予算・再計画の順に具体化していきます。

## まとめ

- ループの主語は harness であり、自律を止める責任も harness にある。
- 止まらない自律はコスト・副作用・信頼の三方向で「事故」を起こす。
- 止める仕組みは後付けの機能ではなく、ループを書く時点での前提として組み込む。
