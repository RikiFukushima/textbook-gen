## このセクションで学ぶこと

- 終了条件には「完了による終了」と「打ち切りによる終了」の二種類があること
- 完了判定の主語はモデル、打ち切り判定の主語は harness であること
- 曖昧なゴールを検証可能な終了条件に落とし込む考え方

## 終了には二つの系統がある

ループを止める条件は、性質の違う二つに分けて考えると整理できます。

一つは **完了による終了** です。タスクが目的を達したからもう回す必要がない、という「成功の終わり方」です。foundations で見たとおり、これはモデル発のシグナルで判断します。モデルが「終わった」と表明したら harness がそれを受けてループを抜けます。

もう一つは **打ち切りによる終了** です。上限に達した、エラーが続いた、時間切れになった、という「成功とは限らない終わり方」です。これは harness 発の判断です。モデルがまだ続けたくても、harness が安全のために強制的に止めます。

この二つは責任の所在が違います。完了はモデルが宣言し harness が承認する。打ち切りは harness が単独で決める。どちらが欠けても危険です。完了判定だけだとモデルが永遠に「まだ終わっていない」と言い続ければ止まりません。打ち切りだけだと、成功してもしなくても同じ上限まで回す無駄なエージェントになります。

## 完了シグナルをどう表現するか

完了をモデルに宣言させる典型は **専用ツールを用意する** 方法です。たとえば `finish` や `submit_answer` というツールを定義しておき、モデルがそれを呼んだら harness はループを終了して結果を返します。「自由文で『完了しました』と書く」よりも、ツール呼び出しという構造化されたシグナルにするほうが、harness 側で確実に検知できます。

```python
def run_loop(agent, task):
    for step in range(MAX_STEPS):       # 打ち切り条件(harness)
        action = agent.decide(task)
        if action.tool == "finish":      # 完了シグナル(モデル)
            return action.args["result"]
        observation = execute(action)
        agent.observe(observation)
    return "打ち切り: ステップ上限に到達"   # 打ち切りによる終了
```

この `for` ループは、二系統の終了を一目で表しています。`finish` 検知が完了による正常終了、`MAX_STEPS` への到達が打ち切りによる終了です。

## 曖昧なゴールは検証可能にする

注意点は、ゴールが曖昧なままだと完了判定もできないことです。「いい感じに整理して」のような指示は、モデルがいつ「終わった」と言えるか自分でも判断できません。設計時には、ゴールを **検証可能な形** に落とす努力が要ります。「テストが全て通る」「指定フォーマットの JSON を返す」「3 件以上の候補を出す」のように、達成をコードや観測で確認できる条件にできれば、完了判定が安定します。検証可能なゴールがあると、モデルの「終わった」という自己申告を harness 側で裏取りでき、誤った完了宣言で中途半端な結果を返す事故も防げます。曖昧なゴールほど無限ループに陥りやすい、と覚えておいてください。

## まとめ

- 終了条件は「完了(モデル発)」と「打ち切り(harness 発)」の二系統で設計する。
- 完了シグナルは自由文より専用ツールなど構造化された形で受け取ると確実。
- 曖昧なゴールは検証可能な条件に落とすことで完了判定が安定し、暴走を防げる。
