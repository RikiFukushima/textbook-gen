## このセクションで学ぶこと

- エージェントの失敗は大きく「ツール障害」「モデル誤り」「環境変化」の三種に分けられる
- 失敗の種類ごとに正しい回復手段が異なる
- 同じエラー文面でも原因が違えば対処が変わるため、harness 側で見立てが要る

## なぜ「分類」から始めるのか

design カリキュラムの 01 章で、ツールのエラーをそのままモデルに返す設計を学びました。エラーを隠さずモデルへ渡せば、モデルは次の一手を考え直せます。しかしモデルに丸投げすればよいわけではありません。**何が壊れたのかを harness 側でも見立てておかないと、回復の打ち手を選べない**からです。

エラー回復は「とりあえずもう一度やってみる」だけでは成り立ちません。やり直して直る失敗もあれば、何度やっても同じ結果にしかならない失敗もあります。その違いは、失敗の「種類」で決まります。だからエラー回復の出発点は、失敗を分類することです。

## 三つの失敗

エージェントの失敗は、原因の所在で大きく三種に分けられます。

**ツール障害** は、ツールの実行そのものが失敗するケースです。ネットワークが切れた、外部 API がタイムアウトした、権限が足りずファイルを書けなかった、といった失敗です。モデルの判断は正しかったのに、外界の都合で失敗します。多くは時間をおけば直る可能性があり、リトライが有効な代表例です。

**モデル誤り** は、モデルの出力そのものが間違っているケースです。ツールの引数を取り違える、存在しないツール名を呼ぶ、誤った前提で次の行動を決める、といった失敗です。ここで同じ呼び出しをそのまま再実行しても、モデルが考え直さない限り同じ失敗を繰り返します。回復には「エラーを返して考え直させる」ことが要ります。

**環境変化** は、実行の途中で外界の状態が変わってしまうケースです。読もうとしたファイルが別の処理で消えていた、参照していた値を他のプロセスが書き換えていた、といった失敗です。最初の前提が崩れているので、計画ごと立て直す必要があります。

## 同じ文面でも原因は違う

たとえば「ファイルが見つかりません」というエラー一つとっても、原因は分かれます。モデルがパスを打ち間違えたなら **モデル誤り**、さっきまで存在したファイルが消えたなら **環境変化**、ストレージが一時的に応答しないだけなら **ツール障害** です。表に出るメッセージは同じでも、リトライすべきか・考え直させるべきか・計画から作り直すべきかは正反対になります。

だから harness は、エラーを単に文字列として転送するだけでなく、「これはどの種類に近いか」を見立てる材料(HTTP ステータス、例外の型、リトライ回数など)を持っておくと、回復の打ち手を機械的に絞り込めます。

## まとめ

- 失敗は「ツール障害(外界の都合)」「モデル誤り(出力の間違い)」「環境変化(前提の崩れ)」に大別できる
- 種類ごとに有効な回復手段が違い、リトライ・考え直させる・計画し直すのいずれかに対応する
- 同じエラー文面でも原因が違えば対処は変わるので、harness 側でも原因を見立てる
