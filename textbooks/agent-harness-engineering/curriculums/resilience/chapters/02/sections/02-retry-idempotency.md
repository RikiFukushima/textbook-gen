## このセクションで学ぶこと

- リトライは万能ではなく、「やり直しても害がない操作」にだけ安全に使える
- 冪等な操作は何度実行しても結果が同じなので安心してリトライできる
- 非冪等な操作はそのままリトライすると二重実行の副作用を生む

## リトライは「種類」を選ぶ

前のセクションで、ツール障害は時間をおけば直ることが多く、リトライが有効だと述べました。ネットワークの瞬断や外部 API の一時的なエラーは、何度か待ってやり直すだけで回復します。これがリトライの本来の使いどころです。

ただし、リトライしてよいかどうかは失敗の種類だけでは決まりません。**「その操作をもう一度実行しても害がないか」** も同じくらい重要です。読み取り操作のように、何度実行しても外界を変えない操作なら、安心してやり直せます。問題は、外界を書き換える操作です。

## 冪等な操作は安全にやり直せる

ここで鍵になるのが **冪等性** です。冪等な操作とは、同じ操作を何度実行しても結果が変わらない操作のことです。

たとえば「ファイル `report.txt` の内容を `done` で上書きする」操作は冪等です。一度実行しても十回実行しても、最終状態は `done` のままです。途中で失敗したかどうか分からなくても、もう一度実行すれば正しい状態に収束します。だからリトライしても安全です。

読み取り(GET 相当)、固定値での上書き、「存在しなければ作る」といった操作は冪等になりやすく、リトライと相性が良いものです。

## 非冪等な操作は二重実行を生む

逆に、実行のたびに結果が積み上がる操作は **非冪等** です。

典型例は「請求を 1 件作成する」「メールを送る」「カウンタを 1 増やす」といった操作です。タイムアウトしたので念のためリトライしたら、実は最初の実行は成功していて、結果として請求が二重に作られた——これは現実によく起きる事故です。失敗したように「見えた」だけで、外界はすでに変わっていたわけです。

こうした操作をリトライ可能にするには、二重実行を防ぐ工夫が要ります。代表的なのが **冪等キー** です。操作ごとに一意な ID を付けて送り、サーバ側が「このキーは処理済み」と覚えておけば、同じキーでの再実行は無視されます。これにより、非冪等な操作を見かけ上は冪等に扱えるようになります。重複検知のテーブルを持つ、実行前に「すでに済んでいないか」を問い合わせる、といった方法も同じ狙いです。

## 注意点 — むやみなリトライは事故を増やす

リトライには上限とバックオフ(待機)を必ず付けます。即座に何度もやり直すと、落ちかけた外部サービスにとどめを刺したり、無限ループに近い挙動になったりします(前章のループ制御とも地続きです)。そして、**操作が非冪等なまま自動リトライを仕込むのが一番危険**です。回復のつもりが副作用を増やしてしまいます。

## まとめ

- リトライは「やり直しても害がない操作」にだけ安全に使える回復手段
- 冪等な操作(読み取り・固定値の上書きなど)は何度実行しても結果が同じで、安心してリトライできる
- 非冪等な操作(作成・送信・加算など)は冪等キーや重複検知で守ってからリトライする
