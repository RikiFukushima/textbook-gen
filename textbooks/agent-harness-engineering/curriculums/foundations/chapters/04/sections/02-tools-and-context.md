## このセクションで学ぶこと

- Claude Code のファイル操作・コマンド実行を「ツール」として読む
- 会話が続くほど応答が変わる現象を「コンテキスト」として読む
- 観察できる挙動から harness の 2 要素を裏付ける

## ツール — 「読んだ」「実行した」が画面に出る

モデルそのものはテキストを出力するだけで、ファイルを開いたりコマンドを走らせたりはできません(第 1 章)。にもかかわらず Claude Code はファイルを書き換え、テストを実行します。これを可能にしているのが **ツール** です。

観察してみましょう。「このファイルを直して」と頼むと、Claude Code はまず対象ファイルを **読み**、次に **書き換え**ます。画面には「ファイルを読みました」「編集しました」といった経過が出るはずです。コマンドを実行するときも同様に、何を実行したかが表示されます。これらはモデルが直接やっているのではなく、モデルが「このファイルを読みたい」という意思をツール呼び出しの形で出力し、harness の側が実際の読み書きを代行している、という流れです。

確かめ方は簡単です。存在しないファイルを指定してみると、Claude Code は「見つからなかった」という **ツールからの結果** を受け取り、それを踏まえて次の手を打ちます。エラーが返ってきて反応が変わるなら、そこには確かにツールという外界との接点が働いています。

## コンテキスト — 続けるほど噛み合っていく

もうひとつ、会話を続けてみてください。最初に「この関数の名前は snake_case で」と伝えておくと、後のやりとりでもその方針が引き継がれます。さっき読んだファイルの内容を、改めて貼り直さなくても覚えています。これが **コンテキスト** の働きです。

第 1 章で見たとおり、モデル自体は記憶を持ちません。毎ターン、過去のやりとりや参照したファイルの内容を **まとめて入力に詰め直す** ことで、あたかも会話を覚えているように振る舞わせているのが harness です。だから「さっきの話を踏まえてくれる」のは、モデルが賢いからというより、足場が文脈を運び続けているからだと読めます。

逆に、長い会話の途中で前半の細かい指示が薄れることがあります。これは入力に詰め込める量に限りがあり、何を残し何を落とすかを harness が選んでいるためだと推し量れます。

## 注意点 — 「覚えている」の正体を取り違えない

「Claude Code は記憶力がいい」と感じたら、それはモデルの記憶ではなく **コンテキスト管理の働き** だと言い換えてみてください。この読み替えができると、応答が噛み合わなくなったときに「モデルが悪い」ではなく「文脈が運べていないのでは」と原因を切り分けられます。ツールとコンテキストは、外界に触れる手と、状況を保つ記憶という、補い合う 2 要素です。

## まとめ

- ファイルの読み書きやコマンド実行は、モデルではなくツールが代行する挙動。
- 会話が噛み合い続けるのは、harness がコンテキストを毎ターン運んでいるから。
- 「覚えている」をコンテキストの働きと読み替えると、不調の原因を切り分けやすい。
