## このセクションで学ぶこと

- 逐次表示を「観測」という 5 つ目の要素として読む
- 5 要素のチェックリストで任意のエージェントを分解する手順を持つ
- 観察から harness の設計を推し量る評価の習慣を身につける

## 観測 — そもそも「観察できる」こと自体が一要素

ここまで、Claude Code のツール・コンテキスト・権限・ループを実際の挙動から読んできました。最後の要素である **観測** は、少し特別です。なぜなら、前の 3 つのセクションで私たちが「画面に出る」「経過が表示される」と当たり前のように観察できたこと自体が、観測という要素が働いている証拠だからです。

第 2 章で見たとおり、観測はエージェントが何をしようとし、何をしたかを外から見えるようにする仕組みでした。Claude Code が「このファイルを読みました」「このコマンドを実行します」と逐次表示するのは、自律的に動くループの一部始終を利用者に晒すためです。もしこの表示がなければ、ツールが動いたかもコンテキストが運ばれたかも確かめようがなく、本章の観察は成り立ちませんでした。**観察できるように作られていること** が、そのまま観測という設計判断の現れなのです。

## 5 要素チェックリストで道具を分解する

ここで身につけたいのは、Claude Code に限らず **どんなエージェントでも同じ目で分解する** 習慣です。新しいツールに触れたら、次の 5 つを順に問うてみてください。

```text
□ ツール   何を外界に対して実行できるか(検索・ファイル・API・コマンド…)
□ コンテキスト 何を覚えているように見えるか / どこで忘れるか
□ 権限     何を勝手にやり、何の前で止まって確認するか
□ ループ   一度の指示でどこまで自分で進むか / いつ止まるか
□ 観測     何をしているかをどこまで見せてくれるか
```

このリストを当てると、「賢いか / 賢くないか」という漠然とした印象が、**設計の良し悪し** に分解されます。たとえば「便利だが何をしているか不安」と感じるツールは、たいてい観測が弱い。「すぐ確認で止まって遅い」と感じるなら権限が厳しめ、「途中で話が噛み合わなくなる」ならコンテキストの運びに限界がある、という具合に、不満の正体を要素の名前で言えるようになります。

## 注意点 — 評価はするが、内部実装は断定しない

この読み解き方を使うときの作法を、改めて確認します。チェックリストで見えるのは **外から観察できる挙動** であり、その裏でどんなプロンプトやアルゴリズムが動いているかは確定できません(第 1 セクションの釘刺しのとおり)。だから結論は「観測が弱いツールだ」ではなく「**観測が弱いように見える**」と留めます。

それでも、この姿勢には十分な実用価値があります。第 2 章の最後で見たように、5 要素は支え合っていて一つ欠けると崩れます。観察で「どの要素が手薄か」が見えれば、そのツールがどこで破綻しやすいかを **使う前に予測** できます。harness の目を持つとは、道具を「すごい / すごくない」で受け取るのをやめ、5 つの引き出しに分けて冷静に値踏みできるようになることです。

## まとめ

- 挙動が「観察できる」こと自体が、観測という要素の現れ。
- 任意のエージェントは 5 要素チェックリストで分解でき、漠然とした印象が設計の評価に変わる。
- 見えるのは挙動だけなので断定はせず、「どの要素が手薄か」から破綻しやすさを予測する。
