## このセクションで学ぶこと

- 同じモデルでも、取り囲む harness が違えば挙動も能力も別物になることを理解する
- エージェントの差は「モデルの賢さ」だけでなく「harness の設計」で決まると捉える
- harness を設計対象として見る視点(本書全体の出発点)を持つ

## 同じエンジンでも、車体が違えば別の乗り物になる

前のセクションで、harness をモデルを取り囲む足場だと定義しました。ここで強調したいのは、**同じモデルでも、周りにどんな harness を組むかで、まるで別物のように振る舞う**という事実です。モデルをエンジンに喩えるなら、同じエンジンでも、軽トラックに載せるかレーシングカーに載せるかで、走り方も用途もまったく変わります。AI エージェントの「個性」や「強さ」の多くは、実はエンジン(モデル)ではなく車体(harness)の側から生まれています。

## 同じモデル、違う harness、違う挙動

たとえばまったく同じ LLM を、3 つの異なる harness で囲んでみましょう。

- **会話履歴を渡すだけの harness**: 過去のやり取りは覚えていますが、ファイルもコマンドも触れません。せいぜい賢いチャット相手どまりです。
- **ファイル操作とコマンド実行の道具を与えた harness**: 同じモデルが、コードを読んで書き換え、テストを走らせる「開発者」のように動き出します。
- **道具に加えて、危険な操作だけ人間に確認を求める harness**: 同じモデルが、暴走しにくい「慎重な作業者」になります。

モデルは一文字も変えていないのに、できること・できないこと・安全性がすっかり変わりました。違いを生んだのは、すべて harness の側の設計です。

## 注意点:強いエージェントは「強い harness」から生まれる

ここから本書の出発点となる視点が見えてきます。エージェント同士を比べたとき、その差を「どちらのモデルが賢いか」だけで説明しようとすると、本質を取り逃がします。実際には、**どんな道具を与え、どこまで記憶させ、どこで止め、どう人間と協調させるか**という harness の設計が、能力の差を大きく左右します。Claude Code のようなツールが頼りになるのも、背後のモデルだけでなく、その harness が丁寧に作り込まれているからです。

だからこそ、harness は「勝手にできあがるもの」ではなく、**意図して設計するもの**として捉える必要があります。次章からは、この harness を「ツール・コンテキスト・権限・ループ・観測」という構成要素に分解し、一つずつ設計対象として見ていきます。

## まとめ

- 同じモデルでも、取り囲む harness が違えば挙動も能力も別物になります。
- エージェントの差は、モデルの賢さだけでなく harness の設計から生まれます。
- だから harness は意図して設計する対象であり、それが本書のテーマです。
