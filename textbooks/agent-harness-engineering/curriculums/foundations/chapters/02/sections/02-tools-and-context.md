## このセクションで学ぶこと

- ツールはモデルに「外界へ働きかける手足」を与える
- コンテキストはモデルに「いま何が起きているか」を毎ターン渡す
- ツールとコンテキストは入出力の両輪として対になる

## ツール — 外界に触れる手足

モデルは入力に対して文字列を返すだけで、自分でファイルを読んだり Web を検索したりはできません。**ツールは、その「手足」をモデルに与える要素です。** 検索する、計算する、ファイルを書く、外部 API を叩く――こうした操作をモデルが「呼び出せる」形にしたものがツールです。

仕組みはシンプルです。harness はモデルに「あなたはこういう操作が使えます」とツール定義(名前・説明・引数)を渡しておきます。モデルは必要だと判断すると「このツールをこの引数で呼びたい」という出力を返し、実際に実行するのは harness 側です。実行結果は再びモデルに戻されます。つまりモデルは**何をしたいかを言うだけ**で、外界を実際に動かすのは harness です。この分担が、確率的関数にすぎないモデルに行動の能力をもたらします。

ツールが補うのは、第 01 章で見た「外界に触れられない」という限界です。ツールがなければ、モデルはどれだけ賢くても閉じた部屋の中で言葉を返すだけで終わります。

## コンテキスト — 状況を渡す

ツールが出力側(外界への働きかけ)だとすれば、**コンテキストは入力側です。** モデルは状態を持たないので、毎ターン「いま何が起きているか」を改めて渡してあげなければ、直前のやり取りすら覚えていません。会話履歴、関連文書、システム指示、そしてツールの実行結果――これらをまとめてモデルに渡す入力一式がコンテキストです。

ここで重要なのは、**コンテキストに入っていないものは、モデルにとって存在しない**という点です。どれだけ正しい情報がデータベースにあっても、コンテキストに載らなければモデルは参照できません。逆に、間違った情報や古い履歴が混ざれば、モデルはそれを前提に答えてしまいます。コンテキストは「モデルに見せる世界そのもの」だと考えると掴みやすいでしょう。

コンテキストが補うのは「記憶を保てない」という限界です。harness が履歴や状況を毎回詰め直すことで、状態を持たないモデルが、あたかも文脈を覚えているかのように振る舞えます。

## 具体例 — 両輪として動く

ファイル内のバグを直すエージェントを考えます。まずツール(ファイル読み取り)で対象を読み、その結果がコンテキストに積まれます。次のターン、モデルはそのコンテキストを見て原因を考え、ツール(ファイル書き込み)で修正します。この修正結果もまたコンテキストに戻ります。**ツールで外界から情報を引き込み、コンテキストでモデルに渡し直す**――この往復が両輪として回っているのが分かります。

## 注意点

コンテキストウィンドウには上限があり、無制限に情報を詰められるわけではありません。何を入れ、何を削るかの取捨選択は harness 設計の中心テーマの一つですが、その具体は後続カリキュラム(design)に委ねます。ここでは「ツールが手足、コンテキストが状況」という役割分担を掴めれば十分です。

## まとめ

- ツールはモデルに外界へ働きかける手足を与え、「外界に触れられない」限界を補う。
- コンテキストは毎ターン状況を渡し、「記憶を保てない」限界を補う。
- 両者は入出力の両輪で、コンテキストに無いものはモデルには存在しない。
