# 状態と差分を読む

## このセクションで学ぶこと

- git status で作業状態を把握する
- git diff でステージ前の変更を確認する
- git diff --staged でステージ済みの変更を確認する

## いまどうなっているかを見る

コミット前に「何が変わっていて、何がステージに乗っているか」を確認する習慣は、意図しない変更の混入を防ぎます。そのための基本が **`git status`** です。

```bash
git status
```

実行すると、未追跡のファイル、変更されたがまだステージしていないファイル、ステージ済みでコミット待ちのファイルが、それぞれ分けて表示されます。前セクションまでに学んだ「未追跡 / 追跡対象」「ワークツリー / ステージ」の状態が、ここで一目で確認できます。

## 変更の中身を見る

ファイル名だけでなく、行単位で何が変わったかを見るのが **`git diff`** です。`status` が「どのファイル」を答えるのに対し、`diff` は「どの行をどう変えたか」を答えます。

```bash
git diff                 # まだステージしていない変更
git diff --staged        # ステージ済み（コミット予定）の変更
```

ここで重要なのが 2 つの使い分けです。オプションなしの `git diff` は **まだステージしていない**変更を表示します。一方 `git diff --staged` は **すでに `git add` 済み**で次のコミットに入る変更を表示します。

## コミット前の確認フロー

実務では、コミット前に次の順で確認すると安全です。

1. `git status` で全体像を見る
2. `git diff` で乗せる前の変更を吟味する
3. 必要なものだけ `git add` する
4. `git diff --staged` でコミット内容を最終確認する
5. `git commit` する

この一手間で、「デバッグ用の `console.log` を消し忘れたままコミットしてしまった」「実験用に書き換えた設定を戻し忘れた」といった、ありがちな事故をぐっと減らせます。

慣れてくると `git status` と `git diff` は無意識に打つようになります。それくらい基本的で、かつ頻繁に使うコマンドです。逆に言えば、この 2 つを飛ばして add・commit を繰り返すと、意図しない変更が少しずつ履歴に紛れ込んでいきます。状態と差分を読む力は、これ以降のすべての操作の土台になります。

## まとめ

- `git status` で未追跡・変更・ステージ済みの状態を一覧できる
- `git diff` は未ステージ、`git diff --staged` はステージ済みの差分を表示する
- コミット前に status → diff → add → diff --staged の順で確認すると事故を防げる
