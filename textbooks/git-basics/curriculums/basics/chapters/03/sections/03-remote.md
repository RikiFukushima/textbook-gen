# リモートリポジトリとの連携

## このセクションで学ぶこと

- リモートとローカルの関係
- git push で変更を共有する
- git pull / clone で変更を受け取る

## リモートはもう 1 つの複製

第1章で学んだとおり、Git は分散型なので、**リモートリポジトリ**もローカルと対等な複製の 1 つです。違いは役割で、チーム全員が「ここを正とする」と決めた共有の窓口として使います。GitHub はこのリモートを提供するサービスの代表例です。

既存のリモートから作業を始めるときは **`git clone`** で履歴ごと複製します。

```bash
git clone https://github.com/example/project.git
```

クローンすると、複製元のリモートには自動で **origin** という名前が付き、以降はこの名前で参照できます。

## 変更を送る・受け取る

ローカルで積んだコミットをリモートに送るのが **`git push`**、リモートの最新の変更を受け取るのが **`git pull`** です。

```bash
git push origin main      # ローカルの main を origin に送る
git pull origin main      # origin の main の変更を取り込む
```

ここで第1章の「オフラインで作業できる」が効いてきます。日々のコミットはローカルで完結し、区切りのよいところで push してチームと共有する、という流れになります。push するまで、自分のコミットはまだ誰にも見えていません。

## 共有作業の基本リズム

複数人で 1 つのリモートを使うときは、次のリズムが基本です。

1. 作業前に `git pull` で最新を取り込む
2. ローカルで作業し、コミットを積む
3. `git push` で共有する

特に **1 の「作業前の pull」**が大切です。古い状態のまま作業して push しようとすると、リモートが先に進んでいて `! [rejected]` のようなメッセージで弾かれることがあります。これは事故ではなく、「先に他の人の変更を取り込んでね」という Git からの合図です。その場合は一度 pull し、必要なら前セクションで学んだマージ(時にはコンフリクト解消)を経てから push します。

このリズムを身につけると、第1章で学んだ分散型の利点を実感できます。普段はローカルで自由に作業し、共有したいときだけリモートとやりとりする。pull・作業・push のリズムが、チーム開発の土台になります。

## まとめ

- リモートは役割上の共有窓口で、技術的にはローカルと対等な複製
- `git push` で変更を送り、`git pull` / `git clone` で受け取る
- 「作業前に pull、区切りで push」のリズムがチーム開発の基本
