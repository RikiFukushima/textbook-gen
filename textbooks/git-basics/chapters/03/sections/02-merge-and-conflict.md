# マージとコンフリクト

## このセクションで学ぶこと

- git merge でブランチを合流させる
- コンフリクトが起きる仕組み
- コンフリクトの解消の流れ

## ブランチを合流させる

作業ブランチが完成したら、その変更を本流に取り込みます。これが **`git merge`** です。基本は「取り込み先に移動してから、取り込みたいブランチを指定する」順になります。

```bash
git switch main           # 取り込み先(main)に移動
git merge feature         # feature の変更を main に取り込む
```

変更箇所が重ならなければ、Git は自動でうまく合流させてくれます。ここまでが理想的なケースです。

## コンフリクトはなぜ起きるか

問題は、**2 つのブランチが同じファイルの同じ箇所を別々に変更していた**ときです。Git はどちらを採用すべきか判断できず、**コンフリクト(衝突)**として人間に判断を委ねます。

たとえば `main` で 1 行目を「こんにちは」に、`feature` で同じ 1 行目を「Hello」に変更していたら、Git はどちらが正しいか分かりません。これは Git の不具合ではなく、人間にしか決められない判断を正しく差し戻している状態です。

## コンフリクトを解消する

コンフリクトが起きると、該当ファイルに**コンフリクトマーカー**が挿入されます。

```text
<<<<<<< HEAD
こんにちは
=======
Hello
>>>>>>> feature
```

`<<<<<<< HEAD` から `=======` までが取り込み先(現在のブランチ)、`=======` から `>>>>>>>` までが取り込む側の内容です。解消の手順は次のとおりです。

1. ファイルを開き、マーカーを参考に正しい最終形に手で書き直す(マーカー記号も消す)
2. `git add` で解消済みとしてステージする
3. `git commit` でマージを完了する

あわてず「どちらの意図も理解してから 1 つにまとめる」のがコツです。どちらか一方を残すとは限らず、両方の変更を組み合わせるのが正解のこともあります。

もし途中で「いったんマージをやめたい」と思ったら、`git merge --abort` でマージ前の状態に戻せます。間違えても元に戻せると分かっていれば、落ち着いて対応できます。コンフリクトは慣れないうちは身構えがちですが、起きる場所と直し方は決まっているので、必ず解消できます。次のセクションでは、こうした変更をチームと共有するリモート連携を学びます。

## まとめ

- `git merge` はブランチの変更を取り込んで履歴を合流させる
- 同じ箇所を別々に変えるとコンフリクトが起き、人間の判断が必要になる
- マーカーを手で解消 → add → commit の順でマージを完了する
