# コミットと履歴の確認

## このセクションで学ぶこと

- git commit で変更を履歴に確定する
- よいコミットメッセージの書き方
- git log で履歴を読む

## コミットする

ステージに乗せた変更を履歴に確定するのが **`git commit`** です。`-m` オプションでコミットメッセージを添えます。

```bash
git commit -m "READMEにプロジェクト概要を追加"
```

コミットすると、その時点のスナップショットに**コミットハッシュ**という一意の識別子が付き、履歴の鎖に 1 つ要素が加わります。最新のコミットには **HEAD** という目印が付き、「いま自分がどこにいるか」を表します。

## よいコミットメッセージ

コミットメッセージは未来の自分やチームへの手紙です。「修正」「更新」だけでは、後から履歴を読んでも何をしたのか分かりません。**何を・なぜ**変えたのかが伝わるように書きます。

- 悪い例: `修正`
- よい例: `ログインボタンが二重送信される不具合を修正`

1 行目に要点を簡潔に書き、詳細が必要なら空行を挟んで本文を続けるのが一般的なスタイルです。前章で学んだ「1 コミット = 1 つの意図」を守ると、メッセージも自然と書きやすくなります。逆に、メッセージが「修正いろいろ」のようにうまく書けないときは、1 つのコミットに複数の意図を詰め込みすぎているサインかもしれません。そのときはコミットを分けられないか見直してみましょう。

## 履歴を読む

これまでのコミットは **`git log`** で確認できます。

```bash
git log                  # 詳細な履歴
git log --oneline        # 1コミット1行の要約
```

`--oneline` を付けると、コミットハッシュの先頭とメッセージ 1 行だけが並び、全体像をすばやくつかめます。たとえば「いつこのバグが入ったのか」を調べたいとき、まず `--oneline` で全体を俯瞰し、怪しいコミットのハッシュを見つけてから詳しく調べる、という流れが定番です。

履歴を読むときは「いつ・誰が・なぜ」を意識すると、プロジェクトがどう育ってきたかが見えてきます。そして読みやすい履歴は、ここまで学んだ「意味のある単位のコミット」と「何を・なぜが伝わるメッセージ」の積み重ねで作られます。次のセクションでは、コミット前に「いま何が変わっているか」を確認する方法を学びます。

## まとめ

- `git commit -m` でステージの変更を履歴に確定する
- コミットメッセージは「何を・なぜ」が伝わるように書く
- `git log`(特に `--oneline`)で履歴を読み返せる
