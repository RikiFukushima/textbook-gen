# リポジトリの作成とステージング

## このセクションで学ぶこと

- git init でリポジトリを作る
- git add で変更をステージに乗せる
- ファイルの状態(追跡対象かどうか)の考え方

## リポジトリを作る

新しくバージョン管理を始めるには、対象のディレクトリで **`git init`** を実行します。

```bash
mkdir my-project
cd my-project
git init
```

これでカレントディレクトリに `.git` が作られ、ここが 1 つのリポジトリになります。前章で学んだとおり、この `.git` の中に履歴がすべて格納されていきます。既存のプロジェクトを後から Git 管理下に置く場合も、そのディレクトリで `git init` するだけです。

## 変更をステージに乗せる

ファイルを作っただけでは、まだ履歴には何も記録されません。コミットの前に、**`git add`** で変更をステージに乗せる必要があります。

```bash
echo "# My Project" > README.md
git add README.md        # 1ファイルだけ
git add .                # カレント以下のすべての変更
```

`git add .` はまとめて乗せられて便利ですが、前章で触れたとおり「意味のある単位でコミットを分ける」観点では、ファイルを個別に指定する習慣も大切です。

特に気をつけたいのが、不要なファイルの巻き込みです。たとえばパスワードを書いた設定ファイルや、エディタが勝手に作る一時ファイルが作業ディレクトリにあると、`git add .` はそれらもまとめてステージに乗せてしまいます。一度コミットして共有してしまうと取り消しが面倒なので、ステージに乗せる前に次のセクションで学ぶ `git status` で「何が乗ろうとしているか」を確認する癖をつけると安全です。

## 追跡対象と未追跡

Git はすべてのファイルを最初から管理するわけではありません。一度も `git add` していないファイルは **未追跡(untracked)** で、Git は中身の変更を気にしません。一度 `git add` すると **追跡対象(tracked)** になり、以降は変更が検知されるようになります。

この区別があるおかげで、ビルド生成物や一時ファイルなど「履歴に残したくないもの」を未追跡のまま放置できます(本格的に除外したい場合は `.gitignore` を使いますが、それは後の章で扱います)。

ここまでの流れを図にすると、ファイルは次のように状態を移っていきます。

```mermaid
flowchart LR
  U[未追跡 / 変更あり] -->|git add| S[ステージ済み]
  S -->|git commit| C[コミット済み<br/>履歴に確定]
  C -->|ファイルを編集| U
```

次のセクションでは、ステージした変更を実際にコミットして履歴を作ります。

## まとめ

- `git init` でディレクトリをリポジトリ化する(.git が作られる)
- `git add` で変更をステージに乗せてからコミットする
- ファイルには未追跡と追跡対象の状態があり、git add で追跡対象になる
