# ReAct ループの実体 — Thought / Action / Observation の循環

## このセクションで学ぶこと

- ReAct ループの 1 周で実際に何が LLM・ホスト・ツール間でやり取りされるか
- 過去の Thought / Action / Observation が「履歴」として次ターンに渡される仕組み
- 終了条件をどこで判定するか(最終回答 / 上限到達 / エラー)

## 02-04 の復習 — 設計の型としての ReAct

02-04 で、ReAct は Thought → Action → Observation のループだ、と学びました。ここではその「設計の型」を、**実装として何が起きているか** のレベルに踏み込みます。プロンプト 1 枚で完結していた CoT と違い、ReAct は **複数回の LLM 呼び出し** と **ホスト側のループ制御** が組み合わさって動くことを押さえてください。

## 1 ターンで何が起きているか

ReAct ループの 1 周(以下「ターン」と呼びます)では、次の流れが繰り返されます。

```mermaid
flowchart TB
  Start([開始]) --> Init[履歴(スクラッチパッド)= 空]
  Init --> Call[ホストが LLM を呼ぶ:<br/>ユーザー入力 + ツール一覧 + 履歴]
  Call --> Out{LLM の出力は?}
  Out -->|Thought + Action| Exec[ホストが Action を実行]
  Exec --> Obs[Observation を取得]
  Obs --> Append[履歴に Thought/Action/Observation を追記]
  Append --> Check{終了条件?}
  Check -->|未達| Call
  Check -->|達成| Finalize[最終回答を返す]
  Out -->|Final Answer| Finalize
  Finalize --> End([終了])
```

ポイントは、**ループの主体はホスト側** だということです。LLM はあくまで「Thought と Action を出力する」「あるいは最終回答を出力する」関数として呼ばれているにすぎません。条件分岐とループはホストのコードで書かれていて、LLM 自身は「いま自分が何周目か」を知らないまま動いています。

## 履歴(スクラッチパッド)が記憶を担う

LLM はステートレスです。ターン間で記憶を持ちません。それでも「さっき検索して 28℃ という結果が返ってきた」という文脈を踏まえて次の判断ができるのは、**ホストが履歴(スクラッチパッド)を維持しているから** です。

履歴は、これまでの Thought / Action / Observation を時系列に積み上げたテキストで、毎ターンの LLM 呼び出しのプロンプトに丸ごと添えられます。典型的なプロンプト構造はこんな形です。

```
[System]
あなたは ReAct エージェントです。次のツールが使えます: search, calc, ...
答えが出るまで Thought / Action を続けてください。

[User]
東京の今日の最高気温と、それを華氏に直した値は?

[Scratchpad — これまでの履歴]
Thought: まず東京の今日の最高気温を調べる必要がある。
Action: search("東京 今日 最高気温")
Observation: 28℃

Thought: 28℃を華氏に変換する。
Action: calc("28 * 9/5 + 32")
Observation: 82.4

[次のステップを Thought + Action(または Final Answer)で出力してください]
```

ターンが増えるほど履歴は長くなり、コンテキストウィンドウを消費していきます。これが ReAct の見えにくいコスト構造で、10 ターン回せば毎ターンで全履歴をプロンプトに乗せるため、**入力トークンはほぼ二次関数的に増える** ことになります。第 1 章の lost in the middle もここで効いてきます。長い履歴の真ん中の Observation を、モデルが見落とすことが起きます。

## 終了条件はホストが見張る

ループの **終了条件はホスト側が判定** します。代表的な終了条件は次の 3 つです。

- **最終回答が出た**: LLM の出力が `Final Answer:` を含む(あるいは Function Calling の場合、ツール呼び出しではなく通常テキストを返した)。これが正常終了。
- **最大反復回数に達した**: あらかじめ設定した上限(典型的には 5 〜 15 ターン)を超えた。これは「諦め」の終了で、現時点でわかっていることをまとめて返すかエラーにする。
- **連続エラー / 同一 Action の繰り返し**: ツールが失敗し続ける、または同じ Action と引数を何度も出している。後者は無限ループの兆候で、検出したら止める。

「最終回答が出るまで回し続ける」設計は危険です。**LLM は自信たっぷりに迷い続けることがある** ので、ホスト側の物理的な上限が安全弁として必要です。次節の Function Calling では、この終了判定がもっとクリーンに書けるようになります。

## 注意点 — ターンは安くない

各ターンで LLM 呼び出しが入ること、履歴が膨らみ続けること、ツール呼び出しのレイテンシも乗ること。この 3 つが重なって、ReAct エージェントは単発のチャットより **1 〜 2 桁高くつく** のが普通です。「ツール経由でちょっと調べる」だけのつもりが、10 秒・数十円規模のコストになることもあります。

実務では、ターンごとの履歴を要約して圧縮する、過去の Observation のうち重要なものだけを残す、といった「履歴の節約」が現実的な改善ポイントになります。

## まとめ

- ReAct ループの主体はホスト。LLM は毎ターン「Thought + Action」または「Final Answer」を返す関数として呼ばれる
- 履歴(スクラッチパッド)が記憶を担い、毎回プロンプトに添えられる。コンテキストとコストを食う
- 終了条件はホストが見張る。「答えが出るまで」だけでは危険、上限と無限ループ検出を必ず入れる
