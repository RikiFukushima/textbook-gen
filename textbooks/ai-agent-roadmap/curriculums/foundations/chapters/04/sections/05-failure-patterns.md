# Agent が失敗するパターン — 無限ループ、誤ったツール選択、計画破綻

## このセクションで学ぶこと

- Agent が壊れる主要 3 パターン(無限ループ・誤ったツール選択・計画破綻)の症状と原因
- 各パターンに対応するガードレールの設計方針
- 「Agent が向かないタスク」を見極めて Chain に戻す勇気

## 失敗は「どこで止まるか」で分類する

03-05 で RAG の失敗を Retrieval / Augmentation / Generation の層で切り分けたように、Agent の失敗もパターンで分類するとデバッグが進みます。本節では実務でよく見る 3 つを順に整理します。

- **無限ループ**: そもそも止まらない
- **誤ったツール選択**: 動いてはいるが、間違った道具を呼ぶ
- **計画破綻**: 多段の手順の途中で迷子になる

## 失敗 1 — 無限ループ

最も派手な失敗が **無限ループ** です。同じ Thought を繰り返す、同じツールを同じ引数で何度も呼ぶ、ツールがエラーを返しているのに気にせず再試行を続ける、といった症状で現れます。

原因は典型的に次のどれかです。

- **ツールの結果を Observation として正しく反映できていない**: 履歴に乗っていても、モデルが読み飛ばしている。lost in the middle(01-05)の影響。
- **「タスクが解けない」と認める出力をしない設計**: 「資料に無い」と返してよいと教えていない。
- **`tool_choice` の強制呼び出しが効きすぎている**: 04-04 の機能で「必ずツールを呼べ」と縛ると、答えが出てもツールを呼び続ける。

ガードレールはシンプルで、**ホスト側で物理的な上限を必ず持つ** ことです。

- 最大反復回数(典型 5 〜 15 ターン)
- 同一 Action と同一引数の連続検出(2 回連続なら停止 or 異なる戦略を促す)
- タイムアウト(壁時計と累積トークンの両方)

これは「LLM を信用しない設計」であり、ReAct ループを書く際の必須装備です(04-03 で触れた終了条件の話の延長です)。

## 失敗 2 — 誤ったツール選択

次に多いのが **誤ったツール選択** です。「ファイル検索すべき場面で Web 検索を呼ぶ」「計算すべきところを検索で済ませてしまう」「引数を間違えて意味のないクエリを投げる」など、動いてはいるが結果が外している状態です。

原因の多くは **ツール定義の説明不足** です。04-02 で書いたとおり、`description` に「何ができる」だけが書いてあって「いつ使うべきか」が無いと、LLM は似たような名前のツールから適当に選びます。次のような書き換えが効きます。

```
# 弱い説明
"description": "ファイルを検索する"

# 強い説明
"description": "社内ドキュメントの全文検索。社内マニュアルやFAQに書かれた事実を知りたい場合に使う。Web 上の最新情報には使わないこと。"
```

「使うべきでない場面」まで書く、と覚えてください。あわせて、似たツールを並べるときは **使い分けの条件を相互に書く**(`search_docs` には「Web 検索が必要なら `web_search` を使うこと」と書き、逆も書く)と、選択の安定性が大きく上がります。

## 失敗 3 — 計画破綻

3 つめが **計画破綻** です。シンプルな質問では問題ないのに、3 〜 5 ステップ以上の手順が必要なタスクで途中の前提を見失う、サブゴールを忘れる、ループの後半でユーザーの本来の質問とズレた回答に向かう、といった失敗が起きます。

原因の中心は **履歴の膨張と注意の散逸** です。前節で触れたとおり、ターンを重ねるごとに履歴が伸び、モデルはその中央の情報を見落としやすくなります。「3 ターン前に検索した結果」をモデルが思い出せず、もう一度同じ検索を走らせる、というのが典型例です。

対策は層を分けて打ちます。

- **計画ステップを明示的に持たせる**: タスク開始時に「やるべきステップの一覧」を出力させ、進捗を履歴の先頭に置く。これだけで方向感が安定する。
- **要約圧縮**: 古い Observation を要約して履歴サイズを抑える。10 ターン目あたりからの定石。
- **サブエージェント化**: 大きなタスクは、独立したサブタスクに分け、それぞれを別の(短い履歴の)Agent に解かせて結果だけを集約する。
- **構造化された状態**: 履歴を散文に積むのではなく、`progress`・`open_questions`・`facts` のような構造に整理する。

最後の点は実装フレームワーク(LangGraph、各種 Agent SDK)が支援する領域で、自前で全部やる前に既存ツールの仕組みを使うのが現実的です。

## それでも Agent が向かないタスクはある

3 つのパターンへのガードレールをいくら整えても、**Agent が原理的に向かないタスク** はあります。

- **手順が事前に書き下せるタスク**: 04-01 のとおり、これは Chain で十分。Agent にする意味が薄い。
- **副作用が大きく、間違いの取り返しがつかないタスク**: 大量メール送信、本番 DB の破壊的変更、課金、社外への通知など。Agent の確率的な判断を間に挟むのはリスクが高すぎる。最低でも人間確認のステップ(human-in-the-loop)を挟む設計にする。
- **強い時間制約があるタスク**: ReAct は本質的に複数回 LLM を呼ぶため、サブ秒応答は難しい。

「Agent で頑張って作り込んだが、結局 Chain + 強めの分岐で書き直したほうが速くて安定する」という結末は実務で頻繁に起こります。**Agent から Chain へ戻す判断ができる** ことも、設計力のひとつです。

## まとめ

- 無限ループは上限・重複検出・タイムアウトで物理的に止める
- 誤ったツール選択はツール定義の `description` を「いつ使うか・いつ使わないか」まで書いて改善する
- 計画破綻は明示的な計画、履歴の要約、サブエージェント化で対応。それでも難しければ Chain に戻す
