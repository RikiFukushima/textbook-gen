# なぜ LLM の評価は難しいのか

## このセクションで学ぶこと

- 従来のソフトウェア・ML と比べて、LLM の評価で何が変わるのか
- 出力が確率的・自由形式であることが評価の不安定さを生む構造
- 「正解が一意でない」状況を扱うために必要な発想の転換

## 従来のテストが効かなくなる場所

通常のソフトウェアでは、入力に対する期待出力が決まっています。`add(2, 3)` は必ず `5` を返すべきで、違っていればバグです。機械学習の分類でも、「この画像は猫」のように正解ラベルが定まっていれば、正解率や F1 で評価できます。

LLM のアプリケーションでは、この前提が二重に崩れます。第 1 章で見たように、**出力は確率分布からのサンプリング** なので、同じ入力でも実行ごとに表現が揺れます。さらに、出力が自由文・コード・要約・対話といった **自由形式** であるため、「正解は何か」を一つの文字列に固定すること自体が難しくなります。「適切な要約」は無数にあり得ますし、「丁寧な回答」もまた一通りではありません。

つまり LLM の評価では、(1) 同じ入力でも結果がブレる、(2) ブレた結果が「正解の範囲内」かどうかを機械的に判定しづらい、という二段構えの難しさがぶつかってきます。

## 「正解が一意でない」をどう扱うか

この難しさを正面から受け止めるために、評価の世界では発想を切り替えます。鍵になるのが **参照あり評価** と **参照なし評価** の区別です。

- **参照あり評価**: あらかじめ用意した期待出力と比較する。分類タグの一致、数値の一致、抽出したエンティティの集合一致など、答えが離散的に決まるタスクに向きます。
- **参照なし評価**: 唯一の正解を持たず、「忠実性」「一貫性」「読みやすさ」のような観点ごとに採点する。要約・翻訳・対話応答のように、許容される正解が広いタスクで使います。

参照なし評価では、観点と判定基準を言語化した **ルーブリック**(評価チェックリスト)が中心になります。たとえば「資料に書かれていない事実を加えていないか」「ユーザーの質問の核心に答えているか」を 0/1 や 3 段階で採点する、といった具合です。後の節で扱う LLM-as-a-Judge は、このルーブリックをモデルに渡して採点させる手法と捉えると理解しやすくなります。

## 「ブレ」とどう向き合うか

確率的な出力に対しては、**1 回の結果で良し悪しを判断しない** のが基本姿勢です。02-05 の Self-Consistency と発想は似ていて、同じ入力を複数回流して結果の分布を見ます。たまたま当たった/外れたのか、それとも安定して当たる/外れるのかを区別しないと、評価結果そのものがノイズに埋もれます。

実務では次のような手順がよく取られます。

- **代表的な入力セット(eval set)を固定** し、毎回同じ問題で評価する
- **複数回サンプリング** して平均と分散を見る(temperature の影響を理解した上で)
- **観点別にスコアを分ける**(忠実性は高いが流暢さが低い、など、後で改善ターゲットが見える形に保つ)

注意したいのは、**「正答率 90%」のような単一指標だけで判断しない** ことです。LLM アプリの良し悪しは多次元で、平均では潰れてしまう失敗(たとえば 03-05 で見たような検索失敗)を見落とします。失敗事例そのものをリストとして保管し、改善後にどれが解消されたかを追うほうが、長い目で見て信頼できます。

## まとめ

- LLM の評価は「出力のブレ」と「正解の幅広さ」の二段で難しい
- 参照あり/参照なしの区別と、観点を切り分けたルーブリックが基本道具
- 単一指標で見ず、複数回サンプリングと失敗事例の蓄積で品質を捉える
