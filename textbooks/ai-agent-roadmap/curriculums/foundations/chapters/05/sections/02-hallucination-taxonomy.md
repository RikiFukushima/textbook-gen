# ハルシネーションの分類

## このセクションで学ぶこと

- ハルシネーションを「事実誤認 / 文脈逸脱 / 過信」の 3 タイプで切り分ける視点
- それぞれの発生メカニズムと対処方針の違い
- 「対策」と「許容」を分けて考える実務上の構え

## なぜ分類するのか

ハルシネーションを一括りに「嘘をつく」と捉えてしまうと、対策が「もっと賢いモデルを使う」「プロンプトを強くする」あたりで止まってしまいます。実際には、ハルシネーションには性質の違う複数のパターンがあり、**起こる原因も対策も別物** です。問題を切り分けて、どのタイプがどれくらい出ているかを把握できるようにするのが、ここでの目標です。

ここでは現場で頻出する 3 つのタイプ — **事実誤認 / 文脈逸脱 / 過信** — に整理して扱います。

## 事実誤認 — 世界や資料と食い違う

最も典型的なのが **事実誤認** です。第 3 章で触れたとおり、LLM はパラメトリック知識の中に「いつ」「どこ」「誰」の情報を厳密に区別して持っているわけではありません。学習データに無い、もしくは曖昧にしか含まれていない事実について聞かれると、それらしい言葉で穴埋めをしてしまいます。

事実誤認は二段階で起こりえます。

- **世界の事実との不一致**: そもそも実在しない論文・人物・URL を出してくる、年代を取り違える、など。
- **与えた資料との不一致**: RAG で渡した文書には書いていない数値や結論を、それでも回答に混ぜてしまう。

対策の主軸は **根拠を持たせる** ことです。RAG で関連文書を渡し、Instruction で「資料に書かれていないことは答えない」と縛り、出力に引用 ID を付与させる。03-05 で見たとおり、検索失敗・文脈過多・生成迷走の層ごとに改善ポイントは違います。完全になくすのではなく、「根拠なしの主張をしにくい構造」をプロンプトと出力スキーマで作り込むのが現実的です。

## 文脈逸脱 — 内容は正しいが、求められたものではない

次が **文脈逸脱** です。事実としては合っているのに、ユーザーの要件・指示・出力スキーマから外れている、というパターンです。

- 「3 行で要約して」と頼んだのに段落で返ってくる
- JSON スキーマを指定したのに自然文で混ぜ返してくる
- 質問は「次に何をすべきか」なのに、背景知識の解説だけで終わる
- 翻訳タスクなのに、原文の意図を「補足」して創作してしまう

事実誤認とは別系統の失敗で、原因は **指示の弱さ・曖昧さ** か、**モデルが自分のクセに従ってしまった** ことにあります。対策としては、02-01 で扱ったプロンプトの 4 要素(Instruction / Context / Input / Output)を明確に分ける、出力フォーマットを JSON や Markdown 見出しで強制する、Few-shot の例で期待形を示す、といった設計が効きます。Function Calling のように **構造化出力をモデル側に強制する仕組み** も、文脈逸脱への有効な抑止になります。

## 過信 — 確からしさを実態以上に見せる

3 つめが **過信** です。これは独立した嘘というより、**事実誤認や文脈逸脱を「断定口調」で出力する** ことで生まれる被害の増幅器のような失敗です。

代表的な症状は次のとおりです。

- 不確実な事柄を「明らかに〜です」「必ず〜になります」と言い切る
- 存在しない論文タイトル・URL・関数名を、もっともらしい書式で引用する
- 自分が誤った前提に立っていることを認識せず、追加の問いにもその前提で答え続ける

過信が怖いのは、**ユーザーが検証コストを払わずに信じ込んでしまう** 点です。対策は出力の信頼度を「言葉として表に出させる」ことに尽きます。「不確かな箇所は『推測ですが』と前置きする」「引用が必要な主張には `[doc-XX]` を付ける」「該当文書がなければ『資料には記載がありません』と返す」といった指示を明示する、引用 ID の実在チェックをアプリ側で行う、などの組み合わせで抑え込みます。

## 「ゼロにする」より「気づける」設計を

注意したいのは、ハルシネーションは確率的生成の構造から完全には消えないという事実です。だからこそ、**ゼロにする努力と同じくらい、出てしまったときに気づける仕組み** が重要になります。引用 ID の実在チェック、出力スキーマのバリデーション、評価データでの定点観測 — どれもこの「気づける化」のための投資です。次の節では、こうした検知と評価をどんな手段で回すか、その地図を確認します。

## まとめ

- ハルシネーションは事実誤認・文脈逸脱・過信の 3 タイプに切り分けて対処する
- 事実誤認は根拠付け、文脈逸脱は構造化出力、過信は信頼度の明示で抑える
- 完全にゼロにはできない前提で「気づける」検証の仕組みを併せ持つ
