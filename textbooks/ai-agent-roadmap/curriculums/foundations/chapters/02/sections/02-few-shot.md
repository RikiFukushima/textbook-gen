# Zero-shot と Few-shot

## このセクションで学ぶこと

- Zero-shot と Few-shot の違いと、それぞれの使いどころ
- 例示(shot)が出力の形式と判断基準を伝える仕組み
- Few-shot が効かない・むしろ悪化するケース

## 例を見せるかどうかでアプローチが変わる

タスクの解かせ方には大きく 2 つあります。

- **Zero-shot**: 例を一切示さず、指示だけで解かせる方法。「次のレビューが好意的か否定的か判定してください」だけを渡す形。
- **Few-shot**: 入力と望ましい出力のペアを数件示してから本題を解かせる方法。LLM はこの例示を見て、形式や判断基準を **その場で学習** します。

LLM が学習(重み更新)なしに、プロンプト内の例だけで挙動を変えられる性質を **in-context learning** と呼びます。Few-shot はこの性質を意図的に利用するテクニックです。

## 具体例 — 例示で形式と基準を揃える

レビュー文の感情判定を例にとります。Zero-shot だと出力の表記は揺れがちです。

```
入力: 配送が遅いが商品は良かった。
出力: ニュートラル寄り、やや肯定?
```

Few-shot にすると、出力フォーマットと「混在ケースをどう扱うか」の判断基準まで揃えられます。

```
入力: 期待通りでした。
出力: positive

入力: 二度と買わない。
出力: negative

入力: 普通でした。特に印象なし。
出力: neutral

入力: 配送が遅いが商品は良かった。
出力:
```

例示によって「`positive` / `negative` / `neutral` の 3 値を小文字で返す」「混在は商品評価を優先する」といった暗黙のルールが共有されます。指示文で同じことを文章で書くより、例を 3 件見せるほうが正確に伝わることがよくあります。

例の選び方にもコツがあります。**目的の入力に近い分布の例**、**判断が割れやすい境界ケース**、**陽性と陰性のバランス**を意識すると効きます。3〜5 件で十分な場合が多く、増やしすぎはコンテキストを圧迫します。

## 注意点 — Few-shot が効かない・悪化するケース

Few-shot は万能ではありません。次のような場面では効果が薄い、または逆効果になります。

- **形式の指定で十分なタスク**: Output 指定だけで形が決まるなら、例示は冗長です。
- **例に偏りがある**: 同じパターンの例を並べると、モデルがその偏りを真似て本来は別カテゴリの入力まで同じラベルに寄せてしまいます。
- **複雑な推論が要るタスク**: 「例の入出力だけ」では中間の考え方が伝わりません。これは次節の Chain-of-Thought が解く問題です。
- **コンテキストが逼迫している**: 例示はトークンを大量に消費します。長い参照資料と併用するときは、例の数を絞るか要約する必要があります。

また、例の中に **誤った出力** が紛れていると、モデルがそれを「正解」として学習し、品質が崩れます。Few-shot を導入したら、まず例自身の妥当性をチェックしてください。

## まとめ

- Zero-shot は指示のみ、Few-shot は例示で形式と基準を渡す方法
- LLM の in-context learning により、例 3〜5 件で挙動を大きく揃えられる
- 例の偏り・誤りは品質を悪化させる。複雑な推論には次節の CoT が必要
