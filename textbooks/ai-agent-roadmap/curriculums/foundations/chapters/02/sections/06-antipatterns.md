# プロンプト設計のアンチパターン集

## このセクションで学ぶこと

- 実務で頻出するプロンプトの失敗パターンを類型化して認識できること
- 各アンチパターンがなぜ失敗するかの根本理由
- 同じ失敗を避けるための具体的な書き換え方針

## アンチパターン 1 — 「いい感じに」依存

「適切に判断してください」「いい感じにまとめてください」のような **基準を相手任せ** にする書き方は、最も頻出する失敗です。モデルは何が「適切」かを毎回違う基準で解釈するため、出力が安定しません。

書き換えの方針はシンプルで、**判断基準を明示する** こと。たとえば「適切に判断」を「次の 3 つの条件のうち 2 つ以上に当てはまれば採用」のように具体化します。基準が書きにくいタスクは、Few-shot で例を見せるのも有効です(§02-02)。

## アンチパターン 2 — 矛盾する指示

「簡潔に。ただし詳細も省かないで」「丁寧に。ただし短く」のように **両立しない要求** を同時に出すと、モデルはどちらかに偏った出力を返し、安定しません。

書く側にとって「両方欲しい」のは本音ですが、優先順位を明示する必要があります。「200 文字以内にまとめる。情報が多すぎる場合は背景より結論を優先する」のように、**衝突したときどちらを取るか** まで書きます。

## アンチパターン 3 — 過剰な丁寧さと脅し

「絶対に」「必ず」「決して」「いかなる場合も」を多用したり、「間違えたらシステムが停止します」のような **脅し** を入れたりするプロンプトをよく見かけます。短期的に効くこともありますが、**例外を吸収できず脆く** なります。実運用では「絶対」と書いた制約に反する妥当な入力が必ず現れます。

代わりに、**条件と例外をセットで** 書きます。「個人情報を含む文は出力しない。ただし氏名のみで個人特定できない場合は許可する」のように、想定する例外を先に書いておくと、未知の入力でも判断が安定します。

## アンチパターン 4 — Context の詰め込みすぎ

「念のため」と参照資料を全部貼ると、コンテキストが圧迫され、第 1 章で見た **lost in the middle** で重要箇所が埋もれます。コストも増えます。

書き換え方針は 2 つ。**今回のタスクに必要なものだけ載せる**、または **検索で必要部分だけ動的に渡す**(これが第 3 章 の RAG)。資料の見出し一覧と本文を別ターンで分けて渡し、モデルに「どの節を読むか」を選ばせる方法も有効です。

## アンチパターン 5 — 暗黙の前提

書き手の頭の中では当然でも、モデルには見えていない前提があるとずれた出力になります。「先週のミーティング」「うちのチームのルール」「いつもの形式」のような **コンテキスト依存の語** が代表例です。

対策は、暗黙の前提を **Context として明示** すること。「うちのチームのルール = 次の 3 項目: ...」と書き出すだけで、出力が安定します。プロンプトをレビューするときは「これは社外の人が読んでも意味が通るか?」を基準にすると、抜けた前提に気付きやすくなります。

## アンチパターン 6 — Output 指定なしで JSON を期待する

「JSON で返してください」だけ書いて、**スキーマを示さない** とフィールド名・型・ネスト構造が揺れます。後段のパースで毎回落ちて、再試行のループに陥ります。

最低限、キーの一覧と型、必須/任意の区別、サンプル出力を 1 件示します。可能なら API 側の **構造化出力(JSON Schema 指定)** 機能を使うと、形式エラーは大幅に減ります。

## アンチパターン 7 — 再現性を考えない検証

「動いた」「動かない」の判断を毎回手で行い、`temperature` も記録しない検証は、再現性がなく改善が積み上がりません。

検証時は **入力・パラメータ・出力をセットで保存** し、プロンプト改修の前後で同じ入力セットで比較できる状態を作ります。評価セット(代表的な 20〜50 件)を持つだけで、改善が回ります。詳しくは第 5 章で扱います。

## まとめ

- 基準を相手任せにしない、矛盾を残さない、暗黙の前提を書き出す
- 「絶対」より「条件と例外」、詰め込みより必要箇所の選択
- 形式は JSON Schema で固定、検証は入力と出力をセットで記録して比較する
