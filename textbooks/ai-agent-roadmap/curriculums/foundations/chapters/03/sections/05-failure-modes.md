# RAG が効かないケース

## このセクションで学ぶこと

- 検索失敗・文脈過多・生成迷走という 3 つの典型的な失敗パターン
- それぞれがパイプラインのどの工程に起因するかの切り分け方
- RAG が苦手とする「知識ではなく推論」のタスクの見極め

## 失敗は「層」ごとに切り分けて見る

RAG が期待どおりに動かないとき、漠然と「精度が悪い」と捉えると改善のしようがありません。前節のパイプラインを思い出してください。RAG の失敗は、おおむね次の 3 つの層のどこかで起きています。

- **Retrieval 層の失敗 → 検索失敗**: そもそも正しい文書が取れていない
- **Augmentation 層の失敗 → 文脈過多**: 取れているが、入れすぎ・長すぎで埋もれる
- **Generation 層の失敗 → 生成迷走**: 良い資料が渡っているのに、モデルが活かせない

それぞれ症状と原因が違うので、現象から逆引きして層を特定するのが基本姿勢です。

## 検索失敗 — 取りたい文書が上位 K 件に入らない

最も多いのが **検索失敗** です。「正しい文書はストアにあるのに、Retrieval で上位 K 件に入らない」状態を指します。

典型的な原因は次のとおりです。

- **言い換え過ぎ・専門用語のミスマッチ**: ユーザーが俗称で聞いているのに、文書は正式名称で書かれていて、ベクトルでも距離が遠くなっている。
- **固有名詞・型番がベクトルに乗らない**: 03-02 で触れたとおり、字面そのものはベクトル検索の不得手領域。
- **チャンクが大きすぎて意味が薄まる**: 1 チャンクに複数の話題が入り、平均化されて埋め込みが「特徴のない方向」を向いている。
- **メタデータでのフィルタを忘れている**: 権限外の文書や、古いバージョンが混じって上位を取ってしまう。

切り分けるには、**正解の文書 ID をあらかじめ用意した質問セット(評価データ)** を作り、その質問に対して「正解文書が上位 K 件に入る割合(Recall@K)」を測るのが定石です。検索失敗が確認できたら、ハイブリッド検索の導入、チャンクサイズの見直し、クエリ書き換え(同義語展開、関連語の補完)などで改善します。

## 文脈過多 — 取れているのに、入れすぎて埋もれる

次に多いのが **文脈過多** です。検索自体は当たっているのに、上位 K 件を欲張って増やしたり、1 件あたりのチャンクが長すぎたりして、プロンプトが膨らんでしまうケースです。

第 1 章の 01-05 で見た **lost in the middle** がここで効いてきます。長いコンテキストの中央に置かれた情報は読まれにくく、せっかく取り出した正解チャンクが「真ん中に埋もれて参照されない」事態が起きます。さらに、関連度の低いチャンクが混ざるほど、モデルがどれを根拠にすべきか迷いやすくなります。

対策の方向はシンプルで、**K を減らす・チャンクを短くする・Re-ranking で精度の高い上位だけに絞る** の 3 つに集約されます。Recall を取りに行きすぎず、Precision とのバランスを取る感覚が要ります。「とりあえず K=20 で渡せば安心」という発想は、しばしば逆効果になります。

## 生成迷走 — 良い資料が渡っているのに活かせない

3 つめが **生成迷走** です。Retrieval も Augmentation も適切なのに、最終的な回答が変、というパターンです。

よくある症状は次のとおりです。

- **資料を無視して、モデルの重み(パラメトリック知識)で答えてしまう**
- **資料の一部だけを拾い、矛盾する文を平気で出力する**
- **資料には書かれていないことを「補足」として加える**(ハルシネーション)
- **引用 ID を間違える、あるいは付け忘れる**

ここはプロンプト設計の領分です。「参考資料に書かれていないことは答えない」「該当する記述が見つからなければ『資料には記載がありません』と返す」「回答中の主張ごとに `[doc-XXX]` を付ける」といった指示を Instruction に明示する、Output スキーマを JSON で固定する、といった対応が効きます。出力に毎回引用 ID を要求する運用は、回答品質のセルフチェックとしても機能します。

## RAG そのものが向かないタスクもある

最後に、これら 3 層の対策をいくら丁寧にやっても、**RAG が原理的に向かないタスク** があることを押さえておきましょう。

- **論理推論・算術・コード実行**: 答えはどの文書にも書かれておらず、「考えて出す」必要がある。これは RAG ではなく、第 4 章で扱う Tool Use(計算機、コード実行器、API)で解く領域です。
- **広い俯瞰や要約**: 「全社の方針を 1 ページで要約して」のように、全文を読み通す必要があるタスクは、上位 K 件のチャンクだけでは構造的に取りこぼします。長文要約パイプラインや別系統の前処理が必要になります。
- **意見・判断**: 客観的な事実が文書に書かれていないタイプの質問は、いくら検索しても根拠が出てきません。

「答えがどこかの文書に書いてあるか?」を最初に問い、Yes なら RAG、それ以外は別の手段、と切り分けるのが健全な適用判断です。

## まとめ

- RAG の失敗は Retrieval / Augmentation / Generation の層ごとに切り分けて診断する
- 検索失敗はハイブリッド検索やチャンク見直し、文脈過多は K と長さの削減、生成迷走はプロンプト設計で対応する
- 推論・算術・全体要約のように「答えが文書に無い」タスクは RAG ではなく別の手段(Tool Use 等)で解く
