## このセクションで学ぶこと

- 出力フィルタリングが「LLM の生出力を信用しないラストライン」である理由を説明できる
- Guardrails AI と NeMo Guardrails の設計思想の違いを把握できる
- 出力カテゴリ検査と JSON Schema 検証を組み合わせたガードレールを設計できる

## なぜ生出力を信用してはいけないか

System Prompt をどれだけ丁寧に書いても、LLM は確率的に振る舞います。**99% 期待どおりに動いても、残りの 1% が PII を吐き、攻撃命令に従い、JSON を壊します**。本番プロダクトとしては 1% の事故が許容できないシナリオが必ずあるため、出力をそのままユーザーや下流システムに渡してはいけません。

出力フィルタリングは、LLM 呼び出しの**直後・利用の直前**に挟むラストラインです。前段の対策(System Prompt 隔離、サニタイズ、スコープ最小化)を全て通り抜けた問題出力を、ここで止めます。逆に言えば、ガードレールだけに頼って前段を疎かにしてはいけません。**ガードレールは万能フィルタではなく、最終チェックポイント**だと位置づけます。

## 二つの代表ライブラリ

実装に使うライブラリは状況で選びます。代表的な OSS が二つあります。

**Guardrails AI** は Python ベースで、`RAIL` 仕様や Pydantic で「期待する出力構造」を宣言し、その構造に対する Validator(個人情報検出、毒性検出、SQL インジェクション検出、JSON Schema 適合など)を組み合わせる思想です。**粒度が細かく、構造化出力を扱う API バックエンドに向く**特性があります。検査に失敗した場合に「再試行」「修正」「例外」のどれを選ぶかをポリシーで宣言できるのも特徴です。

**NeMo Guardrails** は NVIDIA が提供する OSS で、`Colang` という会話 DSL を用いて「禁止トピック」「許可された対話フロー」「入出力レール」を宣言的に書きます。**チャットアプリのような対話文脈に強く**、ユーザーの発話・LLM の応答・外部 Tool 呼び出しの各段にレールを敷けます。複雑な対話設計を一元管理したい場合に有効です。

両者は競合というより補完的で、API レイヤは Guardrails AI、会話レイヤは NeMo Guardrails、というように使い分ける構成も実務では珍しくありません。

## 検査の粒度を二つに分けて設計する

ガードレールの設計で混乱しがちなのは、**「何を検査するか」**の粒度です。実用上、二つの軸に分けて整理すると見通しが良くなります。

**出力カテゴリの検査**は、生成テキストに対する内容ベースのチェックです。PII / 暴力的表現 / 不適切な助言(医療・法律)/ 競合製品への言及など、ポリシー違反のカテゴリを定義し、それぞれに検出器(正規表現・分類モデル・LLM-as-a-Judge)を当てます。検出時の挙動は「マスキング」「再生成」「ユーザーへの定型応答に差し替え」のいずれかを選びます。

**出力形式の検査**は、構造化出力に対するスキーマ検証です。`tool_call` や JSON モードで返ってきた出力を JSON Schema や Pydantic で検証し、必須フィールドの欠落・列挙値違反・型不一致を機械的に弾きます。**Agent ループで Tool 引数を生成する場面では、スキーマ検証が落ちれば「もう一度生成し直す」フィードバックループを組む**のが定番です。

実プロダクトでは、この二つを両方掛けます。カテゴリ検査で「言ってはいけないこと」を弾き、形式検査で「壊れた出力」を弾く。両方を一つのリクエストパスに収めるとレイテンシが効くため、軽い検査(正規表現・Schema)を先に、重い検査(LLM-as-a-Judge)を後にし、早期失敗を意識します。

## 注意点

ガードレールは検査コストと誤検知のバランスが難しいコンポーネントです。**過剰に厳しくすると、正当な質問にも「お答えできません」を返す過敏な Agent**になり、ユーザー体験が崩れます。ログから誤検知率と見逃し率を継続的にモニタし、しきい値を運用フェーズで調整できる構造にしておきます。また、ガードレールに対する評価データセットも、本文評価とは別に持っておくと判断がぶれません。

## まとめ

- ガードレールは生出力をユーザー・下流に渡す前のラストラインで、前段対策の代わりではない
- Guardrails AI は構造化出力向け、NeMo Guardrails は対話フロー向けと使い分ける
- 検査は「出力カテゴリ」と「出力形式」の 2 軸で設計し、誤検知率を運用で調整する
