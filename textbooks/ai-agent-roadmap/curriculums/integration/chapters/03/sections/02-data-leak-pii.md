## このセクションで学ぶこと

- データリークが起きる 3 経路(学習データ・ログ・検索対象)を分けて捉えられる
- 各経路で起きる典型的なリーク事故と対策の対応関係を説明できる
- PII を扱う際にオプトアウト・ログマスキング・テナントフィルタを設計に組み込める

## データリークの 3 経路を分けて考える

「LLM にデータが漏れる」と一括りで議論すると対策が散漫になります。実務では**漏れる場所を 3 経路に分けて、それぞれに対策を当てる**のが定石です。経路を取り違えると、片方を完璧に塞いだのに反対側から漏れる、という事故が起きます。

3 経路とは「学習データ」「ログ」「検索対象」です。性質と対策がそれぞれ異なります。

## 経路 1: 学習データとして外に出てしまう

最も警戒されるのは、自社の機密情報や顧客の PII がモデルプロバイダ側で**次世代モデルの学習に利用される**経路です。Consumer 向け API では入力データが学習に使われる前提のプランもあり、契約条件を読まずに使うと、社内ドキュメントが将来のモデルに浸透していくリスクがあります。

対策はシンプルで、**プロバイダ側のオプトアウトを契約・設定の両方で確定させる**ことです。OpenAI の Enterprise / API デフォルト、Anthropic の Commercial Terms、Azure OpenAI / Bedrock のように「入力は学習に利用しない」と明文化されたプランやリージョンを選びます。BAA(医療)や類似の DPA 締結が必要な業界では、契約書面で保持期間・サブプロセッサ・地域もまとめて確認します。エンドユーザーに「あなたの入力は学習に使われません」と説明できる根拠を、契約とコンソール設定の両方で持っておきましょう。

## 経路 2: ログに PII が残る

観測性のためにトレースを残すと、**プロンプト本文・入出力・Tool 呼び出しの引数**がそのまま蓄積されます。LangSmith や Langfuse、独自の DWH のいずれであっても、PII やクレジットカード番号、API キーが平文で残れば、それは新しい漏洩面になります。アクセス権限を持つ社員数が増えるほどリスクは線形に拡大します。

対策は二段構えです。**入口でのマスキング**として、SDK 側で送信前に正規表現や Microsoft Presidio などの PII 検出器で置換する。**出口でのアクセス制御**として、ログストアのアクセス権限を最小化し、PII が含まれるトレースを別バケットに隔離する。さらに、保持期間を 30 日 / 90 日 / 1 年などポリシーで切り、古いトレースは自動削除します。デバッグのためにマスキングを一時的に無効化したい誘惑は強いですが、その例外は原則として認めず、再現用の合成データで代替する習慣をつけます。

## 経路 3: 検索対象から間違って出てくる

マルチテナント SaaS で最も事故が多いのがこの経路です。RAG の検索対象に**テナント A のデータとテナント B のデータが同じインデックスに混在**しており、フィルタ漏れで B のユーザーに A の文書が引用されてしまう、というパターンです。LLM 本体は何も悪くなく、検索層の設計ミスです。

対策は「テナント分離をデータ層でも検索層でも徹底する」ことに尽きます。具体的には、ベクトル DB のメタデータに `tenant_id` を必ず付与し、検索クエリには**クライアントが指定する `tenant_id` ではなく、認証セッションから派生したサーバー側の値**でフィルタをかけます。可能ならインデックス自体をテナント単位で分け、誤クエリでも他テナントを引けない物理分離にしておくとさらに安全です。検索層の単体テストに「異なるテナントの文書がヒットしないか」を必ず含めます。

## 注意点

3 経路はそれぞれ担当チームが分かれがちで、責任の隙間からリークが起きます。学習データはセキュリティ部門、ログは SRE / 観測基盤チーム、検索対象はプロダクトの開発チーム、というふうに縦割りになりやすい。**RACI を 3 経路で明示し、誰がオプトアウトを管理し、誰がマスキングルールを更新し、誰がテナントフィルタを検証するかを決め切る**ことが、設計初期の重要タスクです。

## まとめ

- データリークは「学習データ」「ログ」「検索対象」の 3 経路に分けて対策する
- それぞれにオプトアウト・ログマスキング・テナントフィルタという別の手当てが要る
- 3 経路は担当が分かれやすいため、責任分界を設計初期に決める
