## このセクションで学ぶこと

- 社内データを扱う Agent / RAG の設計原則をチェックリストとして列挙できる
- 最小権限・テナント分離・監査ログ・データ保持期間の意義を説明できる
- モデルプロバイダ選定で見るべき観点(地域・契約・データ保持)を判断できる

## なぜ設計原則を先に並べるか

セキュリティは**あとから足せません**。Agent / RAG を一旦リリースしてから「やっぱり権限を絞ろう」と方針を変えると、データ移行や再インデックスを伴い、最悪の場合は作り直しになります。本セクションは、設計初期に必ずチェックすべき原則をリスト化したものです。新しい LLM 機能を社内データに接続する前に、この 5 項目を順に確認してください。

## 設計原則チェックリスト

**1. 最小権限(Least Privilege)**

Agent が呼べる Tool と参照できるデータを、業務遂行に必要な最小限に絞ります。「とりあえず全 API を渡す」設計は事故の温床です。Tool 単位で利用可否を切り、ユーザーロール × Agent ロールの行列で許可を定義します。検索対象も同様で、**ユーザーがそもそも閲覧権限を持たないデータは検索層でフィルタしてから LLM に渡す**のが鉄則です。LLM の出力で権限制御をしてはいけません(LLM は確率的に漏らします)。

**2. テナント分離**

マルチテナント環境では、テナントを跨いだデータ混入が最大級のインシデントです。前節で触れたとおり、ベクトル DB のメタデータ・検索クエリ・キャッシュキー・ログのいずれにも `tenant_id` を一貫して通します。**サーバー側で派生した認証セッションの `tenant_id` だけを信頼**し、クライアントから来た値は使いません。検索層の単体テストに「他テナント文書がヒットしないか」を必ず含めます。

**3. 監査ログ**

誰が、いつ、どの Agent を呼び、どの Tool を実行し、どのデータを参照したかを、後追いできる粒度で残します。インシデント発生時の影響範囲特定と、コンプライアンス監査(SOC2・ISO27001・GDPR の DPIA など)の両方で必須です。PII を含み得るプロンプト本文と、誰が呼んだかの構造化メタデータは別ストアに分け、**アクセス権限を別系統で管理**するのが扱いやすい構成です。

**4. データ保持期間**

ログ・トレース・キャッシュ・ベクトル化済みデータの**それぞれに保持期間を明文化**します。短すぎればデバッグや監査ができず、長すぎれば漏洩面が広がります。実務的には「トレース 30 日 / 監査ログ 1 年 / 学習用データセット 永続(明示同意あり)」のような階層を作り、自動削除ジョブで強制します。**期限切れデータは「いつか消そう」では消えません**。

**5. モデルプロバイダ選定**

最後に、ホスティング先のプロバイダ選定です。見るべき観点は次のとおりです。

- **地域**: データレジデンシ要件に合致するか(国内処理・EU 域内・米国)
- **契約条件**: Enterprise / BAA / DPA の有無、サブプロセッサ開示、賠償責任の上限
- **データ保持ポリシー**: 入力データの学習利用と保持期間がオプトアウト可能か
- **SLA**: 稼働率・レイテンシ・サポート応答時間
- **モデル品質**: ベンチマーク・自社評価データセットでのスコア

純粋な性能だけで選ぶと、契約条件で詰む(または法務に止められる)ケースがあります。**性能と契約は別軸**として早期に並行検討します。

## 注意点

これらの原則はすべて**設計初期に決め切る**ことに意味があります。リリース直前のチェックリストとして使うと、根本的な作り直しが発生します。プロジェクトキックオフのときに本セクションを開き、各項目の責任者と決定事項を 1 ページの設計メモにまとめておくと、後段の議論が安定します。また、原則は静的ではなく、規制(EU AI Act など)やモデルプロバイダの契約変更で見直しが必要です。半年ごとに棚卸しする運用を組み込みましょう。

## まとめ

- 最小権限・テナント分離・監査ログ・データ保持期間・プロバイダ選定の 5 項目を設計初期に決める
- 権限制御は LLM 出力ではなく検索層・Tool 層で行う
- 性能と契約条件は別軸として並行評価し、半年単位で棚卸しする
