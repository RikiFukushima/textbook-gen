## このセクションで学ぶこと

- LLM アプリが確率的システムであり、同じ入力でも結果が一意に再現できないことを理解する
- 観測性 (Observability) とトレーシング (Tracing) の違いを把握する
- 「事後に何が起きたかを復元できる」設計を運用前提として組み込む意義を説明できる

## 確率的システムとしての LLM アプリ

従来の Web アプリケーションは、基本的に決定論的なシステムでした。同じリクエストを投げれば同じレスポンスが返り、不具合が起きてもログとスタックトレースを辿れば原因にたどり着けます。再現テストも比較的容易で、デバッグの基本は「再現してブレークポイントを置く」ことでした。

LLM アプリは、この前提が崩れます。temperature をゼロに設定しても、モデル側の更新やサンプリングの揺らぎ、検索対象データの変化により、同じ入力で同じ結果が出る保証はありません。Agent が複数ステップにわたって推論する場合、その経路はさらに発散します。「再現できない不具合」が日常的に発生する世界です。

だからこそ、起きた事象を後から復元できるよう、実行中の挙動をできる限り残しておく必要があります。これが LLM アプリにおける観測性の出発点です。

## 観測性とトレーシングの関係

観測性は、システム全体の状態を外部から推し量れる度合いを指す広い概念です。一般には次の 3 つのシグナルで構成されます。

- ログ: イベント単位のテキスト記録
- メトリクス: 数値の時系列(レイテンシ、エラー率など)
- トレース: 1 リクエストに紐づく処理の時系列ツリー

LLM アプリで特に重要なのがトレースです。1 回のユーザー入力に対して、リトリーバル → プロンプト構築 → LLM 呼び出し → ツール実行 → 分岐 → 再帰呼び出し、と内部処理が複雑に枝分かれします。これを 1 本の木として捉えられないと、どこで何が起きたか追えません。

## 具体例: 「たまに変な答えが返る」を追う

たとえば「同じ質問なのに、たまに無関係な回答が返る」という報告が来たとします。決定論的な発想で再現を試みても、3 回に 1 回しか起きないため原因が掴めません。

ここでトレースが残っていれば、当該リクエストの「リトリーバル結果に関係ない文書が混入していた」「Conditional Edge が想定外の分岐に進んだ」「LLM が出力したツール呼び出し引数が壊れていた」といった具体的な事象を、後から確認できます。再現できないなら、起きた瞬間の証拠を残しておくしかありません。

## 注意点: ログとトレースを混同しない

print デバッグの延長で「ログを出しているから観測できている」と考えると、運用に入った瞬間に行き詰まります。1 リクエスト分のログがどこから始まりどこで終わるかを追うだけで時間が溶けるからです。トレースは「1 リクエスト = 1 ツリー」という構造を最初から持つため、観測コストが桁違いに下がります。後付けで導入するより、開発初期から組み込むのが鉄則です。

## まとめ

- LLM アプリは確率的で、同じ入力での再現が保証されない
- だからこそ、起きた事象を残すトレースが運用の前提になる
- ログだけでなく、1 リクエスト = 1 ツリー の構造を持つトレースを最初から組み込む
