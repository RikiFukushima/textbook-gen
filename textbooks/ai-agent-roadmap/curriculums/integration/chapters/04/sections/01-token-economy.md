## このセクションで学ぶこと

- LLM の課金は入力トークンと出力トークンで単価が異なる
- System Prompt や RAG 検索結果でコンテキストは想像以上に膨らむ
- 月額試算は「平均トークン × QPS × 月秒数」のシンプルな掛け算から始める

## 入力と出力で単価が違うという前提

LLM の API はほぼすべて「トークン課金」で、しかも入力(プロンプト側)と出力(生成側)で単価が違います。ざっくり言えば、出力の方が入力よりも数倍高いプロバイダが多く、これは生成のコストが推論時の自己回帰計算に依存しているからです。具体的な単価は 2025 年から 2026 年にかけても変動が激しいため、本書では絶対値ではなく相対比で考える流儀を取ります。

おおまかな相対感は次の表のように整理できます。実際の値はプロバイダや時期で動きますが、構造としては似た形になります。

| 種類 | 相対単価(入力を 1 とした目安) | 主な構成要素 |
| --- | --- | --- |
| 入力(高品質モデル) | 1.0 | System Prompt / 履歴 / RAG 検索結果 / ユーザ質問 |
| 出力(高品質モデル) | 3 - 5 | 生成された回答本文 |
| 入力(軽量モデル) | 0.05 - 0.2 | 同上 |
| 出力(軽量モデル) | 0.1 - 0.5 | 同上 |
| Prompt Cache ヒット時の入力 | 0.1 - 0.25 | 同じ System Prompt の再利用部分 |

ここで重要なのは「高品質モデルと軽量モデルの間には、おおむね一桁分の単価差がある」「Cache が効くと入力単価がさらに一桁近く下がる」という関係です。コスト設計はこの段差をどう使うかの設計だと考えてください。

## コンテキスト膨張という静かな罠

入力トークンは「ユーザのメッセージ」だけだと思っていると痛い目を見ます。実際にリクエストに乗るのは次のような積み上げです。

- System Prompt(役割定義・出力フォーマット指示・禁止事項):500 - 2000 トークン
- Few-shot 例(数件):500 - 3000 トークン
- 会話履歴(直近 N ターン):状況次第で 1000 - 10000 トークン
- RAG の検索結果(Top-K のチャンク):2000 - 8000 トークン
- ユーザの質問本文:50 - 500 トークン

合計すると、ユーザが入力したのは 1 行でも、API に投げられているのは 1 万トークンを超える、というのが本番運用ではごく普通に起こります。さらに会話履歴は対話のたびに伸び続けるので、放置すると 1 セッションの後半ほどコストが指数的に増えていきます。

## 月額の試算は掛け算から始める

コスト見積もりは、いきなり厳密な計算を目指さず、次の素朴な掛け算からスタートしてください。

- 月額コスト ≒ 1 リクエストの平均トークン数 × 単価 × QPS × 月秒数

たとえば 1 リクエストあたり入力 5000 + 出力 500 トークン、想定 QPS 1、月秒数 2.6M 秒(約 30 日)とすれば、月のトークン量は単純計算で数十億トークン規模になります。ここに前述の相対単価を掛ければ、桁のオーダーが見えてきます。重要なのは小数点以下の精度ではなく、「自分のサービスが月数千円規模なのか、月数十万円規模なのか」の桁を素早く把握することです。

## 注意点 — 推測でなく計測から始める

ここで強調したいのは、トークン経済の最適化は **推測ではなく実測から始める** ということです。コンテキストが何トークンになっているか、出力が平均何トークン返っているかは、観測性(2 章)を仕込めばすぐ見えます。逆に、観測の仕組みもないまま「Prompt を短くしました」「モデルを変えました」と動くのは、目を閉じてダーツを投げているのと同じです。最初の一手は必ず「現状の平均入力・出力トークン数」をダッシュボードに出すことから入りましょう。

## まとめ

- 入力と出力の単価は違い、構造としては出力が高く、軽量モデルや Cache で大幅に下がる
- 実際の入力トークンは System Prompt・履歴・RAG 結果の合算で、想定の数倍に膨らみやすい
- 月額試算は「平均トークン × QPS × 月秒数」で桁を掴むところから、必ず計測を伴って始める
