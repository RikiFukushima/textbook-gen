## このセクションで学ぶこと

- LLM-as-a-Judge がなぜ実務で広く使われるかを説明できる
- Judge LLM が持ち込むバイアスと、その緩和策を理解する
- どのメトリクスを LLM-as-a-Judge で測り、どこは人手評価が必要かを判断できる

## なぜ LLM に LLM を評価させるのか

LLM の出力品質を「正解と一致するか」で測ろうとすると、表現の揺れに弱すぎてほとんど機能しません。「東京」と答えるべきところで「日本の首都」と答えても意味は同じですが、文字列一致では不正解になります。かといって毎回人手で採点していたら、評価コストが開発コストを上回ります。

そこで登場したのが **LLM-as-a-Judge** です。GPT-4 級の大きなモデルに評価ルーブリックと共に出力を渡し、「この回答は文脈に忠実か」「ユーザーの質問に答えられているか」を採点させます。Ragas や LangSmith の評価機能、そして社内独自の評価スクリプトのほとんどはこの手法を内部で使っています。

## 具体例 — 単純な Judge プロンプト

実務で使う Judge プロンプトは、おおむね次のような構造になります。

```text
あなたは厳密な評価者です。以下のルーブリックに従って 1-5 で採点してください。

[ルーブリック]
5: 質問に完全に答えており、提示された文脈にも忠実
3: 部分的に答えているが、文脈外の情報を含む
1: 質問と無関係、あるいは誤情報

[質問]
{question}

[提示された文脈]
{context}

[回答]
{answer}

JSON で {"score": int, "reason": str} を返してください。
```

ポイントは **数値スコアだけでなく理由(reason)を一緒に出させる** ことです。理由を吐かせると採点の一貫性が上がり、後でレビューする時の手掛かりにもなります。

## 落とし穴 — Judge も LLM である

LLM-as-a-Judge は便利ですが、**Judge 自体が LLM である以上、LLM のバイアスをそのまま引き継ぎます**。実務で特に問題になるのが次の 3 つです。

- **自己優遇バイアス**:GPT-4 が GPT-4 の出力を採点すると、他社モデルより甘くなりがちです。**評価対象と異なる系統のモデルを Judge に使う** か、複数の Judge の平均を取るのが緩和策です。
- **位置バイアス**:A と B の回答を比べさせると、先に出した方が選ばれやすい傾向があります。**提示順をランダム化** し、両順序の結果を平均します。
- **長さバイアス**:長くて詳しい回答を「丁寧」と誤認して高く採点しがちです。ルーブリックに「簡潔さも評価する」と明記するか、文字数を別途モニタリングします。

さらに、**Judge LLM が知らないドメイン**(社内固有の業務用語、最新の規約変更など)では、Judge 自身が誤判定します。専門領域では、最初に 30-50 件だけ人手で採点し、その結果と Judge の結果の相関を取って **校正(キャリブレーション)** してから本格運用に乗せましょう。

## 注意点 — 「Judge を信じすぎない」運用設計

実務での原則は **「Judge LLM のスコアは『指標』であり『真実』ではない」** という姿勢です。スコア改善を追ううちに Judge を gaming するようなプロンプトに収束していることもあります。具体的には次を回します。

- **抜き取り人手評価**:Judge が「5 点」とした回答を週に 20 件だけ人が見る
- **Judge 入れ替え検証**:四半期に 1 回は別の Judge モデルで同じデータを評価し、ズレを記録
- **クリティカルな判定は人手必須**:法的・医療的な正確性判定は LLM-as-a-Judge に任せない

## まとめ

- LLM-as-a-Judge は人手評価をスケールさせる中核技術だが、Judge 自身のバイアスを必ず引き継ぐ
- 自己優遇・位置・長さの 3 バイアスを意識し、Judge モデル分離・順序ランダム化・ルーブリック明記で緩和する
- Judge のスコアは指標であり真実ではない。抜き取り人手評価と Judge 入れ替え検証を必ず併走させる
