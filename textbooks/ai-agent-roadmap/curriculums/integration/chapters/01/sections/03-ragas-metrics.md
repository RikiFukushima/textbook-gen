## このセクションで学ぶこと

- Ragas の 3 つの主要メトリクスがそれぞれ何を測っているかを直感で説明できる
- 3 メトリクスがどの問題を切り分けるか(検索の問題か生成の問題か)を理解する
- Ragas を導入する際の最小構成と注意点を把握する

## RAG 評価のデファクト・Ragas

RAG パイプラインは「検索フェーズ」と「生成フェーズ」が直列につながっており、最終出力が悪い時にどちらが原因か特定するのが難しい構造です。**Ragas は検索と生成を別メトリクスで測る** ことで、この切り分けを楽にしてくれる OSS ライブラリです。2025-2026 時点では RAG 評価のデファクトと言える存在で、LangChain や LlamaIndex とも統合されています。

Ragas が提供するメトリクスは複数ありますが、まず押さえるべきは **Faithfulness / Answer Relevancy / Context Precision** の 3 つです。それぞれを表で整理します。

| メトリクス | 何を測るか | 必要な入力 | 悪い時の主な原因 |
| --- | --- | --- | --- |
| Faithfulness | 回答が文脈に忠実か(ハルシネーションの少なさ) | 質問・回答・文脈 | 生成 LLM が文脈を無視して創作している / プロンプトで「知っていることを答えて」と誘導している |
| Answer Relevancy | 回答が質問に答えているか | 質問・回答 | 質問の意図を読み違えている / 余談が多い / 「分かりません」だけで終わっている |
| Context Precision | 検索結果のうち関連情報の割合 | 質問・検索された文脈・正解(または期待関連性) | 検索クエリの設計が悪い / チャンクが大きすぎる / 埋め込みモデルがドメインに合っていない |

ポイントは **3 つを別々に測ることで、改善対象が「検索か」「生成か」「プロンプトか」を判別できる** ことです。たとえば Context Precision は高いのに Faithfulness が低ければ、検索は良いが生成 LLM が文脈を活かせていない、と切り分けられます。

## 具体例 — 最小構成のスコア解釈

社内 FAQ Bot の評価で次のような結果が出たとします。

| ケース | Faithfulness | Answer Relevancy | Context Precision | 解釈 |
| --- | --- | --- | --- | --- |
| A | 0.92 | 0.88 | 0.85 | 検索・生成ともに良好 |
| B | 0.40 | 0.85 | 0.90 | 検索は良いのに、生成が文脈を無視。プロンプトを「文脈にない情報は答えない」へ修正 |
| C | 0.88 | 0.45 | 0.30 | 検索結果が的外れ。埋め込み再構築・チャンク戦略の見直しが先決 |
| D | 0.95 | 0.40 | 0.80 | 検索は十分。回答が質問にズレている。質問理解のプロンプト改善が必要 |

このように、**3 指標の組み合わせから対処方針が一意に決まる** のが Ragas を使う最大の効用です。「とりあえずプロンプトを書き換える」「とりあえずチャンクサイズを変える」という当て推量から抜け出せます。

## 注意点 — Ragas も LLM-as-a-Judge

Ragas のメトリクスは内部で **LLM-as-a-Judge を使っている** ため、前のセクションで挙げたバイアス(自己優遇・位置・長さ)はすべて持ち込まれます。実務では次を意識します。

- **Judge モデルを明示**:`evaluator_llm` に何を使ったかを評価ログに必ず残す。GPT-4 と Claude では同じ出力でもスコアが変わります。
- **絶対値より相対値**:「Faithfulness 0.85 だから合格」ではなく、「先週 0.80 だった同じデータセットで 0.85 になったから改善」と相対比較で使う。
- **小さなゴールデンセットで始める**:50 件あれば傾向は見えます。最初から数千件用意しようとして着手が遅れるより、小さく回す方が改善が進みます。
- **コストに注意**:Ragas は 1 ケースあたり Judge LLM を複数回呼び出します。500 件をフル評価すると API 費が無視できない額になるので、CI 自動実行は小さく、本格評価は週次など頻度を区切るのが現実的です。

## まとめ

- Ragas は RAG パイプラインを Faithfulness / Answer Relevancy / Context Precision で切り分け評価できる
- 3 指標の組み合わせから「検索を直すか・生成を直すか・プロンプトを直すか」が一意に決まる
- Ragas も LLM-as-a-Judge なので、Judge モデルを明示し、絶対値より相対比較で使う
