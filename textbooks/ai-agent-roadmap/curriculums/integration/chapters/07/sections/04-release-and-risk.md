## このセクションで学ぶこと

- 段階的リリースを 3 フェーズで設計できる
- AI プロジェクト固有のリスクを 3 種類に分類して書ける
- 各リスクに対する軽減策を明文化できる

## なぜ AI プロジェクトはリスクを先に書くのか

従来のソフトウェアでも障害対応計画は書きますが、AI プロジェクトでは **「期待した品質が出ない」こと自体が日常リスク** です。LLM は確率的に振る舞い、評価が完璧でも本番のエッジケースで間違えます。だからこそ、設計段階でリスクを書き出し、軽減策とセットで明文化することが、合意とリスク低減の両面で効きます。本章のメッセージの中核です。

リリース計画書は次の 2 ブロックで構成します。

1. **リリースフェーズ**:段階的リリースのフェーズ定義と移行条件
2. **想定リスクと軽減策**:AI 固有リスクを類型化して列挙

## 段階的リリースの 3 フェーズ

Ch06 で扱った段階的リリース戦略と整合させて、次の 3 フェーズで書きます。

- **フェーズ 1: 内部利用**:開発チーム + 業務オーナー数名(5-10 名)が日常業務で使う。期間 2-4 週間。狙いは UX 不具合・明らかな品質問題の早期発見。**移行条件**: 重大バグなし、Faithfulness が評価計画の合格ラインを満たす。
- **フェーズ 2: 限定公開**:対象ユーザーの 10-30% にロールアウト。期間 2-4 週間。狙いは想定外の質問パターンの収集と、オンラインメトリクスの初期測定。**移行条件**: 役立った率が目標値を満たす、重大な誤回答インシデント 0 件、コスト実績が想定内。
- **フェーズ 3: 本番投入**:全ユーザーに公開。継続運用サンプリング評価が稼働し、コスト・レイテンシのアラート整備が完了している状態にする。

各フェーズの **「移行条件」が書かれているか** が設計書レビューの要です。「内部で 2 週間使ってみて良さそうなら次へ」では条件になっていません。**評価メトリクスの合格ラインと、インシデント件数の閾値** で書きます。

## AI プロジェクト固有のリスク 3 種

リスクは類型化すると抜け漏れが減ります。以下の 3 カテゴリで書くと、AI プロジェクトの典型的リスクを概ねカバーできます。

### リスク 1: 誤回答による業務影響

LLM は自信を持って間違えます。コーチ支援 Bot なら「過去 QA が存在しないのに、もっともらしい架空の回答を提示する」「禁止表現を含む回答を生成する」などが典型例です。

- **軽減策**:Faithfulness の合格ラインを高めに設定、根拠ドキュメントを必ず引用表示、最終出力は人間が確認する運用(自動投稿しない)、ガードレールで禁止語を弾く。
- **検知方法**:週次サンプリング評価で誤回答率を追跡、UI に「誤回答を報告」ボタンを設置。

### リスク 2: コスト超過

トークン消費は予測しにくく、想定の数倍に膨らむことがあります。コンテキスト膨張、Agent の Tool 呼び出し連鎖、Cache ミスなどが原因です。

- **軽減策**:月次コスト上限のアラート(80% / 100%)、Prompt Cache の活用、入力長の上限制御、軽量モデルへのフォールバック。
- **検知方法**:LangSmith でトレースごとのトークン消費を可視化、日次コストのダッシュボード化。

### リスク 3: PII リーク

社内データを扱う際の最重要リスクです。受講生の氏名・連絡先・成績などが意図せず Bot の応答に含まれる、ログに残る、外部 LLM プロバイダに渡る、の 3 経路で漏れる可能性があります。

- **軽減策**:取り込み前のマスキング処理、ログ保存前の PII フィルタ、外部 API を使う場合は契約上のデータ取り扱い条項を確認、検索インデックスから機密文書を分離。
- **検知方法**:出力サンプリングで PII パターン(メール・電話番号など)を機械チェック、月次の人手レビュー。

## 注意点 — 軽減策がない「リスクのリスト」は意味がない

よくある失敗は、リスクを列挙するだけで軽減策と検知方法が書かれていないパターンです。リスクは **「リスク・軽減策・検知方法」の 3 点セット** で書きます。3 点セットが揃わないリスクは、設計書の段階で「対応不可」と判断されているのと同じです。スコープから外すか、フェーズ移行条件として「これは未対応であることを承認する」と明記してください。

また、リスクを書く順番は **「業務影響が大きいもの順」** にします。コスト超過より誤回答や PII リークの方が事業影響が大きいことが多いため、上から読むと優先度が伝わる構成にします。

## まとめ

- 段階的リリースは「内部 → 限定 → 本番」の 3 フェーズで、各フェーズに数値の移行条件を書く
- AI 固有リスクは「誤回答・コスト超過・PII リーク」の 3 類型で整理する
- リスクは「リスク・軽減策・検知方法」の 3 点セットで書き、業務影響が大きい順に並べる
