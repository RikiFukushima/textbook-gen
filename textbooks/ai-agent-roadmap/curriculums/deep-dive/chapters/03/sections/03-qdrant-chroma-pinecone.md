# Qdrant / Chroma / Pinecone の比較 — 設計思想と運用性

## このセクションで学ぶこと

- Qdrant・Chroma・Pinecone それぞれの「設計思想」と立ち位置の違い
- セルフホスト vs マネージドのトレードオフをどう見るか
- 開発フェーズ・本番フェーズで自然に選ばれる組み合わせ

## 3 つの製品は「同じ機能の比較」ではなく「思想の比較」

専用のベクトル DB と呼ばれる製品はいくつもありますが、ここでは代表的な 3 つ — **Qdrant・Chroma・Pinecone** を取り上げます。機能リストを並べると似たり寄ったりに見えますが、**何を主役に置いて設計したか** が大きく違います。

| 製品 | 設計の主役 | 典型的なデプロイ | 強み | 弱み |
| --- | --- | --- | --- | --- |
| Qdrant | 検索の表現力と運用性 | セルフホスト or マネージド | ペイロードフィルタが強力、Rust 製で軽快 | 自前運用なら他 DB と同じく面倒見が必要 |
| Chroma | 開発者体験(DX) | 組み込み / ローカル / クラウド | `pip install` で即動く、API がシンプル | 大規模本番向けの実績は他社に劣る |
| Pinecone | フルマネージド | SaaS のみ(自前ホスト不可) | インフラを完全に外注できる | 料金が読みづらい、ベンダロックインが強い |

この表を頭に入れた上で、それぞれを少し掘り下げます。

## Qdrant — 表現力と運用性の両立

Qdrant は **Rust 製のベクトル DB** で、セルフホストとマネージド(Qdrant Cloud)の両方が提供されます。設計上の特徴は、**ペイロード(メタデータ)とベクトル検索を一体で扱える** ことです。`color = "red" AND price < 1000` のような条件を index 側で評価しながら ANN 検索を進められるため、フィルタが効きすぎてヒット数がゼロにならない仕組みも備えています。

Qdrant は単体プロセスとしての挙動が軽く、Docker ですぐ立ち上がります。Kubernetes 上での運用や水平スケールにも対応しており、**「自社の VPC 内で動かしたいが運用負荷は抑えたい」** という要件に合います。トレードオフとして、コミュニティ版を自前運用するなら監視・バックアップ・アップグレードの面倒は自分で見る必要があります。

## Chroma — 開発体験を最優先した軽量プロダクト

Chroma の売りは、徹底した **シンプルさと組み込みやすさ** です。`pip install chromadb` の直後に Python から数行で永続化までできます。最初は組み込みモード(同一プロセス内で SQLite を使う形)で動かし、必要になったらサーバモードに切り替える、というスムーズな移行パスを意識した設計になっています。

実装は LangChain や LlamaIndex のチュートリアルの「最初のベクトル DB」としてもよく出てきます。逆に言うと、Chroma は **プロトタイピングと社内ツール向け** の色が濃く、巨大トラフィックや厳密な SLA を求める本番系には別の選択肢を検討する組織が多いです。最近はクラウド版も展開していますが、商用プラットフォームとしての歴史は他より浅めです。

## Pinecone — インフラを完全に切り離す SaaS

Pinecone は **フルマネージド専業** のベクトル DB です。セルフホスト版は存在せず、API 経由でしか触れません。インフラ運用を完全にベンダーに預けられるため、**「アプリ側の実装に集中したい」** チームに刺さります。スケーリング・レプリケーション・バックアップが裏で動き、操作はほぼ「インデックスを作る」「upsert する」「query する」の 3 つに集約されます。

注意点は **料金モデルとロックイン** です。Pinecone は時間課金+読み書き量課金の組み合わせで、データ量と QPS に比例してコストが伸びていきます。さらに自社の DB に直接触れないため、生のベクトルや index を取り出して他製品に移すのは作業量が大きくなります。本番ワークロードが軌道に乗ると、コスト最適化と移植性が継続的な検討テーマになります。

## どう選ばれるか — フェーズによる自然な分かれ目

ここまでを実務感覚に落とすと、次のようなパターンに収束しがちです。

- **PoC・ハッカソン・社内デモ**: Chroma の組み込みモードか pgvector
- **本番だが社内インフラに乗せたい**: Qdrant(セルフホスト)か pgvector
- **本番でインフラ運用したくない・スピード優先**: Pinecone か各クラウドのマネージド版
- **PoC で組んだ Chroma を本番に出す段階**: Qdrant か Pinecone への移行を検討

「どれが優れているか」ではなく、**チームのインフラ事情・コスト許容度・運用人数で自然に答えが決まる** のがベクトル DB 選定の本質です。次節では、この決め方をもう一段構造化した判断軸を整理します。

## まとめ

- Qdrant は表現力と運用性の両立、Chroma は DX、Pinecone はフルマネージドが設計の主役
- 3 製品は機能ではなく「何を主役にしたか」で見ると違いが鮮明になる
- PoC は軽量側、本番は運用ポリシーで Qdrant/Pinecone/pgvector に振り分けるのが定石
