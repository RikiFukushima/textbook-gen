## このセクションで学ぶこと

- 「RAG をベースに FT で形式を安定化」など、3 アプローチの併用パターンを理解する
- 役割分担を明確にしながら手段を重ねる設計の勘所を掴む
- 併用時に発生しがちな運用上の落とし穴を把握する

## なぜ組み合わせるのか

判断フローでは「一つずつ試して打ち止め」を基本にしましたが、現実のプロダクトでは、ひとつの手段だけで完結することはむしろ少数派です。**知識・振る舞い・文脈はそれぞれ別の問題**である以上、複数の問題が同時に存在する案件では複数の道具を重ねるのが自然な帰結です。重要なのは、**何のためにそれぞれを使うかという役割分担を明文化すること**です。

## 例 1:RAG をベースに、形式安定化のために小規模 SFT

社内ナレッジを参照しながら、決まった JSON で回答するインターナルな問い合わせ Bot を考えます。要件は二つあります。

- 知識:社内ドキュメントに基づいて答える(知識の問題)
- 振る舞い:回答は常に決まった JSON スキーマで返す(形式の問題)

この場合、まず RAG で知識側の問題を解きます。インデックスの設計、検索、引用の組み立てまでを RAG レイヤーが担います。一方で、検索結果をプロンプトに同梱しただけでは「JSON で返してください」という指示にモデルが従わない、あるいは前置きが混ざる、という現象が運用の中で頻発します。

ここで効くのが、**形式の安定化に絞った小規模な SFT(LoRA)** です。学習データは「検索結果と質問の組 → 正しい JSON 出力」のペアを数百件用意します。重要なのは、**学習データの中に「事実」をなるべく持ち込まない**ことです。事実は推論時に検索結果として注入される前提で、FT はあくまで形式と回答スタイルだけをモデルに刷り込みます。こうすると、知識更新は RAG 側のインデックス更新だけで完結し、FT 済みアダプタは長く使い回せます。

## 例 2:入口で動的プロンプト、本体は標準モデル + RAG

複雑な業務 Agent では、ユーザーの入力タイプによって必要な指示が大きく変わります。「データ抽出」「要約」「分類」「会話的な相談」といった用途を同じプロンプトでこなそうとすると、指示が膨らみ、どの用途でも中途半端な品質になりがちです。

この場合は、まず**入口で軽い分類器(あるいは LLM 自身)で入力タイプを判定**し、用途に応じてプロンプトテンプレートと Few-shot 例を**動的に差し替える**設計が効きます。本体は標準モデル + RAG のままで、FT はしません。コードでイメージするとこうなります。

```python
def build_prompt(user_input: str, retrieved_docs: list[str]) -> str:
    intent = classify_intent(user_input)  # "extract" | "summarize" | "classify" | "chat"
    template = TEMPLATES[intent]
    examples = FEW_SHOT[intent][:3]
    return template.format(
        examples="\n\n".join(examples),
        context="\n---\n".join(retrieved_docs),
        question=user_input,
    )
```

この設計の良さは、**新しい用途を追加するときに「テンプレートと例を足すだけ」で済む**点です。FT を選んでいた場合、新用途のたびに再学習が必要になります。動的プロンプトは「プロンプト改善を運用に乗せるための仕組み」と理解するとよいです。

## 例 3:三層すべてを薄く重ねる構成

大規模なカスタマーサポート Agent では、三層すべてを薄く重ねるケースもあります。

- **FT(薄め)**:口調・トーン・禁止表現を学習させた LoRA
- **RAG**:プロダクトドキュメント・FAQ・過去事例のインデックス
- **プロンプト改善**:案件種別ごとのテンプレート差し替え、出力フォーマット指定

このとき重要なのは、**どの層が何を保証するか**をチーム内で文書化することです。「口調は FT、知識は RAG、出力構造はプロンプト」のように責務を切ると、問題が起きたときにどの層を直すかが即座に決まります。

## 注意点:重ねるほど運用は複雑になる

ハイブリッドは強力ですが、層が増えるほど評価と運用が難しくなります。FT 済みアダプタとベースモデルの組み合わせをいくつ管理するか、検索インデックスの更新と FT のバージョンをどう整合させるか、プロンプトテンプレートの A/B テストをどこで回すか。**増やした層の数だけ、リグレッション検出の仕組みが必要**になります。最初から三層を組むのではなく、フロー通りに段階的に積み上げてください。

## まとめ

- 多くのプロダクトでは「RAG で知識、FT で形式、プロンプトで文脈」の役割分担で組み合わせます
- 動的プロンプトは「プロンプト改善を運用に乗せる仕組み」として強力です
- 層を増やすほど運用が複雑になるため、段階的に積み上げ、責務を文書化します
