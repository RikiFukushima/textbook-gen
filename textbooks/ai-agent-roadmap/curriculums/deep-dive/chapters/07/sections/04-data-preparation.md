## このセクションで学ぶこと

- ファインチューニングにおいて「データの質」が「データの量」より効くケースを説明できる
- SFT 用データの最小要件と、よくある失敗例を区別できる
- 多様性・形式・リークの 3 つの観点でデータを点検する手順を持つ

## なぜ「データの質」が量より効くのか

ファインチューニングを始めて最初にぶつかる誤解が「とにかくサンプル数を集めれば性能が出る」というものです。実際には、SFT は **モデルが既に持っている能力の中から、特定の振る舞い・出力形式を選び取らせる** 工程に近く、少数でも筋の良いデータのほうが、ノイズの多い大量データより明確に効きます。

経験的には、用途を絞った社内 SFT では **数百 〜 数千件のクリーンなペア** で十分な変化が出ることがあります。逆に、数万件あっても出力形式が揺れていたり、誤った正解例が混じっていれば、モデルはそのブレをそのまま学んでしまいます。

## SFT データの最小要件

社内ナレッジを覚えさせる用途・口調を揃える用途のいずれでも、最低限揃えたい条件は次の 3 つです。

1. **形式の一貫性**: instruction / input / output のフィールド構造、Markdown の使い方、改行コードまで含めて統一する。揺れた書き方を混ぜると、推論時の出力も揺れる。
2. **正解の検証**: 各 output が「人間が読んで間違いがない」状態であること。ChatGPT 等の出力をそのまま教師にすると、もっともらしいが事実誤認のあるデータが混入しやすい。
3. **想定シナリオの代表性**: 推論時に来そうな入力分布を、学習データが薄くてもいいから網羅していること。「FAQ の 9 割が技術系なのに学習データの 9 割が事務問い合わせ」だと当然うまくいかない。

データ 1 件のイメージは以下のような JSONL です。

```json
{"instruction": "次の議事録を箇条書きで 3 行に要約してください。", "input": "...議事録本文...", "output": "- 次回会議は 6/15\n- A 案で進めることが決定\n- 詳細仕様はワーキンググループで詰める"}
```

## 多様性・形式・リーク — 3 つの点検観点

データを準備し終わったら、学習に流す前に必ず以下の観点で点検してください。

- **多様性**: 出力テンプレートが似すぎていないか。たとえばすべての回答が「結論から申し上げますと、…」で始まっていると、モデルは中身ではなく定型句を学んでしまう。サンプルをランダムに 20 件並べて、構造が単調になっていないかを目視で確認する。
- **形式**: フィールド名・改行・スペースの揺れがないか。`output` の末尾改行の有無だけでも、学習結果は変わる。`jq` や簡単なスクリプトで構造チェックをかけるとよい。
- **リーク**: 評価用データが学習用に紛れていないか。ドキュメント由来のペアを分割する場合は **ドキュメント単位で train/val を分ける** ようにし、同じ元ドキュメントから生成されたペアが両方に入らないようにする。

## 注意点 — 「合成データを足せば賢くなる」は危険

大規模モデルに教師データを書かせて量を増やす「合成データ拡張」は便利ですが、**元モデルの癖や誤りをコピーするだけになるリスク** が常にあります。合成データを使う場合は、

- どの程度を合成にしているかを記録する
- 人間によるサンプリングレビューを必ず挟む
- 完全合成だけで学習せず、人間検証済みデータと混ぜる

の 3 点を守ってください。これを怠ると、評価指標は良く見えるのに本番で奇妙な失敗をする、というよくある罠に落ちます。

## まとめ

- 量より質が効く局面が多く、数百 〜 数千件のクリーンなペアでも十分な変化が出る
- 形式の一貫性・正解の検証・想定シナリオの代表性が SFT データの最小要件
- 学習前に「多様性・形式・リーク」の 3 観点で必ず点検する
