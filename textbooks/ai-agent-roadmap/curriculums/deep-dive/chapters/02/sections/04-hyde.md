## このセクションで学ぶこと

- HyDE が質問と文書の埋め込み空間のミスマッチをどう解消するかを説明できる
- HyDE が特に効くクエリのパターンと、逆効果になるパターンを判断できる
- 単純な書き換えと HyDE の違いを区別できる

## 「質問の埋め込み」が文書を当てづらい理由

ベクトル検索の精度を観察していると、ある違和感に気付きます。**質問の文体と回答文書の文体は、しばしば大きく違う**ということです。質問は「LangGraph で並列にツールを呼ぶには?」のような短く疑問形の口語であり、文書は「LangGraph では `add_node` で並列実行が可能なノードを定義し...」のような解説的・宣言的な文体です。両者を同じ埋め込みモデルに通しても、表層の文体差のために類似度が伸び切らない、というのが現場でよく見る現象です。

HyDE (Hypothetical Document Embeddings) は、この問題に **質問を「仮の回答」に変換してから埋め込む** という発想で対処します。手順はシンプルで、

1. LLM に「この質問への簡潔な回答(または解説)を書いて」と仮の回答を生成させる
2. その仮回答を埋め込みベクトル化する
3. それを retriever のクエリベクトルとして検索する

仮回答の **内容が正しい必要はない**(あくまで「正しそうな回答文体の文」が出ればよい)というのが面白い点です。生成された仮回答は文書側と文体が近いため、埋め込み空間で本物の正解文書に近づきやすくなります。

```mermaid
flowchart LR
  Q[質問] --> L[LLM]
  L --> H[仮回答<br/>Hypothetical Answer]
  H --> E[埋め込み]
  E --> V[Vector Search]
  V --> D[実在の文書]
```

## Query Rewriting との違い

前節の Query Rewriting と似ているように見えますが、目的が異なります。Query Rewriting は **検索クエリとしてより適した質問形** を作るのが目的で、出力は依然として「質問」です。HyDE は **回答側の文体に寄せた埋め込みベクトル** を作るのが目的で、出力は「(仮の)回答」です。

実装としては Query Rewriting と組み合わせて使うこともできます。「質問を自己完結化 → HyDE で仮回答 → その埋め込みで検索」という多段構成は、長い対話で漠然とした質問が来る RAG で実効性のあるパターンです。

```python
# 擬似コード: HyDE
def hyde_search(query, llm, embedder, retriever):
    hypothetical = llm.generate(
        f"次の質問に対する 3-5 文の回答を書いてください: {query}"
    )
    vec = embedder.embed(hypothetical)
    return retriever.search_by_vector(vec, top_k=10)
```

## 効くクエリ・効かないクエリ

HyDE が特に効くのは、**質問が短く抽象的で、答えそのものを書いた文書を探したい** ケースです。「Tree of Thoughts とは?」「LoRA の利点は?」のような **解説型クエリ** や、**コーパスが解説記事中心** のときに有効です。

逆に効かない、あるいは逆効果になるケースもあります。第一に、**固有名詞・型番・エラーコードを直接探す質問**。仮回答を生成する過程で具体的な ID が薄まり、Hybrid Search の BM25 側が捉えていた情報が失われます。第二に、**LLM が幻覚で誤った前提を入れた仮回答**を作るケース。例えば質問が現実とずれている場合、仮回答が完全に作り話になって全く違う文書群を引き寄せます。第三に、**コーパスが質問形式の Q&A 集** など、文書側がもともと質問文体に近い場合は、HyDE の効果が薄くなります。

費用面も無視できません。検索1回あたり LLM 呼び出しが加わるので、応答までのレイテンシとコストが増えます。**ベクトル検索単独で取りこぼしの大きいクエリ群に絞って HyDE を発動する**(クエリ分類で振り分ける)という設計が現実的です。

## まとめ

- HyDE は「質問と文書の文体差」を仮回答経由で埋める発想
- 解説型・抽象型の質問に効きやすく、ID 直当ての質問では逆効果
- Query Rewriting と組み合わせ、必要なクエリだけに適用するのが実務的
