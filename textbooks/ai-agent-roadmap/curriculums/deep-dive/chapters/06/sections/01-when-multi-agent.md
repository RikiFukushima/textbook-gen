## このセクションで学ぶこと

- Multi-Agent を採用する前に「単一 Agent + Tool」で足りないかを問う姿勢を持てる
- 複数 Agent に分けたほうがよい典型条件(役割の独立性・並列性・コンテキスト分離)を挙げられる
- Multi-Agent 化で増えるコスト(レイテンシ・通信・デバッグ難度)を理解できる

## まずは「単一 Agent + Tool」で十分か疑う

Multi-Agent という言葉は派手で、複雑な問題を Agent 同士の会話で解かせる構図はそれ自体に魅力があります。しかし実務では、まず「単一 Agent に Tool を持たせるだけで足りないか」を必ず先に問うことを推奨します。前章で見たように、LangGraph の StateGraph は 1 つの Agent でも条件分岐・ループ・人間介入を表現でき、ほとんどの業務タスクはこの範囲で十分対応できます。

Multi-Agent にすると、見かけ上は役割が綺麗に分かれて分かりやすくなりますが、その裏側で次のコストが必ず増えます。

- **レイテンシ**: Agent 間でメッセージが行き来する分、ターン数が増える
- **トークン消費**: 各 Agent が独立した system prompt と履歴を持ち、二重三重に消費する
- **デバッグ難度**: どの Agent のどのターンで何が起きたかを追うのに、ログ設計が一段難しくなる
- **失敗モードの複合化**: 1 体が暴走すると他の Agent も巻き込まれる

これらを払ってでも分けたい理由がない場合、単一 Agent + Tool に留めるのが現実的な選択です。

## Multi-Agent にしたほうがよい典型条件

逆に、次のような兆候があれば Multi-Agent を真剣に検討する価値があります。

1. **役割の独立性**: 「コードを書く役」「コードをレビューする役」のように、視点や評価基準が本質的に異なる。同じ Agent に両方やらせると自己評価が甘くなる。
2. **コンテキスト分離が必要**: 参照すべき知識・プロンプト・ツールセットが役割ごとに大きく異なり、1 つの system prompt にまとめると肥大化して指示が薄まる。
3. **並列性**: 「複数の候補を独立に試したい」「複数ドキュメントを同時に要約したい」など、本質的に並列で走らせると速くなる作業がある。
4. **権限の分離**: 書き込み権限を持つ Agent と読み取り専用の Agent を分けるなど、安全上の境界を明確にしたい。

特に 1 番目の「自己評価が甘くなる問題」は、Multi-Agent を導入する最も実利的な動機の 1 つです。生成と批評を同じ Agent にやらせると、自分の書いた文章を「OK」と判定してしまいやすく、Reviewer Agent を分けることで初めて批判的な視点が安定します。

## 具体例: 翻訳タスクで比べてみる

例えば「日本語の技術文書を英訳する」というタスクを考えます。

- **単一 Agent**: 「翻訳して」のプロンプト 1 回で済む。多くの場合これで十分。
- **Multi-Agent**: Translator が初訳を作り、Reviewer が違和感のある箇所を指摘し、Translator が修正する。品質は確かに上がるが、3 倍以上のトークンと時間を使う。

社内ナレッジ向けの一括翻訳なら単一 Agent、顧客に出すマーケティング文書なら Multi-Agent、というように **「品質を上げるためにコストを払う価値があるか」** で判断します。

## 注意点: 「すごそうだから」で分けない

Multi-Agent は構成図が華やかなので、技術検証として組んでみるのは楽しい題材です。ただ本番運用に乗せるなら、**「単一 Agent でやってみた結果、何がどう足りなかったから分けた」** という経緯を説明できることが、設計判断の質を担保します。先に分けてしまうと、後から「実は単一 Agent でも同じ精度だった」と気づいて構成を畳むのは難しくなります。

## まとめ

- まず単一 Agent + Tool で足りないかを問い、足りない理由を言語化してから Multi-Agent を検討する
- 役割の独立性・コンテキスト分離・並列性・権限分離が分割の主な動機になる
- Multi-Agent はレイテンシ・トークン・デバッグ難度を必ず増やすため、払う価値があるかで判断する
