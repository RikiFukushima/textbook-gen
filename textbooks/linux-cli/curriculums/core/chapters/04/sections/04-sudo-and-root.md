## このセクションで学ぶこと

- root がすべての権限チェックを素通りする管理者ユーザーであることを理解する
- `sudo` で「必要なコマンドだけ」root 権限で実行できる
- `sudo` を使うときの基本的な作法とリスクを知る

## root — 権限チェックを受けない唯一のユーザー

ここまで見てきた rwx のルールには、1 つだけ例外がありました。**root**(別名スーパーユーザー)です。root はすべての権限チェックを素通りし、どのファイルでも読み書きでき、どのプロセスでも止められ、システム設定も自由に変更できます。

この力はシステム管理に不可欠です。ソフトウェアのインストール、`/etc` 配下の設定ファイルの編集、所有者の変更(前セクションの `chown`)などは、いずれも root の権限が必要です。

しかし、力が強すぎるゆえの危険もあります。root にとっては「保護されたファイル」という概念がないため、**タイプミス 1 つでシステムの中核ファイルを消したり壊したりできてしまいます**。第 1 章で「CLI にゴミ箱はない」と学びましたが、root ではその怖さがシステム全体に広がるのです。だからこそ、root として常時ログインして作業するのは避けるのが鉄則です。

## sudo — 必要な 1 コマンドだけ root になる

そこで登場するのが `sudo`(superuser do)です。`sudo` は**普段は一般ユーザーのまま、指定した 1 コマンドだけを root 権限で実行する**仕組みです。

```bash
apt update
# Permission denied などのエラー(システム領域を触るため一般ユーザーでは不可)

sudo apt update
# [sudo] password for alice: ← 自分のパスワードを入力
# 正常に実行される
```

聞かれるのは root のパスワードではなく**自分のパスワード**です。「sudo グループに入っている信頼されたユーザーが、本人確認の上で権限を借りる」というイメージで、誰がいつ sudo を使ったかは記録にも残ります。

`whoami` と組み合わせると、何が起きているかがよく分かります。

```bash
whoami        # alice
sudo whoami   # root
```

コマンドの実行中だけ root になり、終われば一般ユーザーに戻る。この「必要な瞬間だけ強い権限を借りる」スタイルは、**最小権限の原則**と呼ばれる安全設計の実践です。

## 注意点 — sudo を付ける前に一呼吸

`sudo` は便利な分、使い方を誤ると被害も最大になります。作法として 2 つ覚えておきましょう。

1 つ目は、**`Permission denied` を見ても反射的に `sudo` を付けない**ことです。エラーは「システムの大事な場所を触ろうとしていますよ」という警告でもあります。本当に root 権限が必要な操作なのか、単にパスや対象を間違えているだけなのかを、一呼吸おいて確認しましょう。

2 つ目は、**インターネットで見つけた `sudo` 付きコマンドをそのまま実行しない**ことです。root 権限のコマンドは、何をするか理解できないまま実行すると取り返しがつきません。特に `sudo rm -rf` を含むコマンドは、対象のパスを必ず自分の目で確かめてから実行してください。

## まとめ

- root は権限チェックを受けない管理者ユーザーで、強力な分タイプミス 1 つの被害も大きい
- `sudo` は自分のパスワードで「1 コマンドだけ」root 権限を借りる仕組み
- Permission denied に反射で sudo を付けず、理解できないコマンドに sudo を付けない
