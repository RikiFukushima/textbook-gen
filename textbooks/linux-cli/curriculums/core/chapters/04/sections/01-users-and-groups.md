## このセクションで学ぶこと

- Linux が複数のユーザーで共有する前提で設計された OS であることを理解する
- `whoami` や `id` で「自分が誰で、どのグループに属しているか」を確認できる
- 権限が「所有者・グループ・その他」の 3 区分で決まることを知る

## Linux は「みんなで 1 台を使う」前提で生まれた

Linux の祖先である UNIX は、コンピュータが高価で貴重だった時代に、大学や研究所で **1 台のマシンを大勢で共有する**ために作られました。何十人ものユーザーが同じマシンに同時にログインして作業する環境では、「このファイルは誰のものか」「誰が読んでよくて、誰が書き換えてよいのか」を区別する仕組みがなければ大混乱になります。

そのため Linux には、**マルチユーザー**を前提とした仕組みが最初から組み込まれています。すべてのユーザーはアカウントを持ち、すべてのファイルには「所有者」がいて、操作のたびに「あなたにその権限はありますか?」というチェックが走ります。

「自分しか使わないノート PC やサーバーには関係ない話では?」と思うかもしれませんが、そうではありません。1 人で使うマシンでもこの仕組みはまったく同じように動いていて、あなたは「数あるユーザーのうちの 1 人」としてログインしています。だからこそ、自分のマシンなのに `Permission denied`(権限がありません)と怒られる、という現象が起きるのです。

## 具体例 — 自分が誰かを確かめる

まず、いまの自分が誰としてログインしているかは `whoami` で確認できます。

```bash
whoami
# alice
```

もう少し詳しい情報は `id` で見られます。

```bash
id
# uid=1000(alice) gid=1000(alice) groups=1000(alice),27(sudo),1001(developers)
```

`uid` はユーザーを識別する番号、`gid` と `groups` は所属している**グループ**です。グループは「ユーザーをまとめる単位」で、会社でいう部署のようなものです。たとえば `developers` グループにプロジェクトのファイルへの書き込みを許可しておけば、メンバーを 1 人ずつ設定しなくても「グループに入れるだけ」で権限を共有できます。

また、ユーザーは人間だけではありません。Web サーバー用の `www-data` のように、**プログラム専用のユーザー**も存在します。プログラムを権限の弱い専用ユーザーで動かしておけば、万一乗っ取られても被害をそのユーザーの範囲に閉じ込められる、という安全設計です。

## 権限は 3 つの視点で決まる

ファイルにはそれぞれ「所有者(owner)」と「所属グループ(group)」が設定されています。そして Linux は、ファイルへのアクセスを次の 3 区分で判断します。

- **所有者** — そのファイルの持ち主
- **グループ** — ファイルが属するグループのメンバー
- **その他(others)** — それ以外の全員

「所有者には読み書きを許す、グループには読み取りだけ、その他には何も許さない」のように、区分ごとに別々の権限を設定できるのがポイントです。この 3 区分の読み方は次のセクションで詳しく扱います。

なお、この権限チェックには 1 つだけ例外があります。**root** と呼ばれる管理者ユーザーは、すべての制限を素通りしてあらゆる操作ができます。root については本章の最後で詳しく見ます。

## まとめ

- Linux はマルチユーザー前提の OS で、1 人で使うマシンでも同じ権限の仕組みが動いている
- `whoami` で自分のユーザー名、`id` で uid と所属グループを確認できる
- ファイルへのアクセスは「所有者・グループ・その他」の 3 区分で判断され、root だけが例外
