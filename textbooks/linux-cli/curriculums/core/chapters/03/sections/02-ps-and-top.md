## このセクションで学ぶこと

- `ps` で「その瞬間に動いているプロセス」を一覧できるようになる
- 実務の定番 `ps aux` の主要な列(USER・PID・%CPU・%MEM・COMMAND)を読めるようになる
- `top` でリアルタイムに観察し、`q` で抜けられるようになる

## 写真の ps、実況中継の top

プロセスを確認する道具は大きく 2 つあります。`ps` は **実行した瞬間のスナップショット(写真)** を表示し、`top` は **数秒ごとに更新され続ける実況中継(動画)** を表示します。「あのコマンドの PID を知りたい」なら `ps`、「なんだか動作が重い、何が CPU を食っているのか」なら `top`、が基本の使い分けです。

まずは `ps` を引数なしで打ってみましょう。

```bash
ps
#   PID TTY          TIME CMD
#  3100 pts/0    00:00:00 bash
#  4310 pts/0    00:00:00 ps
```

表示されるのは **自分の端末で動いているプロセスだけ** です。シェル(bash)と、いま実行した `ps` 自身が見えます。

## 具体例 — 全体を見るなら ps aux

システム全体を見たいときは、定番の形 `ps aux` を使います(a と x を合わせて全プロセスを表示。a: 端末に紐づくプロセスをユーザー問わず、x: 端末を持たないプロセスも、u: 詳細表示)。

```bash
ps aux
# USER       PID %CPU %MEM    VSZ   RSS TTY   STAT START   TIME COMMAND
# root         1  0.0  0.1 168932 11784 ?     Ss   09:00   0:02 /sbin/init
# riki      4201  0.3  1.2 712340 98212 pts/0 S    10:21   0:01 python app.py
```

読むべき列は 5 つで十分です。

- **USER**: 誰のプロセスか
- **PID**: プロセスの識別番号(kill などで使う)
- **%CPU / %MEM**: CPU・メモリをどれだけ使っているか
- **COMMAND**: 何のコマンドか

一覧は長大になるので、実務では前章で学んだ `grep` と組み合わせて絞り込むのが定番です。

```bash
ps aux | grep python
# riki      4201  0.3  1.2 ... python app.py
```

`|`(パイプ)は「左のコマンドの出力を、右のコマンドの入力につなぐ」記号です。詳しくは後のカリキュラムで扱いますが、この形はイディオムとして丸ごと覚えてしまって構いません。

リアルタイムに観察したいときは `top` です。

```bash
top
```

画面が切り替わり、CPU 使用率の高い順にプロセスが並んで数秒ごとに更新されます。上部には負荷の指標(load average)やメモリの使用状況も出ます。**終了は `q`** です。`less` と同じ要領ですね。

## 注意点 — ps は「その瞬間」しか写さない

`ps` の結果は写真なので、表示された直後に状況が変わっていることは普通にあります。「さっき見えた PID がもういない」こともあるため、kill などで使う直前に確認し直すのが安全です。

また、`ps aux | grep python` には **grep 自身のプロセスが混ざって表示される** ことがあります(コマンド列に `grep python` と出ている行がそれです)。探していたプロセスと見間違えないようにしましょう。

## まとめ

- `ps` はスナップショット、`top` はリアルタイム表示。`top` から抜けるのは `q`
- 実務の定番は `ps aux`。USER・PID・%CPU・%MEM・COMMAND の 5 列が読めれば十分
- `ps aux | grep 名前` で絞り込める。ただし grep 自身が結果に混ざる点と、ps は「その瞬間」の情報である点に注意
