## このセクションで学ぶこと

- 環境変数が「名前=値」の形でシェルとプログラムが共有する設定メモであること
- `echo $変数名` と `env` で環境変数の中身を確認する方法
- `export` で環境変数を設定する方法と、その効き目の範囲

## 環境変数は「名前=値」の設定メモ

シェルの中には、あなたのホームディレクトリの場所・ユーザー名・使用言語といった情報が、**「名前=値」のペア**としてあらかじめ保存されています。これが **環境変数** です。シェル自身が参照するだけでなく、シェルから起動したコマンドやアプリケーションにもまるごと引き継がれるため、「シェルとプログラムが共有する設定メモ」と考えると実態に近いです。

中身を見るには、変数名の前に `$` を付けます。`$` は「この名前の変数を値に置き換えてから実行せよ」という合図です。

```bash
echo $HOME    # /home/riki      ← ホームディレクトリの場所
echo $USER    # riki            ← ログイン中のユーザー名
echo $SHELL   # /bin/zsh        ← 使っているシェル
env           # 設定済みの環境変数を全部一覧表示
```

`env` を実行すると、想像より多くの変数がずらりと出てきます。すべて覚える必要はありません。「プログラムはこのメモを読んで動作を変えている」という構図だけつかめば十分です。たとえば `cd` を引数なしで打つとホームに戻れるのは、シェルが `$HOME` を読んでいるからです。

## 自分で設定してみる — export

環境変数は自分で作ることもできます。よく使われるのが、好みのエディタを伝える `EDITOR` です。

```bash
MY_NAME="riki"       # ただの「シェル変数」。このシェル内でだけ使える
echo $MY_NAME        # riki

export EDITOR=vim    # export 付きで「環境変数」になる
git commit           # git は $EDITOR を読み、vim でメッセージ編集を開く
```

`export` を付けない変数はそのシェルの中だけの存在ですが、`export` した変数は **シェルから起動した子プロセス(コマンドやアプリ)にも引き継がれます**。`git` のような外部プログラムに設定を伝えたいときは `export` が必要、と覚えてください。

## 注意点

- **`=` の前後にスペースを入れてはいけません**。`MY_NAME = "riki"` と書くと、シェルは `MY_NAME` というコマンドの実行と解釈してエラーになります。
- **`export` した設定は、そのターミナルを閉じると消えます**。別のタブや明日のターミナルには効きません。毎回使いたい設定の永続化は 03-03 の `.bashrc` / `.zshrc` で扱います。
- 環境変数は親から子への一方通行です。子プロセス側で値を変えても、元のシェルや別のターミナルには影響しません。
- 慣習として、環境変数の名前は `HOME` や `EDITOR` のように **大文字** で付けます。

## まとめ

- 環境変数は「名前=値」の設定メモで、シェルから起動したプログラムにも引き継がれる
- `echo $変数名` で個別に、`env` で一覧を確認できる。`$` は値への展開の合図
- `export 名前=値` で設定できるが、効くのはそのシェルの間だけ。永続化は次の課題
