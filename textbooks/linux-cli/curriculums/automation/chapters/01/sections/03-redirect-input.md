## このセクションで学ぶこと

- `<` で stdin の取り込み口をキーボードから**ファイル**に付け替える
- ファイル名を**引数で渡す**場合と `<` で**流し込む**場合の違いを知る
- 矢印の向き(`>` は出ていく・`<` は入ってくる)でリダイレクトを整理する

## 概念 — 入力の取り込み口を付け替える

`>` が出口の付け替えなら、`<` は**入り口の付け替え**です。stdin は初期状態でキーボードにつながっていますが、`<` を使うとファイルにつなぎ替えられます。

```bash
コマンド < ファイル
```

こう書くと、コマンドは「キーボードから打たれた文字」の代わりに「ファイルの中身」を stdin から受け取ります。データは常に**矢印の向きに流れる**と覚えてください。`>` はコマンドからファイルへ出ていく、`<` はファイルからコマンドへ入ってくる、という向きです。

## 具体例 — stdin しか読まないコマンドに食べさせる

「`cat ファイル` のようにファイル名を引数で渡せばいいのでは?」と思うかもしれません。実際、多くのコマンドは引数でファイルを受け取れます。しかし中には**stdin から読むことしかできない**コマンドがあります。代表が文字を置き換える `tr` です。

```bash
tr 'a-z' 'A-Z' < names.txt
# ALICE
# BOB
# CAROL
```

`tr` はファイル名の引数を受け付けない設計なので、`tr 'a-z' 'A-Z' names.txt` はエラーになります。`<` で stdin に流し込むのが正しい渡し方です。

引数で渡す場合と `<` で渡す場合の違いは、`wc -l`(行数を数える)で見るとよくわかります。

```bash
wc -l access.log
# 1024 access.log     ← ファイル名も表示される

wc -l < access.log
# 1024                ← 数値だけ
```

引数で渡したときは、`wc` 自身が「access.log というファイルを開いて数えた」と知っているのでファイル名を添えて報告します。`<` のときは、`wc` には**流れ込んでくるデータしか見えていません**。どこから来たのかを知らないので、純粋な数値だけを出します。スクリプトで「行数の数値だけ欲しい」場面では、この性質をそのまま活用できます。

入力と出力のリダイレクトは同時に使えます。

```bash
sort < names.txt > sorted.txt
```

`names.txt` から流し込み、並べ替えた結果を `sorted.txt` に書き出す——3 本の口のうち 2 本を一度に付け替えた形です。

## 注意点 — 出番が少なく見えるが、考え方が本体

正直に言うと、日常操作で `<` を打つ機会は `>` ほど多くありません。たいていのコマンドが引数でファイルを受け取れるからです。それでも `<` を学ぶ価値があるのは、「**コマンドは stdin から流れてくるデータを処理する装置だ**」という見方が手に入るからです。次の章で学ぶパイプ(`|`)は、まさに「前のコマンドの stdout を次のコマンドの stdin につなぐ」仕組みで、この見方がそのまま土台になります。

## まとめ

- `コマンド < ファイル` で stdin の取り込み口をファイルに付け替えられる
- 引数で渡すとコマンドはファイル名を知っているが、`<` ではデータだけが流れ込む。`tr` のように stdin 専用のコマンドもある
- データは矢印の向きに流れる。「コマンドは stdin を処理する装置」という見方が次章のパイプにつながる
