## このセクションで学ぶこと

- `2>` で stderr(2 番)だけをファイルへ付け替える
- `2> /dev/null` で不要なエラーメッセージを**捨てる**
- 正常な結果とエラーを**別々のファイル**に分けて保存する

## 概念 — 番号を指定して 2 番だけ付け替える

このセクションで、最初に覚えた番号「**2 = エラー**」が効いてきます。実は `>` は `1>` の省略形で、「1 番(stdout)をファイルへ」という意味でした。だから番号を 2 に変えれば、**stderr だけ**を付け替えられます。

```bash
コマンド 2> ファイル
```

これで、エラーメッセージはファイルへ流れ、正常な結果はこれまでどおり画面に表示されます。`>` とちょうど逆の分担です。

## 具体例 — find のノイズを黙らせる

`2>` のありがたみを一番実感できるのが `find` です。システム全体から設定ファイルを探すと、権限のないディレクトリで大量のエラーが出ます。

```bash
find / -name "*.conf"
# find: '/root': Permission denied
# find: '/proc/1': Permission denied
# /etc/resolv.conf
# find: '/var/lib/private': Permission denied
# ...エラーの洪水に、見たい結果が埋もれる
```

エラー(stderr)だけを捨ててしまいましょう。

```bash
find / -name "*.conf" 2> /dev/null
# /etc/resolv.conf
# /etc/sysctl.conf
# ...結果だけがすっきり並ぶ
```

`/dev/null` は「**書き込まれたものをすべて捨てる特別なファイル**」で、いわばデータのブラックホールです。`2> /dev/null` は「エラーはゴミ箱直行」という意味の定番イディオムとして、このまま丸ごと覚えて構いません。

捨てるのではなく**取っておきたい**場合は、普通のファイルを指定します。前のセクションの `>` と組み合わせると、正常な結果とエラーを別々のファイルに分けられます。

```bash
./backup.sh > result.log 2> error.log
```

stdout は `result.log` へ、stderr は `error.log` へ。深夜の自動実行のあと、`error.log` が空なら成功、何か書かれていれば失敗——と一目で切り分けられます。3 本の流れを区別して学んだ成果が、ここで実務の形になります。

ちなみに、両方を **1 つのファイルにまとめたい**ときは `2>&1` という書き方を使います。

```bash
./backup.sh > all.log 2>&1
```

`2>&1` は「2 番の行き先を、いま 1 番が向いている場所と同じにする」という意味です。`> all.log` で 1 番を `all.log` に向けたあと、2 番をそこへ合流させています。この**順番が大事**(先に `> all.log`、後に `2>&1`)とだけ覚えておけば、今は十分です。

## 注意点 — 空白と「見えないエラー」

`2>` は **2 と > の間に空白を入れずに**書いてください。`ls 2 > file` と書くと「`2` という名前のファイルを ls する」+「結果を file へ」という全く別の意味になります。

また、`2> /dev/null` はエラーを「解決」するのではなく「**見えなくする**」だけです。トラブルの原因を調べているときに付けたままだと、肝心の手がかりごと捨ててしまいます。ノイズだとわかっているエラーだけに使い、調査中は外す——この使い分けを忘れないでください。

## まとめ

- `>` は実は `1>` の省略形。番号を変えた `2>` で stderr だけをファイルへ付け替えられる
- `2> /dev/null` は「エラーを捨てる」定番イディオム。`> result.log 2> error.log` で結果とエラーを分離できる
- `2>` は空白なしで書く。エラーを捨てるのは「見えなくする」だけなので、調査中は外す
