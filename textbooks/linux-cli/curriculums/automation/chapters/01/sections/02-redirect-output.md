## このセクションで学ぶこと

- `>` で stdout の行き先を画面から**ファイル**に付け替える
- `>` は**上書き**、`>>` は**追記**。この違いを体で覚える
- `>` が付け替えるのは stdout だけで、**エラーは画面に出続ける**ことを理解する

## 概念 — 出力の行き先を付け替える

前のセクションで見たとおり、stdout は初期状態で画面につながっています。この行き先をファイルに付け替えるのが**リダイレクト**で、記号は `>` です。

```bash
コマンド > ファイル
```

これだけで、画面に出るはずだった正常な結果がそのままファイルに書き込まれます。指定したファイルがなければ新しく作られ、**すでにあれば中身を消して上書き**します。一方 `>>` は、既存の中身を残して**末尾に追記**します。

## 具体例 — 結果の保存とログの蓄積

`ls` の結果をファイルに保存してみます。

```bash
ls /etc > etc-list.txt
cat etc-list.txt
# /etc の一覧がファイルに入っている(画面には何も出ない)
```

リダイレクトした瞬間、画面には何も表示されなくなる点に注目してください。出力が消えたのではなく、**行き先が画面からファイルに変わった**だけです。

`>` と `>>` の違いは、実際に上書きさせてみるのが一番わかりやすいです。

```bash
echo "1 回目" > memo.txt
echo "2 回目" > memo.txt
cat memo.txt
# 2 回目          ← 1 回目は消えた(上書き)

echo "3 回目" >> memo.txt
cat memo.txt
# 2 回目
# 3 回目          ← 末尾に足された(追記)
```

実務での使い分けはシンプルで、「今この瞬間のスナップショットを残す」なら `>`、「記録を積み重ねる」なら `>>` です。たとえば作業ログは追記で育てます。

```bash
echo "backup done: $(date)" >> work.log
```

毎回これを実行すれば、`work.log` に実行履歴が 1 行ずつたまっていきます。後の章で学ぶシェルスクリプトと組み合わせると、自動処理の記録を残す定番パターンになります。

## 注意点 — > は容赦なく上書きする

`>` は確認なしで既存ファイルの中身を消します。`rm` と同じく**ゴミ箱はありません**。大事なファイル名をうっかり `>` の右側に書くと、コマンド本体が実行される前にファイルが空にされてしまいます。「追記のつもりだったのに `>` を打っていた」は定番の事故なので、ログ系のファイルには `>>` を使う癖をつけてください。

もう 1 つ、`>` が付け替えるのは **stdout(1 番)だけ**です。コマンドがエラーを出した場合、stderr は画面につながったままなので、エラーメッセージはファイルに入らず画面に表示されます。失敗にすぐ気付けるという意味では便利な性質ですが、「エラーもファイルに残したい」場合は別の書き方が必要です。それはこの章の最後のセクションで扱います。

## まとめ

- `コマンド > ファイル` で stdout の行き先をファイルに付け替えられる。出力は消えるのではなく行き先が変わる
- `>` は上書き、`>>` は追記。記録を積み重ねる用途では `>>` を使う
- `>` が動かすのは stdout だけ。エラー(stderr)は画面に出続ける
