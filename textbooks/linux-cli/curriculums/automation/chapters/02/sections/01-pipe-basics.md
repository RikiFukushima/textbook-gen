## このセクションで学ぶこと

- パイプ `|` が「前のコマンドの標準出力」を「次のコマンドの標準入力」につなぐ仕組みを理解する
- 中間ファイルを作る二度手間が、`|` の 1 文字で消えることを体感する
- リダイレクト(ファイルへ)とパイプ(コマンドへ)の使い分けを整理する

## パイプの考え方 — コマンドとコマンドを直結する

前の章で、コマンドには標準入力(stdin)・標準出力(stdout)・標準エラー出力(stderr)という 3 本の口があり、`>` で標準出力のつなぎ先をファイルに替えられることを学びました。実は、つなぎ先はファイルだけではありません。**別のコマンドの標準入力**にもつなげます。それを実現するのがパイプ `|` です。

たとえば「`/etc` にあるファイルの数を知りたい」とします。パイプがなければ、いったんファイルに書き出して読み直すしかありません。

```bash
ls /etc > tmp.txt    # 一覧をいったんファイルへ
wc -l < tmp.txt      # そのファイルの行数を数える
rm tmp.txt           # 後片付け
```

パイプを使えば、この 3 行が 1 行になります。

```bash
ls /etc | wc -l
# 138
```

`ls /etc` の標準出力が、画面ではなく `wc -l`(行数を数えるコマンド)の標準入力へ直接流れ込み、最後の結果だけが画面に表示されます。

```mermaid
graph LR
    A["ls /etc"] -- "標準出力" --> P(("|")) -- "標準入力" --> B["wc -l"] --> C["画面: 138"]
```

この `|` は 1973 年、UNIX の開発者の一人である Doug McIlroy の「コマンドの出力を別のコマンドへ流し込みたい」という提案を受けて実装されました。タイトルの「発明」は大げさではなく、この 1 文字が「小さなコマンドを組み合わせて大きな仕事をする」という Linux の使い方そのものを生み出しています。

## 具体例 — 「絞り込み」の相棒として使う

パイプの最初の実用例として定番なのが、**出力が多すぎるコマンドの後ろに `grep` をつなぐ**使い方です。

```bash
history | grep "ssh"     # 過去に打った ssh 関連のコマンドだけ表示
ps aux | grep "nginx"    # 実行中のプロセスから nginx だけ表示
ls -l /etc | grep ".conf" # 一覧から .conf を含む行だけ表示
```

「全部出す → 必要な行だけ通す」という流れは、このあと何度も登場する基本パターンです。パイプは 2 段に限らず、`コマンド | コマンド | コマンド` と何段でもつなげます。

## 注意点 — 流れるのは標準出力だけ・向きは一方通行

1 つ目の注意は、パイプを流れるのは**標準出力だけ**だということです。標準エラー出力はパイプに入らず、画面に直接表示されます。「エラーメッセージを grep で絞り込めない」と感じたら、それはエラーが標準エラー出力に出ているからです(前章の `2>` の話とつながります)。

2 つ目は、`>` との混同です。`|` の右側に置けるのは**コマンド**、`>` の右側に置けるのは**ファイル名**です。`ls | result.txt` と書いても `result.txt` というコマンドを探しに行ってエラーになります。「次に処理させたいなら `|`、保存したいなら `>`」と覚えてください。

## まとめ

- パイプ `|` は、左のコマンドの標準出力を右のコマンドの標準入力へ直結する。中間ファイルは不要
- 「全部出す → grep で絞る」が最初の定番パターン。パイプは何段でもつなげられる
- 流れるのは標準出力のみ(標準エラー出力は流れない)。次のコマンドへは `|`、ファイルへは `>`
