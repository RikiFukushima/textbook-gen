## このセクションで学ぶこと

- CLI の操作は文字として残せるため、保存して何度でも実行できること(自動化)
- コマンドの並びは、そのまま実行可能な手順書になること(再現性)
- 画面のないサーバーを操作する標準手段は CLI であること(遠隔操作)

## 理由 1: 自動化 — 操作を「保存」できる

GUI の操作は、やった瞬間に消えていきます。「このボタンを押して、次にこのメニューを開いて……」という手順は、人間の記憶かマニュアルの中にしか残りません。

一方、CLI の操作はすべて文字です。文字であるということは、ファイルに書いて保存できるということです。たとえば「作業フォルダの中身を日付付きで圧縮してバックアップする」という作業は、コマンドでこう書けます。

```bash
tar czf backup-$(date +%Y%m%d).tar.gz ~/work
```

この行をファイルに書いておけば(複数のコマンドをまとめたこうしたファイルを **シェルスクリプト** と呼びます)、明日も来月も、実行 1 回で同じバックアップが取れます。さらに「毎晩 2 時に自動実行する」といった予約も Linux の標準機能で実現できます。毎晩 GUI で手作業するのとの差は歴然です。

## 理由 2: 再現性 — 手順書がそのまま動く

新しいメンバーの開発環境を整える場面を想像してください。GUI の手順書は「この画面でここをクリック」というスクリーンショットの束になりがちで、ツールのバージョンが上がって画面が変わった途端に破綻します。CLI なら、手順書は実行できるコマンドそのものです。

```bash
# README に書かれたセットアップ手順の例
sudo apt update
sudo apt install -y git nodejs npm
git clone https://github.com/example/app.git
```

読んだ人はこれをコピーして実行するだけで、書いた人と同じ状態にたどり着けます。このように「誰がいつ実行しても同じ結果になる」性質を **再現性** と呼びます。再現性が高いと、作業ミスが減り、引き継ぎが楽になり、トラブルの際に「何をやったか」を正確に振り返れます。実務で CLI が信頼される最大の理由はここにあります。

## 理由 3: 遠隔操作 — サーバーには画面がない

Web サービスを動かしているサーバーは、データセンターやクラウドの中にあります。多くのサーバーにはモニターもマウスもつながっておらず、そもそも GUI がインストールされていないことも珍しくありません。GUI は CPU やメモリを消費するうえ、遠隔から画面を転送すると通信も重くなるためです。

そこで使われるのが **SSH** という仕組みです。手元の PC のターミナルからネットワーク越しにサーバーへ接続し、文字だけでやり取りします。

```bash
ssh user@server.example.com
```

接続してしまえば、手元のマシンと同じ感覚でコマンドを打てます。文字のやり取りだけなので回線が細くても軽快に動き、世界中どこからでも操作できます。「サーバーを触る仕事 = CLI を使う仕事」と言ってよいほど、この領域では CLI が事実上の標準です。

## 注意点: 強力さは事故にも直結する

自動化・再現性・遠隔操作は、いずれも「人手を介さず一気に実行できる」力です。裏を返せば、間違ったコマンドも同じ勢いで一気に実行されます。GUI なら「本当に削除しますか?」と確認ダイアログが出る場面でも、CLI は黙って即実行するのが基本動作です。この性質は今後の学習でも繰り返し登場するので、まずは「CLI は確認してくれない」とだけ覚えておいてください。

## まとめ

- CLI の操作は文字なので保存・再実行でき、シェルスクリプトによる自動化につながる
- コマンドの並びはそのまま実行できる手順書になり、再現性を生む
- 画面のないサーバーの操作は SSH + CLI が事実上の標準
