## このセクションで学ぶこと

- GUI と CLI は、どちらもコンピュータへ指示を出すための「窓口」であること
- GUI は「見て選ぶ」操作に、CLI は「正確に・速く・繰り返す」操作に強いこと
- 2 つは優劣ではなく、得意分野で使い分ける道具であること

## コンピュータへの 2 つの話しかけ方

私たちが PC を使うとき、実際にやっているのは「コンピュータへの指示出し」です。ファイルを開くのも、アプリを起動するのも、すべて指示です。その指示の出し方には、大きく 2 つの方式があります。

1 つ目が **GUI(Graphical User Interface)** です。ウィンドウ・アイコン・ボタンが画面に並び、マウスでクリックして操作します。Windows のエクスプローラーや macOS の Finder が代表例で、ほとんどの人が毎日使っている方式です。

2 つ目が **CLI(Command Line Interface)** です。黒い画面に文字でコマンド(命令文)を打ち込み、結果も文字で返ってきます。映画でハッカーがカタカタやっている、あの画面です。この黒い画面を表示するアプリを **ターミナル** と呼びます(ターミナルまわりの詳しい役割分担は、この章の後のセクションで整理します)。

どちらも行き着く先は同じコンピュータです。違うのは伝え方だけで、GUI は絵とマウス、CLI は文字とキーボードで指示を伝えます。

## 具体例で比べる: ファイル名の一括変更

フォルダの中に 100 個のテキストファイルがあり、拡張子をすべて `.txt` から `.md` に変えたいとします。GUI なら「右クリック → 名前の変更 → 入力」を 100 回繰り返すことになります。10 分仕事のうえ、途中で 1 つくらい打ち間違えそうです。CLI ならこう書けます。

```bash
# 100 個あっても、この 1 行で全部の .txt を .md に変えられる
for f in *.txt; do mv "$f" "${f%.txt}.md"; done
```

今はこの 1 行が読めなくてまったく問題ありません。大事なのは「GUI で 100 回繰り返す作業が、CLI では 1 行になる」という事実のほうです。

逆の例も見てみましょう。写真の明るさを調整したい、スライドのレイアウトを直したい——こうした「結果を見ながら微調整する」作業は GUI の独壇場です。CLI で画像の色味を数値指定して変えることも技術的には可能ですが、目で確かめながら直感的に動かせる GUI のほうが圧倒的に速く、快適です。

整理すると、得意分野は次のように分かれます。

| 観点 | GUI が得意 | CLI が得意 |
| --- | --- | --- |
| 操作の見つけやすさ | メニューを見れば分かる | コマンドを知らないと始まらない |
| 視覚的な調整 | 画像・レイアウト編集など | 不向き |
| 大量の繰り返し | 手作業で何百回 | 1 行で一括処理 |
| 操作の記録・共有 | スクリーンショット頼み | コマンドをコピペするだけ |

## 注意点: 「CLI = 上級者の証」ではない

CLI を学び始めると「すべて CLI でやるのがプロ」と思いがちですが、それは誤解です。現役のエンジニアも、ブラウザやデザインツールは GUI で使い、ファイルの一括処理やサーバー操作は CLI で行う、というように両方を使い分けています。問われるのは「どちらを使えるか」ではなく、「いまの作業にどちらが適しているかを判断できるか」です。

また、最初は黒い画面に威圧感を覚えるかもしれませんが、CLI の正体は「文字での会話」です。打ち込んだ文字に対して、文字で返事が来る。仕組みとしてはチャットに近い、素朴なやり取りにすぎません。

## まとめ

- GUI と CLI はどちらもコンピュータへの指示の窓口で、優劣ではなく得意分野が違う
- GUI は「見て選ぶ・視覚的に調整する」に、CLI は「正確に・速く・繰り返す」に強い
- エンジニアは両者を使い分けており、CLI はそのための新しい選択肢になる
