## このセクションで学ぶこと

- 同じ「サーバー側のエラー」でも 500 と 503 ではまったく意味が違うこと
- 503 が「行儀のよいエラー」と呼ばれる理由と、Twitter のくじらの話
- 502 / 504 が教えてくれる「どの階層で問題が起きているか」

## 同じ 5xx でも、悲鳴の種類が違う

人気アーティストのチケット発売、19 時ちょうど。ページを開くと「503 Service Unavailable」。一方、いつも使っているサービスがある日突然「500 Internal Server Error」を返す。どちらも前のセクションで見た 5xx、つまり「サーバー側のせい」クラスですが、**この 2 つはまったく別の悲鳴**です。

**500 Internal Server Error** は「内部で予期しない何かが起きた」という総称です。プログラムのバグ、処理中の例外、設定ミス — 原因は何であれ、サーバー自身も想定していなかった失敗がここに集約されます。詳しい事情を外部に語らないのは不親切なのではなく、内部のエラー内容をそのまま表示すると攻撃者にシステムの中身のヒントを与えてしまうため、あえて口を閉ざすという設計でもあります。

**503 Service Unavailable** は対照的に「壊れてはいないが、今は無理」という宣言です。アクセス集中による過負荷や、計画的なメンテナンスのときに返されます。つまり 500 は「想定外の故障」、503 は「想定内の満員・休業」。利用者にはどちらも「使えない」ですが、運用するエンジニアにとっては「バグを探すべきか、容量を増やすべきか」を分ける重要な手がかりになります。

## 503 は「行儀のよいエラー」 — そして、くじら

503 には気の利いた仕掛けがあります。応答に **Retry-After** というヘッダーを添えて、「120 秒後にまた来てください」と再訪の目安を伝えられるのです。検索エンジンのクローラーも 503 を「一時的な状態」と解釈するので、メンテナンス中のサイトをすぐに検索結果から消したりはしません。ただ落ちるのではなく、いつ戻るかまで伝える — エラーにも礼儀があるわけです。

この「満員です」という状態を一躍有名にしたのが、かつての Twitter です。2000 年代後半、急成長する Twitter は過負荷のたびにサービス停止を繰り返し、そのとき表示される「小鳥たちが網でくじらを吊り上げる絵」は **fail whale(失敗くじら)** という愛称で呼ばれ、Tシャツが作られるほどのインターネット文化になりました。エラー画面がマスコットになった、503 的状態の最も有名な例です。

## 502 と 504 — 多層化した Web の「どこ」が死んだか

現代の Web サービスは 1 台のサーバーではなく、CDN・ロードバランサー・アプリケーションサーバーといった**何層もの中継**でできています。そこで活躍するのが残りの 5xx です。**502 Bad Gateway** は「中継役が、後ろのサーバーからおかしな応答を受け取った(あるいは応答がなかった)」、504 Gateway Timeout は「後ろからの返事が時間切れになった」。つまり 502 / 504 が出たら、玄関(中継役)は生きていて、その奥で問題が起きていると読めます。番号だけで「どの階層の悲鳴か」の見当がつくのです。

注意点をひとつ。メンテナンス画面を表示するとき、手を抜いて 200 OK で「メンテナンス中です」というページを返してしまうと、検索エンジンはそれを正規のコンテンツとして記憶してしまいます。中身は人間向けの案内でも、機械への返事としては 503 を返すのが正解です。**画面の見た目と、機械が読む番号は別物** — ここを混同すると、メンテ明けに検索結果が「メンテナンス中です」になっている悲劇が起こります。

## まとめ

- 500 は「予期しない内部の故障(バグなど)」、503 は「過負荷やメンテナンスによる一時的な満員・休業」で、同じ 5xx でも意味がまったく違う
- 503 は Retry-After で再訪の目安を伝えられる行儀のよいエラーで、Twitter の fail whale はこの「満員」状態を文化にした
- 502 / 504 は多層化したシステムの「中継より奥」の問題を示し、メンテナンス画面は見た目だけでなく番号も 503 で返すのが正解
