## このセクションで学ぶこと

- 出力形式を明示しないとモデルは自然文に流れ、後工程で扱いづらいことを理解する
- 箇条書き・表・JSON など、用途に応じた形式を言葉で指定できる
- 形式指定は「見出し付き」で具体的に書くほど安定することを知る

## モデルは放っておくと「散文」を返す

第02章では入力側、つまりモデルに渡す指示や文脈の設計を扱いました。本章は反対の出口、つまり**モデルが返す出力の形**を扱います。

LLM は何も指定しなければ、なめらかな自然文(散文)を返すように学習されています。人間相手の会話としては自然ですが、出力を**後工程**で使おうとした瞬間にこれが障害になります。例えば「3つの候補を出して」と頼むと、モデルは「まず1つ目として〜が考えられます。次に〜も有力でしょう。最後に〜という選択肢もあります。」のように文章で返してきます。これを人が読む分には問題ありませんが、プログラムで「2つ目の候補だけ取り出す」のは途端に面倒になります。

そこで、**何の形式で出力してほしいかを言葉で明示する**のが出力制御の第一歩です。

## 用途に形式を合わせる

形式は「後工程が誰か」で選びます。代表的な3つを押さえれば実務の大半はカバーできます。

- **箇条書き**: 人がざっと読んで把握したいとき。「各項目を `- ` で始まる箇条書きで、1行ずつ出力してください」
- **表(Markdown テーブル)**: 複数の項目を属性ごとに比較したいとき。「列を『機能名 / 説明 / 料金』とした Markdown の表で出力してください」
- **JSON**: プログラムで解析して使うとき。「次のキーを持つ JSON 配列で出力してください: name(文字列), price(数値)」

同じ「候補を3つ」でも、指定の仕方でこれだけ変わります。

```text
悪い例:  おすすめのプランを3つ教えて。
良い例:  おすすめのプランを3つ、次の Markdown 表で出力してください。
         列は「プラン名 | 月額 | 向いている人」の3列とし、表以外の文章は書かないでください。
```

「表以外の文章は書かないでください」のような**余計な前置きを禁止する一文**を添えると、`「承知しました。以下が表です」`のような枕詞が消え、後工程がさらに楽になります。

## 注意点 — 「具体的に」「見出し付きで」が安定の鍵

形式指定は、抽象的だとブレます。「きれいにまとめて」「リスト形式で」だけでは、モデルごと・実行ごとに解釈が揺れます。**列名・キー名・順序まで具体的に書く**ほど出力は安定します。

また、複数のブロックを返させたいとき(例: 要約と箇条書きの両方)は、`## 要約` `## キーポイント` のように**見出しで区切る形式を指定する**と、後工程で機械的に分割できます。形式そのものをプロンプトの中に**サンプルとして1つ見せておく**のも非常に効果的で、これは次のセクションのスキーマ指定につながります。

## まとめ

- 指定しなければモデルは自然文を返す。後工程で使うなら形式を明示する。
- 箇条書き・表・JSON を用途(誰が読むか)で使い分ける。
- 列名・キー名まで具体的に、余計な前置きは禁止すると安定する。
