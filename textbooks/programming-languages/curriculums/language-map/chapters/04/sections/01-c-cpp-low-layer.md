## このセクションで学ぶこと

- C と C++ が「ハードウェアに近い」とはどういう意味かを理解する
- 手動でメモリを管理する自由と、その代償(危険さ)を説明できる
- C / C++ が今も現役で使われている代表的な分野を挙げられる

## 「ハードウェアに近い」とは

これまで見てきた Python や JavaScript は、書きやすさを優先し、メモリの細かい管理を言語側が肩代わりしてくれていました。これに対して **C** と、その拡張である **C++** は、もっと **低レイヤ**、つまりハードウェアや OS に近い場所で動くことを得意とする言語です。

「ハードウェアに近い」とは、メモリの「どの番地に」「何バイト」データを置くかをプログラマ自身が指示できる、という意味です。その鍵になるのが、メモリ上の場所を指し示す **ポインタ** という仕組みです。C / C++ はポインタを通じてメモリを直接読み書きできるため、CPU やメモリの動きをほぼそのままコードに反映できます。だからこそ、無駄のない高速な処理が書けるのです。

## 具体例 ― 自由と代償はセット

例えば C では、必要なメモリを `malloc` で自分で確保し、使い終わったら `free` で自分で返す、という **メモリ管理** を手動で行います。

```c
int *data = malloc(100 * sizeof(int)); // 100個分の領域を確保
// ... data を使う ...
free(data);                            // 使い終わったら自分で解放
```

この「自分で全部やる」自由が、速さの源です。Python や JavaScript の多くは、使い終わったメモリを言語側が自動で掃除してくれる「ガベージコレクション」という仕組みを持っています。便利な反面、その掃除はいつ走るか分からず、走っている間はほんの一瞬だけ処理が止まります。C / C++ にはこの裏方処理がそもそも無いので、動作が予測しやすく、ミリ秒・マイクロ秒単位の制御が効きます。OS(Linux など)、デバイスのファームウェア、ゲームエンジン、組み込み機器といった「速度と省メモリが死活問題になる」分野で、C / C++ は今も第一線で使われています。みなさんがいま使っている OS やブラウザの心臓部にも、C / C++ で書かれた部分が必ずと言っていいほど含まれています。

ただし自由には代償があります。`free` を忘れればメモリが少しずつ食い潰され(メモリリーク)、解放済みの領域をうっかり読み書きすれば、プログラムは予期せぬクラッシュやセキュリティ上の穴を生みます。確保した範囲を超えてデータを書き込んでしまう「バッファオーバーフロー」は、世界中の深刻なセキュリティ事故の典型的な原因になってきました。これらは言語が止めてくれず、書いた人間が気をつけるしかありません。「速いが、間違えると大事故になる」のが C / C++ の二面性です。

## 注意点 ― 「速いから万能」ではない

低レイヤで速いと聞くと万能に思えますが、その速さは「人間が正しく書けたら」という前提つきです。書きやすさや安全性は犠牲になっており、習得コストも高め。同じ機能を作るのに、Python なら数行で済むものが C / C++ では何倍もの行数とメモリ管理の手間がかかる、ということも珍しくありません。開発のスピードや人件費まで含めて考えると、Web アプリの画面づくりのような場面で、わざわざ C / C++ を選ぶ理由はほとんどありません。逆に言えば、「ここは 1 ミリ秒でも速くしたい」「メモリを 1 バイトでも節約したい」という所にだけ投入するのが正しい使い方です。**速度や省メモリが本当に必要な所だけで真価を発揮する**、専門家向けの道具だと捉えておきましょう。なお C と C++ は名前こそ似ていますが、C++ は C にオブジェクト指向などの機能を足して大規模開発をしやすくした言語で、両者は別物として扱われます。

## まとめ

- C / C++ はポインタでメモリを直接扱える「低レイヤ」に強い言語。
- 手動のメモリ管理は速さと引き換えに、リークやクラッシュの危険を伴う。
- OS・組み込み・ゲームエンジンなど、速度と省メモリが要の分野が主戦場。
