## このセクションで学ぶこと

- Rust が「C / C++ 並みの速度」と「安全性」を同時に狙う言語であることを理解する
- 所有権という仕組みが、メモリの危険をどう防ぐかをイメージできる
- Rust が向いている場面と、習得コストという現実を把握する

## C / C++ の弱点を埋めにきた言語

前のセクションで、C / C++ は速いけれどメモリ管理を間違えると大事故になる、と見ました。**Rust** はまさにこの弱点に正面から取り組んだ、比較的新しい言語です。狙いはシンプルで、「C / C++ 並みの速度を保ちながら、メモリの事故を仕組みで防ぐ」こと。速さのために安全を捨てる、安全のために速さを捨てる、というこれまでのトレードオフを崩そうとしています。

ポイントは、ガベージコレクションのような実行中の裏方処理に頼らず、**コンパイル時チェック**で安全を担保することです。Java や Python は実行中にメモリを自動で掃除してくれますが、その掃除のぶんだけ速度や予測しやすさを少し犠牲にしています。Rust はこの掃除そのものを無くす代わりに、「危ないコードはそもそもビルドさせない」という方針を取りました。実行前の変換段階で危ない書き方を弾くので、実行時の速度はそのまま。これが「両立」を可能にしている発想です。

## 具体例 ― 所有権というルール

その中心にあるのが **所有権** という考え方です。Rust では、すべてのデータに「持ち主(所有者)」が **ただ 1 つ** いると決まっています。持ち主がスコープ(その変数が有効な範囲)から消えると、メモリは自動で解放されます。`free` を書き忘れる余地がそもそもありません。

```rust
let s1 = String::from("hello");
let s2 = s1;          // 所有権が s1 から s2 へ「移動」した
// println!("{}", s1); // ← これはコンパイルエラー。s1 はもう持ち主ではない
```

C ならコンパイルが通ってしまい、実行時にこっそり壊れるようなコードを、Rust は **コンパイルの段階で止めて**くれます。「間違ったプログラムはそもそもビルドできない」ように設計されているわけです。これにより、**メモリ安全性**を速度と引き換えにせず手に入れられます。

## 注意点 ― タダではない

良いことずくめに見えますが、代償は **習得コスト**です。所有権のルールはコンパイラに厳しくチェックされるため、慣れるまでは「正しいはずなのに通らない」と感じる場面が多く、学習曲線は急です。ベテランでも最初はコンパイラと格闘することになり、「コンパイラと喧嘩する」と冗談まじりに言われるほどです。ただ、その厳しさは裏を返せば「ビルドさえ通れば、メモリの事故はまず起きない」という安心感でもあります。だからこそ、速度と安全性が両方シビアに求められる領域 ― OS の一部、ブラウザの基盤、システムツール、性能が要のサーバ ― で採用が広がっています。一方で、ちょっとした自動化スクリプトやデータ分析のように「書きやすさ最優先」で速度がさほど問われない用途では、まだ Python などの方が手軽で、無理に Rust を選ぶ必要はありません。

## まとめ

- Rust は速度と安全性のトレードオフを崩そうとする新しめの言語。
- 所有権とコンパイル時チェックで、メモリ事故を実行前に防ぐ。
- 習得コストは高いが、速度と安全が同時に要る領域で採用が拡大中。
