## このセクションで学ぶこと

- 「特定の用途のために作られた言語」が存在することを理解する
- SQL がデータベース操作に特化した言語であることを説明できる
- Swift / Kotlin がモバイルアプリ開発の主役であることを把握する

## 汎用言語と用途特化言語

ここまで登場した Python や Go、C / C++、Rust は、幅広い用途に使える「汎用言語」でした。一方で世の中には、**用途特化言語**、つまり「ある目的のためだけに作られた言語」もたくさんあります。対象は狭いぶん、その分野では汎用言語より圧倒的に簡潔に書けるのが特徴です。「何でも屋」ではなく「専門店」と考えると分かりやすいでしょう。

第 1 章で「言語ごとに得意・不得意がある」と学びましたが、用途特化言語はその得意分野を極端なまでに尖らせた存在だと言えます。

## 具体例 ― 代表的な専門店たち

最も身近な例が **SQL** です。これはデータベース(大量のデータを整理して保管する仕組み)に問い合わせるための言語です。SQL の特徴は **宣言型** であること。「どうやって探すか」ではなく「何が欲しいか」を書くだけで済みます。

```sql
SELECT name, age FROM users WHERE age >= 20;
-- 「users から 20 歳以上の name と age が欲しい」とだけ書く
```

探し方の手順は書きません。データベース側が最適な方法を考えてくれます。汎用言語でゼロから書けば何十行にもなる処理が、たった 1 文で表せるのです。

モバイルアプリの世界にも専門店があります。iPhone / iPad 向けには **Swift**(Apple 製)、Android 向けには **Kotlin** が事実上の標準です。どちらもそれぞれのプラットフォームの機能を素直に呼び出せるよう設計されており、「そのOS上で動くアプリを作るなら、まずこれ」という位置づけです。

ほかにも、統計やデータ分析に強い R、Excel などの表計算に組み込まれた式言語、Web ページの見た目を整える CSS、インフラ構成を記述する設定言語など、用途特化の言語は数えきれません。共通するのは「その分野の言葉でそのまま書ける」よう設計されている点で、だからこそ短く・読みやすく書けるのです。

## 注意点 ― 専門店は守備範囲の外では無力

用途特化言語は、対象の外では途端に役に立ちません。SQL でアプリの画面は作れませんし、Swift でデータベースそのものを設計するわけでもありません。だから実務では、**汎用言語と用途特化言語を組み合わせて**使うのが普通です。例えば「アプリ本体は Swift、データの問い合わせは SQL」のように、それぞれの専門店に得意分野を任せます。言語は 1 つに絞るものではなく、適材適所で併用するもの ― この感覚を持っておきましょう。

## まとめ

- 用途特化言語は特定目的のために尖らせた「専門店」型の言語。
- SQL はデータベース問い合わせ専用で、欲しい結果だけを宣言的に書く。
- Swift は iOS、Kotlin は Android アプリ開発の標準。汎用言語と併用するのが実務の基本。
