## このセクションで学ぶこと

- 「型」はデータの種類を表す仕組みであること
- 静的型は実行前に型をチェックすること
- 動的型は実行時に型が決まり、書きやすい反面ミスに気づきにくいこと

## 「型」とは何か

プログラムが扱うデータには、整数、小数、文字列(文章)、真偽値(はい/いいえ)など、いろいろな**種類**があります。この種類のことを**型**と呼びます。型が違えばできることも変わります。たとえば数値どうしなら足し算ができますが、「文字列に数値を足す」ような操作は意味があいまいで、トラブルのもとになります。

そこで多くの言語は、値の型を意識して扱う仕組みを持っています。この型の扱い方の違いが、コンパイル型・インタプリタ型と並ぶもう一つの大きな軸です。大きく**静的型付け**と**動的型付け**に分かれます。

## 静的型付け ― 実行前にチェックする

**静的型付け**の言語では、変数や値の型を実行する前に決め、その時点で矛盾がないかをチェックします。代表例は Go、Java、TypeScript などです。

たとえば「ここには整数が入る」と決めておけば、誤って文字列を入れようとした時点で「型が合いません」と指摘してくれます。プログラムを動かす前に多くのミスを見つけられるのが強みで、特に大規模なプログラムや大人数の開発で力を発揮します。後からコードを読む人にとっても、型が手がかりになって理解しやすくなります。反面、型を書く手間が増え、ちょっとした処理でもやや記述量が多くなりがちです。

## 動的型付け ― 実行しながら型が決まる

**動的型付け**の言語では、型を前もって宣言せず、実行時に値の型が決まります。代表例は Python、Ruby、JavaScript などです。

型を書かなくてよいぶん、コードが短く、思いついたことをすぐ形にできます。試作やデータ分析、学習のように素早く書いて試したい場面と相性が良いです。一方で、型の食い違いがあっても**実行してその箇所に到達するまで気づけない**ことがあります。たとえば数値が来る前提の処理に文字列が紛れ込んでも、その行が動くまでエラーにならず、原因の発見が遅れがちです。

## 二つの軸は組み合わせて考える

実行方式(コンパイル/インタプリタ)と型の方式(静的/動的)は別々の軸なので、組み合わせはいろいろあります。整理しておくと言語の見当がつけやすくなります。

| | 静的型付け | 動的型付け |
| --- | --- | --- |
| コンパイル寄り | Go, Java, C++ | (少数) |
| インタプリタ寄り | TypeScript(実行は JS) | Python, Ruby, JavaScript |

## 注意点 ― 「型がある/ない」ではなく「いつチェックするか」

よくある誤解が、「動的型付けの言語には型がない」というものです。正しくは、**型はあるが事前にチェックしないだけ**です。Python でも値は整数や文字列といった型を持っていて、無理な操作をすれば実行時にエラーになります。違いは型の有無ではなく、**型をチェックするタイミングが実行前か実行中か**という点にあります。この見方を押さえておくと、後で各言語を学ぶときに混乱しません。

## まとめ

- 型はデータの種類で、何ができるかを左右する。
- 静的型は実行前チェックで安全、動的型は手軽だがミスに気づきにくい。
- 違いは型の有無ではなく、チェックのタイミング。
