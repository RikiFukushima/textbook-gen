## このセクションで学ぶこと

- モジュールがパッケージをまとめた配布・依存管理の単位であることを理解する
- `go mod init` で go.mod を作りモジュールを初期化できる
- go.mod と go.sum がそれぞれ何を記録するか説明できる

## モジュールはパッケージの集まり

前のセクションで「パッケージはコードをまとめる単位」だと学びました。**モジュール** は、その一段上の単位です。1 つ以上のパッケージをまとめて「ここからここまでが 1 つのプロジェクト(または公開ライブラリ)です」と区切り、**バージョンと依存関係を管理** します。

現在の Go では、開発するプロジェクトは必ずモジュールとして扱うのが基本です。モジュールの中心になるのが、プロジェクトのルートに置く `go.mod` ファイルです。

## go mod init でモジュールを作る

新しいプロジェクトを始めるときは、ディレクトリを作って `go mod init` を実行します。引数にはモジュール名を渡します。公開する予定があるなら、リポジトリのパス(例: `github.com/yourname/myapp`)をモジュール名にするのが慣例です。

```bash
mkdir myapp
cd myapp
go mod init github.com/yourname/myapp
```

これで次のような `go.mod` が生成されます。

```
module github.com/yourname/myapp

go 1.22
```

`module` 行がモジュール名で、`import` でこのプロジェクトのパッケージを参照するときのプレフィックスにもなります。`go` 行は、このモジュールが想定する Go のバージョンです。

## go.mod と go.sum の役割

外部パッケージを使い始めると、`go.mod` に依存が追記され、もう 1 つ `go.sum` というファイルも作られます。役割は明確に分かれています。

- **go.mod**: どのモジュールの **どのバージョン** に依存するかを宣言する。人が読んで意味が分かる。
- **go.sum**: ダウンロードした依存モジュールの **ハッシュ値** を記録し、内容が改ざんされていないか検証する。中身は機械的な照合用。

```
require github.com/google/uuid v1.6.0   ← go.mod(依存の宣言)
github.com/google/uuid v1.6.0 h1:abc... ← go.sum(検証用ハッシュ)
```

両方とも **バージョン管理(Git など)にコミットする** のが原則です。これにより、他の人や別の環境でも完全に同じ依存関係を再現できます。

## 注意点

`go.sum` は手で編集するものではなく、`go` コマンドが自動で更新します。中身が分からないからと削除すると検証ができなくなるので、消さずにコミットしましょう。また、モジュール名は後から変えると import パスがすべて変わって影響が大きいため、`go mod init` の段階で慎重に決めるのがおすすめです。

## まとめ

- モジュールはパッケージをまとめた配布・依存管理の単位で、go.mod が中心。
- `go mod init モジュール名` でモジュールを初期化する。
- go.mod は依存の宣言、go.sum は改ざん検証用ハッシュ。どちらもコミットする。
