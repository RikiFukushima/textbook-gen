## このセクションで学ぶこと

- 委譲とは仕事をサブエージェントに渡し、別コンテキストで実行させることだと理解する
- git worktree が並列エージェントの衝突を避ける役割を担うことをつかむ
- 委譲が欠けると 1 エージェントがすべてを抱え込み破綻することを理解する

## 委譲 — 自分でやらず、渡す

トリアージで「次はこれ」と決まったら、その仕事を実際に進めます。ここで重要なのが**委譲**という考え方です。委譲とは、見つけた仕事を親エージェントが自分で抱え込まず、**サブエージェント**に渡すことです。

なぜわざわざ渡すのでしょうか。理由は 2 つあります。1 つめは、コンテキストの分離です。親エージェントが全部のタスクを 1 つの会話で抱えると、文脈がどんどん膨らみ、前の仕事の情報が次の仕事の判断を濁らせます。タスクごとにサブエージェントへ委譲すれば、それぞれが必要な情報だけを持った「きれいな机」で作業できます。2 つめは、並列化です。独立したタスクなら、複数のサブエージェントに同時に渡して並行で進められます。

## 実行 — 渡した先で手を動かす

**実行**は、委譲されたサブエージェントが実際にコードを書いたりファイルを編集したりする段階です。ここで複数のエージェントが同じファイルを同時にいじると、互いの変更を上書きし合って衝突します。これを避けるための定番が **git worktree** です。

git worktree は、1 つのリポジトリから複数の作業ディレクトリを切り出す Git の機能です。エージェントごとに別々の worktree を割り当てれば、それぞれが独立したディレクトリで作業でき、衝突しません。

```
# エージェント A 用と B 用に別々の作業場所を用意する
git worktree add ../work-fix-login fix/login
git worktree add ../work-fix-search fix/search
```

サブエージェントは MCP やプラグインを通じて、実際のツール(テスト実行、API 呼び出しなど)にも接続します。実行が「文章を書く」だけで終わらず、実際にコマンドを動かしたり外部サービスを叩いたりして世界を変えられるのは、こうしたツール接続があるからです。これらの構成部品は第 6 章でまとめて扱うので、ここでは「実行の足場になる」とだけ押さえてください。

委譲と実行をまとめると、「親が仕事を配り、子が独立した作業場所で手を動かし、必要なツールにつなぐ」という形になります。この分担があるからこそ、ループは仕事を 1 件ずつ着実に前へ進められます。

## 注意点 — 委譲が欠けると 1 人で全部抱える

委譲のステップが欠けると、1 つのエージェントが発見も実行も全部 1 つのコンテキストで抱え込みます。文脈は膨れ上がり、並列化もできず、やがて判断がぶれて破綻します。良いループは「親は配るだけ、実行は子に任せる」と役割を分けます。この分業が、ループを大きくしてもスケールさせる鍵になります。

## まとめ

- 委譲は、仕事を親が抱えずサブエージェントに渡してコンテキストを分離することです。
- git worktree を使えば、並列エージェントがファイル衝突せずに実行できます。
- 委譲が欠けると 1 エージェントが全部を抱え、文脈が膨れて破綻します。
