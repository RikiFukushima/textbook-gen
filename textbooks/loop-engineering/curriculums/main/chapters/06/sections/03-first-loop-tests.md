## このセクションで学ぶこと

- 「テストを通す」ループを6ステップに沿って具体的に組み立てられるようになる
- ゴールが検証可能(pass/fail)なループが良いループになる理由を理解する
- 構成部品と停止条件を最初のループにどう埋め込むかをつかむ

## なぜ「テストを通す」が最初のループに向くのか

最初に組むループは、**ゴールが検証可能なもの** を選ぶのが鉄則です。その点で「失敗しているテストを通す」ループは理想的です。完了の判定が「全テストが緑か」という pass/fail で、人間の主観が入りません。第05章で見た「コードを良くして」のような曖昧なゴールと違い、止めどきが機械的に決まります。

検証可能なゴールには、もう1つ実務的な利点があります。**進捗を数で測れる** ことです。「失敗テスト 12 件のうち 8 件が緑になった」のように、ループが前に進んでいるかを客観的な数字で追えます。これは第05章の進捗なし検出にもそのまま使えますし、後で人間が様子を見るときの安心材料にもなります。逆に「読みやすくして」のようなゴールは、進んだのか空回りしているのかすら判定できません。

## 6ステップに当てはめる

第02章の6ステップに、このループを乗せてみます。擬似コードで書くと次のようになります。

```
state = load_memory("progress.md")   # 第03章: 状態をディスクから読む

for i in range(MAX_ITER):            # 第05章: 反復上限
    if budget_exceeded(): break      # 第05章: コスト予算

    failing = discover_failing_tests()        # 発見・トリアージ
    if not failing:
        break                                 # ゴール達成チェック → 正常終了

    patch = maker_agent.fix(failing, state)   # 委譲: maker が修正
    result = run_tests()                      # 実行: テストを走らせる

    ok = checker_agent.judge(result, patch)   # 検証: checker = 別エージェント判定
    state = update_memory(state, result, ok)  # 記憶: 進捗を書き戻す

    if no_progress(state): break              # 第05章: 進捗なし検出
                                              # 決定: ループ先頭へ戻る
```

各行が6ステップ(発見・委譲・実行・検証・記憶・決定)と、前章までの設計に対応している点に注目してください。`run_tests()` の結果という客観的事実があるので、checker は「テストが緑か」を根拠に判定でき、自己採点になりません。これが、検証可能なゴールを選ぶことの実利です。曖昧なゴールだと checker は何を根拠に合否を出せばよいか分からず、結局 maker と同じ主観に頼ってしまいます。

ステップを言葉でも追っておきましょう。まず **発見** で `discover_failing_tests()` が失敗テストの一覧を作ります。失敗がなければそこが **ゴール達成チェック** になり、正常終了します。失敗があれば **委譲** で maker に修正を任せ、**実行** でテストを走らせ、**検証** で checker が結果を判定します。その判定と進捗を **記憶** に書き戻し、停止条件に引っかからなければ **決定** としてループ先頭へ戻ります。この一巡が、第02章で図にした循環そのものです。

## 注意点 — まず「最小の安全網」から書く

いきなり完璧なループを書こうとしないでください。最初は `MAX_ITER` とコスト予算という停止条件、そしてメモリの読み書きだけ先に入れ、maker/checker は1つのサブエージェント呼び出しから始めてかまいません。回しながら、衝突が出たら worktree を、手順の再利用が要るなら skills を、外部ツールが要るなら MCP を足していきます。**動く最小のループを先に回し、必要に応じて部品を増やす** のが現実的な順番です。

## まとめ

- 最初のループは pass/fail で完了が判定できる「テストを通す」が向いています。
- 6ステップに乗せると、各行が発見・委譲・実行・検証・記憶・決定と前章の設計に対応します。
- まず停止条件とメモリから書き、worktree や MCP は必要になってから足します。
