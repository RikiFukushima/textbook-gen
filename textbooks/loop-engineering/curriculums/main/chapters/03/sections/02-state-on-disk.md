## このセクションで学ぶこと

- 状態を「会話の外で生き残る場所」に置くという発想
- メモリの置き場として Markdown・課題ボード・外部 DB があること
- 「全部詰める」のではなく「必要な分だけ読み込む」点

## 状態は会話の外に書き出す

前のセクションで見たとおり、会話が切れれば理解も決定も消えます。これを防ぐ答えはシンプルで、**作業の状態を会話の外、ディスクや外部システムに置く**ことです。モデルが覚えていてくれないなら、覚えておく役目を会話の外に肩代わりさせればよい、という発想です。

置き場に求められる条件は一つだけです。**1 回の会話の外で生き残り、何が完了して次に何が来るかを記録できること**。この条件さえ満たせば形式は問いません。

代表的な置き場には次のようなものがあります。

- **Markdown ファイル** — 一番手軽です。リポジトリに `MEMORY.md` のようなファイルを置き、進行状況を書いていきます。人間も読めるのが利点です。
- **課題管理ボード** — タスクの状態を管理する外部システム。Linear のようなボードが一例です。「やること/進行中/完了」が構造的に残ります。
- **外部 DB** — 規模が大きく、構造化して大量に扱いたい場合の選択肢です。

## 「全部詰める」とは違う

ここで一つ区別しておきたいことがあります。状態を残すというと「会話のコンテキストウィンドウに全部詰め込めばいい」と考えがちですが、**それとは違います**。

永続メモリは会話の "外" に置きます。そしてループは各周回で、**そのとき必要な分だけ**そこから読み込みます。たとえば「次にやるタスク」と「直近の決定の根拠」だけ読めばよければ、それだけ読めばよいのです。すべてを毎回コンテキストに流し込む必要はありません。外に蓄積し、必要な分を取り出す、という構図がポイントです。

この区別が大事なのは、コンテキストに詰め込む方式には限界があるからです。会話は際限なく長くできるわけではありませんし、無関係な情報まで詰め込むと、モデルがいま注目すべきことを見失いやすくなります。状態は外に分厚く貯めておき、各周回ではそこから必要な一部だけを切り出して渡す——この分業ができると、ループは長く回っても破綻しにくくなります。「全部覚えさせる」のではなく「覚えておく場所を外に持ち、毎回そこを参照する」と考えてください。

## 注意点

- 凝った仕組みから始める必要はありません。まずは 1 枚の Markdown で十分機能します。
- 置き場を決めても「何を書くか」が曖昧だと役に立ちません。記録すべき中身は次のセクション(03-03)で具体化します。
- ボードや DB は人間とも共有しやすい反面、導入コストがかかります。小さく始めるなら Markdown が無難です。

## まとめ

- 状態は会話の外(ディスク・外部システム)に置き、会話が切れても生き残らせる
- 置き場は Markdown・課題ボード・外部 DB など。条件は「外で生き残り、進捗を記録できる」こと
- コンテキストに全部詰めるのではなく、外に蓄えて必要な分だけ読み込む
